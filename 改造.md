• 核心结论

  - 要做“开放域建模生成器”，关键不是把 LLM 换大，而是把系统从“选现有模板”升级成“理解需求 → 组装物理/控制/信号模块 → 生成
    并验证代码”。
  - 你当前项目还是闭集模板型：候选 model_id 来自知识库目录，校验层也只允许已知模型，见 knowledge_base/
    matlab_model_data.py:202、agents/model_spec_builder.py:24、agents/model_spec_validator.py:153

  必须具备的条件

  - 开放知识底座：不只是几个模板，而是有领域本体、方程、变量、边界条件、常见假设、参数范围、单位体系。
  - 可组合中间表示：不能只靠 model_id，要有类似 IR/DSL 的结构，能表达“刚体 + 流体阻力 + 推进 + 约束 + 事件”。
  - 任务分解能力：先识别领域、对象、动力学、输入输出、约束、环境，再决定建模路线，而不是直接选 Top1 模型。
  - 代码生成能力：生成器要支持“从 IR 渲染代码”，而不是只支持固定模板；否则开放域推出来也落不了地。
  - 验证闭环：自动做语法检查、数值稳定性检查、单位检查、仿真可运行性检查，不通过就修复。
  - 不确定性处理：缺参数时要先澄清、列假设、给默认值来源，而不是强行套最近模板。
  - 评测数据集：要有跨领域建模任务集，衡量“选型对不对、方程对不对、代码能不能跑、结果是否合理”。
  - 更强模型或多模型协作：通常要 LLM + 检索 + 规则/符号工具 + 仿真验证器协同，而不是单靠一次对话生成。

  对你这个项目的具体差距

  - 模型空间是闭集：现在只有固定 model_id 列表，缺“潜艇/鱼雷/水下发射”等新领域模板。
  - LLM 受候选约束：提示词要求优先从候选模型里选，天然不鼓励开放生成，见 agents/model_spec_builder.py:114
  - 校验器禁止新模型：未知 model_id 会直接报错，见 agents/model_spec_validator.py:153
  - 代码生成仍偏模板化：即使 LLM 想出新结构，后端也未必能稳定生成对应 MATLAB 代码。

  落地路线

  - 第一阶段：做“半开放域”——扩知识库、扩模板族、加“无匹配先澄清/拒绝误配”。
  - 第二阶段：引入 IR，把“对象/状态/方程/输入/输出/事件/约束”结构化。
  - 第三阶段：让代码生成从“按 model_id 渲染”升级为“按 IR 组合渲染”。
  - 第四阶段：加自动仿真验证和修复闭环，形成真正可用的开放域系统。

  一句话判断标准

  - 如果系统必须先问“这个需求对应哪个已知模板”，它还是闭集；
  - 如果系统能自己拆出“水下运动 + 阻力 + 推进 + 发射阶段切换”并生成可验证代码，才算开放域。
