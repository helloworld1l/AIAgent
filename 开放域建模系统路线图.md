# 开放域建模系统路线图

## 1. 目的

本文专门回答三个问题：

1. 大模型能替代哪些知识簇。
2. 哪些知识簇绝不能被大模型替代。
3. `Qwen3.5` 在当前项目中的最佳架构位置是什么。

结论先行：

- 如果目标是“开放域建模理解与规划能力”，更强的大模型可以显著扩大系统上限。
- 如果目标是“开放域可执行建模系统”，则不能只靠大模型；仍然必须保留结构化 IR、工具调用、代码组装、校验与回修闭环。
- 对当前项目而言，`Qwen3.5` 最适合放在“开放域理解 + IR 草拟 + 规格修复”的前端决策层，而不是直接替代最终渲染器、验证器和高价值规则库。

这与当前项目已有判断是一致的：系统已经是“半开放域 MATLAB 建模生成系统”，继续向开放域演进的重点仍然是扩大 family schema、增强 `OpenModelIR`、把 draft fragment 升级为可执行渲染，并补齐运行级验证闭环，而不是单纯换一个更大的模型。相关背景见 `详细设计文档.md` 与 `开放模型中间表示草案.md`。

## 2. 问题拆解：开放域建模系统到底缺什么

一个真正可用的开放域建模系统，至少有四层能力：

1. **开放域理解层**
   - 识别用户在说什么对象、场景、过程和约束。
2. **结构化表示层**
   - 把自然语言映射为对象、状态、输入、输出、方程、事件、约束、参数和假设。
3. **可执行生成层**
   - 把结构化表示变成 MATLAB / Python / Simulink 等可执行代码。
4. **验证闭环层**
   - 检查单位、边界条件、数值稳定性、参数范围、求解器选择与运行结果合理性。

大模型，尤其是更强的 `Qwen3.5` 这一类，主要强化前两层；第三、第四层不能只靠预训练知识保证。

## 3. 大模型能替代哪些知识簇

这里说的“替代”，不是指完全删除显式知识，而是指：这些知识不必再以“大而全、逐条硬编码”的方式维护，可以更多交给大模型承担。

### 3.1 可被大模型大幅替代的知识簇

#### A. 长尾术语与别名词典

例如：

- 专业术语别称
- 中英文混用表达
- 口语化需求转正式建模表达
- 跨域同义说法

原因：

- 这类知识分布广、变化快、维护成本高。
- 强模型通常已经学到大量术语关联、别称和上下文含义。
- 只要有少量高价值归一化词表兜底，大模型就能覆盖大量长尾表达。

适合的替代方式：

- 保留少量“高风险冲突词”显式词典。
- 其余长尾术语由大模型结合检索证据归一化。

#### B. 通用背景知识簇

例如：

- 某类系统的典型组成
- 常见物理量之间的粗粒度关系
- 常见工况、常见输入输出、常见建模目的
- 某领域模型的典型使用场景

原因：

- 这些是“理解问题”所需的背景，不一定要硬编码成规则。
- 大模型很适合在不确定时给出候选机制、候选 family 和澄清问题。

#### C. 澄清问题模板库的大部分内容

例如：

- “你说的发射模型是火箭、水下还是导弹？”
- “这是平台层动力学，还是战场层态势？”
- “你更关注轨迹传播，还是状态估计？”

原因：

- 大模型可以根据当前对话、证据和候选冲突，动态生成更贴切的澄清问题。
- 只要保留少量强制澄清的触发规则，问题文本本身不必全部预写死。

#### D. 候选机制扩展知识簇

例如：

- 某个系统可能包含哪些力项、事件、约束、损耗机制
- 哪些输入输出常常同时出现
- 哪些求解方式更可能合适

原因：

- 开放域建模的难点之一，不是“答案唯一”，而是“候选机制空间太大”。
- 强模型在这个环节非常有价值，适合用来生成候选项、补充假设、给出机制组合草案。

#### E. 初稿级方程片段和假设草案

例如：

- 基于重力、阻力、推力生成运动方程草案
- 给出状态量和观测量划分
- 生成初始条件与边界条件的待确认清单

原因：

- 这一步天然适合大模型：把自然语言需求转成半结构化草案。
- 但注意这里只是“草案”，不是最终可信的可执行模型。

### 3.2 结论

大模型最能替代的，是以下三类“高维护成本、低确定性、偏理解与候选生成”的知识簇：

- 长尾术语与别名
- 大部分通用背景知识
- 大部分澄清话术与候选机制扩展知识

这些内容过去往往靠不断扩知识簇来补，现在可以明显减少手工维护量。

## 4. 哪些知识簇绝不能替代

这些知识簇属于“高风险、强约束、可执行性核心资产”，必须保留为显式、可审计、可维护的系统能力。

### 4.1 绝不能替代的知识簇

#### A. 结构化 IR / Schema 契约

包括：

- `TaskIR / ModelIR / CodegenIR`
- family schema
- 参数槽位定义
- 必填字段、可选字段、字段类型
- 事件、约束、对象、状态的标准化表示

为什么不能替代：

- 大模型可以起草，但不能代替系统契约。
- 没有稳定 schema，后续代码生成、校验、trace 和回归就无法稳定运行。

#### B. 单位、量纲与参数范围规则

包括：

- 单位体系
- 量纲一致性
- 关键参数上下界
- 不合理组合的禁止规则

为什么不能替代：

- 大模型知道“常见值”，但不保证数值上严格一致。
- 一旦进入可执行求解，量纲错误和数量级错误会直接毁掉整个仿真。

#### C. 高风险反误配规则库

包括：

- 平台层动力学与战场层态势的分界
- 轨道动力学与大气层内飞行的分界
- 水下发射与火箭发射的分界
- 跟踪估计与一般轨迹传播的分界

为什么不能替代：

- 这些是系统可信性的底线。
- 大模型再强，也会在表述模糊、证据不足时被表面词汇诱导。
- 这类反误配规则必须显式保留。

#### D. 可执行代码组装知识簇

包括：

- `AssemblyPlan`
- `FAMILY_RENDER_RULES`
- `BLOCK_LIBRARY`
- 求解器映射
- 输出绘图与文件组织规范

为什么不能替代：

- 这是从“懂建模”走向“能生成能运行代码”的核心桥梁。
- 大模型可以提出代码草稿，但不适合作为唯一 renderer。
- renderer 必须可复现、可测试、可差分升级。

#### E. 校验器与运行闭环

包括：

- 静态校验
- smoke 校验
- 运行级验证
- 修复回路
- golden 回归集

为什么不能替代：

- 这是系统从“会说”到“能用”的本质差别。
- 预训练知识无法替代执行时的事实验证。

#### F. 工具契约与审批边界

包括：

- 工具输入输出 schema
- 危险操作审批
- 失败回退策略
- 人工确认节点

为什么不能替代：

- 开放域系统最终会接工具、接执行环境、接文件生成。
- 没有明确 tool contract，模型越强，越容易“能力过强但行为不可控”。

### 4.2 结论

绝不能被替代的不是“知识量”，而是“系统约束”。

必须保留为显式资产的，主要是：

- IR / schema
- 单位与数值规则
- 反误配 guardrails
- 代码组装规则
- 校验与回归体系
- 工具契约与审批机制

## 5. `Qwen3.5` 在当前项目中的最佳架构位置

## 5.1 不建议放的位置

不建议让 `Qwen3.5` 直接承担以下唯一职责：

- 最终 renderer
- 唯一 validator
- 唯一数值可信性判断器
- 替代所有 family schema
- 替代所有 `BLOCK_LIBRARY`

原因很简单：

- 这些位置要求稳定、可复现、可测试、可审计。
- 大模型适合做“候选生成与决策支持”，不适合当“唯一执行内核”。

## 5.2 最佳位置：前端决策层 + IR 草拟层 + 修复层

对当前项目，`Qwen3.5` 最适合放在以下三个层级：

### A. 开放域理解 / 路由增强层

职责：

- 判断用户请求是聊天、建模、澄清还是修复
- 识别跨域场景与 family 冲突
- 生成更合理的澄清问题
- 给出候选建模机制和候选 family

价值：

- 降低误配
- 降低“强行套最近模板”的概率
- 提前把高歧义请求导向 clarify

### B. `OpenModelIR` 草拟层

职责：

- 把自然语言需求映射成对象、状态、输入、输出、方程片段、事件、约束和缺失参数
- 生成 draft fragment
- 把现有 family 体系外的请求先纳入“半结构化草案”而不是直接失败

价值：

- 这是从半开放域走向开放域的关键增量。
- `Qwen3.5` 的知识广度和长上下文能力，更适合承担这一步。

### C. `ModelSpec` / IR 修复层

职责：

- 根据 schema 错误、验证错误和运行错误，做最小修复
- 提出缺失槽位、回填建议与更保守的默认值
- 把失败样本重新折返成结构化修复请求

价值：

- 这是大模型在工程系统中最稳定、回报最高的位置之一。
- 因为输入边界清晰、输出契约明确、收益可评估。

## 5.3 在当前项目中的推荐分工

结合当前代码结构，建议这样放置：

- `Qwen3.5`：
  - `task_planner` 的开放域增强版本
  - `OpenModelIR` 草拟器
  - `model_spec_builder` 中的 spec / repair 强模型
- 现有规则与组装层：
  - 保留 `AssemblyPlan`
  - 保留 `FAMILY_RENDER_RULES`
  - 保留 `BLOCK_LIBRARY`
  - 保留静态校验和 smoke 校验
- 后续新增：
  - 更正式的 tool schema
  - 运行错误到 IR 的回修环
  - eval / trace / regression

一句话说：

> `Qwen3.5` 最适合做“开放域理解器 + IR 规划器 + 修复器”，而不是“最终可执行建模系统的唯一内核”。

## 6. 建议的目标架构

建议把未来的开放域建模系统拆成七层：

1. **会话与状态层**
   - 保存上下文、澄清状态、修复状态、审批状态。
2. **检索与显式知识层**
   - 保留高价值知识簇、family schema、规则库、术语归一化和反误配规则。
3. **大模型理解与规划层**
   - 用 `Qwen3.5` 做意图理解、候选机制生成、澄清、`OpenModelIR` 草拟和修复。
4. **结构化 IR 层**
   - 固化为 `TaskIR / ModelIR / CodegenIR`。
5. **组装与渲染层**
   - 基于 `AssemblyPlan + BLOCK_LIBRARY` 输出代码。
6. **校验与回修层**
   - 静态、运行、结果合理性三类校验。
7. **评测与可观测层**
   - trace、回归集、误配集、运行成功率、澄清成功率。

## 7. 分阶段路线图

### 第一阶段：从“扩知识簇”转向“知识簇分层”

目标：不再追求无边界堆知识，而是区分“可交给模型的知识”和“必须显式保留的知识”。

动作：

- 缩减大而全的别名词典
- 保留高风险冲突词和反误配规则
- 把术语归一化从纯规则，改成“规则兜底 + LLM 归一化”

### 第二阶段：引入 `OpenModelIR` 的真正开放表达

目标：让模型能先生成结构化草案，而不是只能在固定 `model_id` 中硬选。

动作：

- 从“候选模板优先”升级为“IR 草拟 + 能落模板则落模板”
- 为对象、状态、事件、方程片段建立统一表示
- 把 draft fragment 从注释保留升级到可执行片段候选

### 第三阶段：让 `Qwen3.5` 成为开放域前端引擎

目标：把强模型放在最值钱的位置。

动作：

- 用 `Qwen3.5` 做开放域路由
- 用 `Qwen3.5` 做 `OpenModelIR` 草拟
- 用 `Qwen3.5` 做 spec / IR 修复
- 用更轻模型或规则做普通聊天和低风险分类

### 第四阶段：补齐执行闭环

目标：从“看起来会建模”变成“能稳定生成可运行模型”。

动作：

- 补单位校验
- 补参数范围校验
- 补求解器与事件一致性检查
- 补运行错误到 IR 的自动回修
- 建 golden 回归样本集

### 第五阶段：走向真正开放域

目标：让系统不仅能覆盖已有 family，还能以 IR 为中心扩展新 family。

动作：

- 新 family 先以 IR 草案接入
- 再沉淀为 schema
- 再沉淀为 block renderer
- 再纳入回归与验证体系

这样就不是“盲目增加知识簇”，而是“让模型先扩边界，再把稳定部分沉淀成系统资产”。

## 8. 对当前项目的直接建议

如果只给最务实的建议，我建议这样推进：

1. 保留现有 `OpenModelIR + AssemblyPlan + renderer + validator` 主线。
2. 不再以“无边界补知识簇”为第一策略，而是把知识簇分成“可替代”和“不可替代”两类。
3. 引入 `Qwen3.5` 到：
   - 开放域路由
   - `OpenModelIR` 草拟
   - `ModelSpec` / IR 修复
4. 不要让 `Qwen3.5` 直接替代：
   - schema
   - renderer
   - validator
   - guardrails
5. 先补 eval 与 trace，再扩大开放域覆盖。

## 9. 最终判断

### “大模型能替代哪些知识簇”

能替代的主要是：

- 长尾术语与别名
- 大部分通用背景知识
- 大部分澄清话术模板
- 候选机制扩展知识
- 初稿级方程片段与假设草案

### “哪些知识簇绝不能替代”

绝不能替代的主要是：

- `OpenModelIR` / schema / tool contract
- 单位、量纲、参数范围规则
- 高风险反误配规则
- `AssemblyPlan`、`BLOCK_LIBRARY`、renderer 规则
- 静态校验、运行校验、回归集与审批边界

### “`Qwen3.5` 在你这个项目里的最佳架构位置”

最佳位置是：

- 开放域理解器
- `OpenModelIR` 草拟器
- `ModelSpec` / IR 修复器

而不是：

- 最终代码渲染器
- 唯一验证器
- 所有知识簇的替代品
