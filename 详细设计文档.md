## 2. 仓库结构与模块边界

### 2.1 顶层结构
| 路径 | 主要职责 |
| --- | --- |
| `main.py` | CLI 入口 |
| `api/` | FastAPI 服务与可选 router |
| `agents/` | 编排、结构化建模、校验、代码生成、DLL 构建封装 |
| `knowledge_base/` | 模型目录、文档索引、主检索、历史联网证据检索、装配规划、block 渲染 |
| `tools/mcp_web_research/` | 联网搜索、网页抓取、结果包落盘、联网证据增量写入 |
| `tools/` | PowerShell 脚本与本地构建 MCP 工具链 |
| `database/` | SQLite CRM 数据层 |
| `tests/` | API、补槽、构建、动态库相关测试 |

### 2.2 当前主线
当前主链路是：

```text
自然语言请求
  -> 混合检索
  -> 任务规划
  -> 生成匹配评估
  -> object/family 澄清（如需要）
  -> StructuredGenerationIR 槽位收集
  -> OpenModelIR
  -> ModelSpec 兼容层
  -> AssemblyPlan
  -> family assembler / block renderer
  -> 静态校验
  -> 可选 smoke 校验
  -> generated_models/*.m
```

若用户显式请求 DLL / 动态库，则生成完成后继续进入本地 MCP 构建链。

若用户显式请求联网研究，则主链路会增加一个旁路子流程：
- 执行 Web 搜索与抓取
- 落盘 `generated_research/` 结果包
- 可选把成功网页证据写入独立 Qdrant collection
- 在后续同类请求中按会话检索历史网页证据并合并回运行时 `docs`

## 3. 入口与运行模式设计

### 3.1 CLI 入口
`main.py` 当前支持 4 种模式：
- `build`：构建知识索引；可选 `--with-qdrant`
- `run`：启动命令行交互模式
- `test`：执行 `CRMAgent.test_query()` 快速测试入口
- `api`：启动 FastAPI 服务

### 3.2 API 入口
`api/server.py` 中直接定义了默认 API：
- `GET /`
- `GET /ui`
- `GET /api/health`
- `GET /api/models`
- `POST /api/chat`
- `POST /api/query`

特点：
- 启动时通过 `startup_event()` 初始化全局 `CRMAgent`
- CORS 当前是全开放配置
- `GET /ui` 直接返回根目录静态文件 `web_ui.html`
- `POST /api/chat` 与 `POST /api/query` 最终都归一到 `CRMAgent`

### 3.3 可选 router 模块
`api/query/routes.py` 提供了 `APIRouter` 版本接口：
- 路由前缀：`/api`
- 具备 `set_agent()` 注入能力
- 当前默认服务未显式 `include_router()`，因此它属于**预留可拆分结构**，不是当前默认执行路径。

## 4. 配置设计

### 4.1 配置加载方式
`config/settings.py` 的配置加载顺序为：
- 优先使用 `pydantic-settings`
- 若缺失，则尝试 `pydantic.BaseSettings`
- 若仍不可用，则回退到内置轻量 `.env` 读取器

因此项目对配置依赖较温和，不会因为缺少 `pydantic-settings` 而直接失效。

### 4.2 关键默认项
默认值主要包括：
- LLM：`OLLAMA_BASE_URL`、`OLLAMA_MODEL`、`OLLAMA_TIMEOUT_SEC`、`OLLAMA_NUM_PREDICT`
- 检索：`EMBEDDING_MODEL`、`EMBEDDING_DEVICE`、`RETRIEVAL_VECTOR_BACKEND`、`RETRIEVAL_BM25_WEIGHT`、`RETRIEVAL_VECTOR_WEIGHT`、`RETRIEVAL_RERANK_BLEND`、`RETRIEVAL_CANDIDATE_MULTIPLIER`
- 联网研究：`WEB_RESEARCH_ENABLED`、`WEB_SEARCH_PROVIDER`、`WEB_SEARCH_MAX_RESULTS`、`WEB_FETCH_MAX_SOURCES`、`WEB_FETCH_TIMEOUT_SEC`、`WEB_CONNECT_TIMEOUT_SEC`、`WEB_FETCH_MAX_CHARS`、`WEB_RESEARCH_DIR`
- 联网证据增量索引：`WEB_RESEARCH_QDRANT_ENABLED`、`WEB_RESEARCH_QDRANT_COLLECTION`、`WEB_RESEARCH_QDRANT_SCOPE`、`WEB_RESEARCH_QDRANT_RETENTION_DAYS`、`WEB_RESEARCH_QDRANT_MAX_CHUNK_CHARS`、`WEB_RESEARCH_QDRANT_CHUNK_OVERLAP`、`WEB_RESEARCH_QDRANT_TOP_K`、`WEB_RESEARCH_QDRANT_CLEANUP_ON_WRITE`
- 修复：`MODEL_SPEC_REPAIR_MAX_ROUNDS`
- 会话：`SESSION_STORE_BACKEND`、`SESSION_HISTORY_SIZE`、`CHAT_HISTORY_WINDOW`、`FALLBACK_HISTORY_WINDOW`、`PLANNER_HISTORY_WINDOW`、`SESSION_TTL_SEC`
- Redis：`REDIS_HOST`、`REDIS_PORT`、`REDIS_DB`、`REDIS_PASSWORD`、`REDIS_KEY_PREFIX`
- API：`API_HOST`、`API_PORT`
- 本地构建：`LOCAL_BUILD_MCP_DRY_RUN`
- 数据库：`DATABASE_URL`

### 4.3 配置边界
- 代码默认 `LOCAL_BUILD_MCP_DRY_RUN=False`，即**配置层默认不要求 dry-run**。
- `MatlabCodegenTool` 内部对环境变量本身的兜底值是 `True`，但 `DynamicLibraryBuildTool` 在初始化时会通过 `ensure_runtime_env_defaults()` 把运行态环境补齐为配置值，因此实际行为以 `settings` 和环境变量为准。
- Docker 编排对部分超时、会话和检索配置进行了覆盖，运行时应区分“代码默认值”和“容器环境覆盖值”。

## 5. 会话与状态管理设计

### 5.1 SessionStore 抽象
`agents/session_store.py` 定义的统一协议包括：
- `get_history(session_id)`
- `append(session_id, role, content)`
- `clear(session_id)`
- `count(session_id)`
- `get_state(session_id, state_key)`
- `set_state(session_id, state_key, value)`
- `clear_state(session_id, state_key=None)`

### 5.2 后端实现
- `InMemorySessionStore`
  - 基于 `defaultdict + deque`
  - 适合开发、单进程调试
- `RedisSessionStore`
  - 历史使用 list
  - 状态使用 string JSON
  - 支持 TTL 与 key prefix
  - 适合多副本 API 场景

### 5.3 自动回退策略
`build_session_store()` 的行为：
- 当 `SESSION_STORE_BACKEND=redis` 时尝试初始化 Redis
- 若 Redis 依赖缺失、连接失败或 `ping()` 失败，则自动回退到 memory

### 5.4 当前保存的关键状态
在线流程中除了普通聊天历史，还维护以下会话态：
- `pending_generation_match`
  - object / family 级澄清挂起状态
- `pending_generation_ir`
  - slot 级补槽挂起状态
- `last_generation_result`
  - 最近一次生成成功结果，用于后续“把刚才那个模型编译成 DLL”这类请求

## 6. 知识库与检索系统设计

### 6.1 知识来源
当前知识来源由两类构成：
- 模型目录：`knowledge_base/matlab_model_data.py`
- 文档目录：`knowledge_base/docs/`

在此之上，运行时还存在一层**可选的联网补充证据层**：
- 本地结果包：`generated_research/`
- 独立网页证据向量库：`WEB_RESEARCH_QDRANT_COLLECTION`

这层证据不参与 `main.py build --with-qdrant` 的主知识库构建，而是由联网研究链路按需增量写入。

模型目录提供：
- `model_id`
- `template_family`
- `default_params`
- `equation_fragments`
- `domain_tags`
- 示例和说明文本

### 6.2 文档加载与切分
`knowledge_base/document_loader.py` 的行为：
- 扫描 `.md` / `.txt`
- 自动提取标题
- 采用字符窗口切分：
  - `chunk_size=480`
  - `chunk_overlap=80`
- 输出统一文档结构：`id + text + payload`

### 6.3 索引构建
`knowledge_base/builder.py` 中的 `KnowledgeBaseBuilder` 负责：
- 聚合模型目录与文档目录
- 生成本地 JSON 索引 `knowledge_base/matlab_knowledge_index.json`
- 在启用时初始化 Qdrant + SentenceTransformer
- 构建并写入 Qdrant collection

输出分层：
- 本地 JSON 索引：始终生成
- Qdrant 向量索引：可选生成

这里的 Qdrant 向量索引指的是**主知识库 collection**。当前结构中，联网证据 collection 由 `tools/mcp_web_research/qdrant_indexer.py` 单独维护，特点是：
- 只接收成功抓取网页的正文分块
- 独立 collection，不与主知识库混写
- 使用幂等 point id 做增量 upsert
- 可按保留期配置执行软 TTL 清理

### 6.4 检索文档类型
`MatlabRAGRetriever` 在线阶段会处理和合并多类文档：
- `model`
- `example`
- `document`
- `family_prototype`
- `family_example`

其中：
- `family_prototype` / `family_example` 不是单纯静态文件，而是由 retriever 结合 family 知识动态构造，用于提升 family 识别、对象澄清和补槽提示质量。

需要特别区分：历史联网证据**不在** `MatlabRAGRetriever` 主检索器内部处理。当前实现是由 `knowledge_base/web_evidence_retriever.py` 单独从独立 collection 中取回，再在 `CRMAgent` 层与实时联网证据、本地知识库证据做有序合并。

### 6.5 检索策略
当前检索策略是混合检索：
- BM25 词法检索
- 向量检索
- blend rerank

向量后端支持：
- `qdrant`
- `local`
- `auto`
- `none`

`auto` 模式行为：
- 优先尝试 Qdrant
- 失败时回退到本地向量
- 若 embedding 或向量后端整体不可用，则退化为纯 BM25

### 6.6 生成匹配评估
`MatlabRAGRetriever.assess_generation_match()` 会在检索结果上做第二次判定，核心输出字段包括：
- `should_generate`
- `reject_reasons`
- `clarify_stage`
- `top_candidate`
- `top_candidates`
- `top_family`
- `family_candidates`
- `suggestions`
- `guardrail`
- `trace`

评估目的不是直接生成代码，而是回答：
- 当前 family 是否足够明确
- 当前领域线索是否冲突
- 当前对象层是否需要先澄清
- 当前是否处于越界或低置信状态

### 6.7 检索健康检查
`get_retrieval_health()` 会返回：
- `hybrid_effective`
- `vector_ready`
- `configured_vector_backend`
- `active_vector_backend`
- `vector_error`
- `effective_bm25_weight`
- `effective_vector_weight`

因此 API 层能实时告诉调用方“当前是否真的跑在混合检索状态”。

## 7. 任务规划与生成门控设计

### 7.1 `RAGTaskPlanner`
`agents/task_planner.py` 中的 `RAGTaskPlanner.plan()` 同时使用：
- 规则规划 `_rule_based_plan()`
- 可选 LLM 规划 `_llm_plan()`

支持的任务类型：
- `chat`
- `matlab_generation`
- `matlab_generation_dll`
- `clarify`

合并策略：
- 若 LLM 给出了合法任务类型，则优先使用 LLM 结果
- 否则回退到规则规划结果

### 7.2 生成门控的两级结构
当前系统实际采用两级门控：
- 第一级：`RAGTaskPlanner` 决定“是不是生成型请求”
- 第二级：`MatlabRAGRetriever.assess_generation_match()` 决定“当前是否适合直接生成，还是要澄清”

这意味着即使规划器判断是生成请求，系统也不会盲目进入代码生成。

### 7.3 典型不直接生成场景
- `no_candidate`
- `domain_conflict`
- `ambiguous_family`
- `ambiguous_candidate`
- `out_of_scope`
- 对象不明确，如“做一个战场态势模型”但未说明关注对象或场景

### 7.4 澄清阶段划分
系统当前显式区分 3 类澄清阶段：
- `object`
- `family`
- `slot`

其中：
- `object` / `family` 主要由 `assess_generation_match()` 决定
- `slot` 主要由 `StructuredGenerationIR` 中的缺槽状态决定

## 8. `CRMAgent` 主编排设计

### 8.1 初始化依赖
`CRMAgent.__init__()` 主要实例化：
- `MatlabRAGRetriever`
- `RAGTaskPlanner`
- `ModelSpecBuilder`
- `ModelSpecValidator`
- `StructuredGenerationIR`
- `MatlabCodeGenerator`
- `MatlabFileGeneratorTool`
- `DynamicLibraryBuildTool`
- `WebResearchTool`
- `WebEvidenceRetriever`
- `SessionStore`

### 8.2 `chat()` 主流程
`CRMAgent.chat()` 是系统主入口，行为依次包括：
1. 校验空输入
2. 处理 `/new`、`/reset`、`/models`
3. 恢复并处理 `pending_generation_ir`
4. 恢复并处理 `pending_generation_match`
5. 判断是否是“对最近一次生成结果做 DLL 构建”的后续请求
6. 执行检索
7. 任务规划
8. 若 `request_web_research=True`，执行 Web Research 子链
9. 聊天或生成分支处理

其中 Web Research 子链当前包含：
- `_perform_web_research()`：搜索、抓取、落盘 `generated_research/` 结果包
- `WebResearchQdrantIndexer.index_sources()`：可选把成功网页证据分块写入独立 Qdrant collection
- `_retrieve_persisted_web_evidence()`：按当前 `session_id` 取回历史联网证据
- `_merge_runtime_docs()`：按“本次联网证据 -> 历史联网证据 -> 本地知识库证据”的顺序合并运行时 `docs`

### 8.3 挂起状态续接
当前系统支持两种恢复：
- `_resume_pending_generation_match()`：接续 object / family 级澄清
- `_resume_pending_generation_ir()`：接续 slot 级补槽

这让系统具备基本的多轮生成状态机，而不是“单轮问答式生成器”。

### 8.4 动态库后续请求
若用户在上次生成成功之后继续说：
- “把刚才那个编译成 DLL”
- “导出动态库”
- “继续构建动态库”

则 `CRMAgent` 会使用 `last_generation_result` 直接进入 `_handle_dynamic_library_followup()`，无需重新生成 `.m` 文件。

### 8.5 生成 trace 设计
`CRMAgent._build_generation_trace()` 会归一化以下信息：
- `event`
- `query_domains`
- `top_family`
- `family_top_share`
- `reject_reasons`
- `clarify_stage`
- `missing_slots`
- `should_generate`
- `final_generated`

trace 来源可同时吸收：
- `match_assessment.trace`
- `generation_ir.trace`
- `slot_collection`

## 9. StructuredGenerationIR 与 OpenModelIR 设计

### 9.1 当前定位
`StructuredGenerationIR` 不是简单 JSON 拼接器，而是一个三层结构化建模编排器：
- family schema registry
- family-aware slot extractor
- clarify policy

### 9.2 `begin_collection()`
当 family 初步明确后：
- 根据 `match_assessment.top_candidate` 或 `top_family` 解析目标 family
- 抽取槽位
- 合并默认值
- 构建 `generation_ir`
- 若缺槽则返回可继续澄清的 IR；若完整则进入 ready 状态

### 9.3 `continue_collection()`
在 slot 澄清阶段：
- 继续抽取补充参数
- 合并已收集槽位
- 重建 / 校正 `slot_collection`
- 最终输出标准化后的 `OpenModelIR`

### 9.4 `OpenModelIR` 结构
`agents/open_model_ir_schema.py` 当前定义了类型化 IR，主要区块包括：
- `task`
- `domain`
- `entities`
- `physics`
- `events`
- `constraints`
- `simulation`
- `outputs`
- `codegen`
- `trace`
- `slot_collection`

其中特别重要的是：
- `physics.parameters`
- `simulation.stop_time`
- `codegen.template_family`
- `codegen.equation_fragments`
- `slot_collection.missing_slots`

### 9.5 family schema 与槽位机制
`agents/structured_generation/schema_registry.py` 维护了 family blueprint：
- family 场景说明
- 识别槽位
- 关键槽位
- 可默认槽位
- 推荐槽位
- 单位、别名、描述

`FamilySlotExtractor` 支持：
- 具名参数抽取
- 带单位数字抽取
- 成对槽位抽取
- 顺序位置抽取

`FamilyClarifyPolicy` 负责：
- 判断是否需要继续澄清
- 生成追问话术
- 生成“按顺序回复”的提示模板
- 识别“使用默认值”或“取消”这类回复

### 9.6 当前落地程度
- `OpenModelIR` 已进入真实主链路
- `OpenModelIRValidator` 已用于 family 支持性、fragment 合法性和结构合法性校验
- 但 IR 的表达自由度仍受 family schema 和 fragment library 限制

## 10. `ModelSpec` 兼容层设计

### 10.1 当前角色
`ModelSpec` 已不是唯一中心结构，但仍承担 3 个作用：
- 兼容旧模板生成链
- 为新 IR 链提供落盘前中间规格
- 承接 `ModelSpecValidator` 自动修复流程

### 10.2 `ModelSpecBuilder`
`agents/model_spec_builder.py` 提供：
- LLM 规格构建
- 启发式规格构建
- LLM 修复函数 `repair_spec_with_llm()`

### 10.3 `ModelSpecValidator`
`agents/model_spec_validator.py` 提供：
- schema 校验
- 语义校验
- 自动修复回合控制

### 10.4 在生成链中的位置
`CRMAgent._handle_generation_intent()` 的当前顺序为：
- 若已有 `generation_ir`，则先 `to_model_spec()`
- 否则调用 `ModelSpecBuilder.build_spec()`
- 调用 `validate_with_auto_repair()`
- 若失败，再尝试启发式 spec 与二次修复

即便系统已引入 `OpenModelIR`，`ModelSpec` 仍是当前生成链的重要兼容关口。

## 11. AssemblyPlan 与组装式渲染设计

### 11.1 `AssemblyPlan`
`knowledge_base/assembly_plan.py` 定义了类型化装配计划，关键字段包括：
- `template_family`
- `governing_form`
- `solver`
- `state_variables`
- `equation_fragments`
- `fragment_defs`
- `parameters`
- `block_groups`
- `render_blocks`
- `domain`
- `assumptions`

### 11.2 `ModelPlanner`
`knowledge_base/model_planner.py` 的职责是把已验证的 `OpenModelIR` 映射成 `AssemblyPlan`：
- 校验 IR
- 解析 family
- 解析 fragment 计划
- 合并参数
- 生成 block 列表与 block group
- 产出 `AssemblyPlan`

### 11.3 fragment 规划
fragment 来源可能包括：
- `codegen.equation_fragments`
- `physics.equation_fragments`
- model 默认 fragments
- family 默认 fragments
- `physics.forces` 中推导出的 fragment

### 11.4 draft fragment 机制
当 fragment 没有现成渲染规则时：
- 仍可进入 `fragment_defs`
- `source` 标记为 `draft`
- `render_mode` 标记为 `comment_only`

其意义是：
- 规划层不会丢失用户提出的新物理片段
- 但渲染层当前不会把它自动转成原生求解代码，只会保留为注释占位或弱表达

## 12. Family Assembler 与 block renderer 设计

### 12.1 `MatlabFamilyAssembler`
`knowledge_base/model_family_codegen.py` 中的 `MatlabFamilyAssembler` 负责：
- 判断是否支持某个 family / model / IR / spec
- 从 spec 或 IR 生成 `AssemblyPlan`
- 根据 plan 生成 assembly
- 渲染最终 MATLAB 脚本

### 12.2 组装路径
当前支持两条组装路径：
- `render_from_ir(generation_ir, spec)`
- `render_from_spec(spec)`

内部最终都会归一到：
- `plan_from_ir()`
- `_assembly_from_plan()`
- `_render_assembly()`

### 12.3 block 模块化结构
`knowledge_base/blocks/` 提供：
- `BLOCK_LIBRARY`
- `FAMILY_RENDER_RULES`
- `FRAGMENT_RENDER_ORDER`

block 组按阶段组织，典型阶段包括：
- `setup`
- `declare`
- `solver`
- `fragment`
- `update`
- `output`
- `custom`

## 13. 代码生成与校验设计

### 13.1 `MatlabCodeGenerator`
`agents/matlab_codegen.py` 是统一生成入口，内部持有：
- `MatlabModelGenerator`（旧模板渲染器）
- `MatlabFamilyAssembler`（IR / family 组装渲染器）
- `OpenModelIRCompatAdapter`
- `MatlabStaticValidator`
- `MatlabSyntaxSmokeTester`

### 13.2 生成策略选择
`generate_from_spec()` 的选择顺序可以概括为：
- 若传入的是 `OpenModelIR`，先适配成 `ModelSpec`
- 若能解析出首选模板模型，则走模板渲染
- 否则若 family assembler 支持 IR，则走 `ir_composable_renderer`
- 否则若 family assembler 支持 spec，则走 spec 组装渲染
- 最后再退回旧模板渲染器

### 13.3 静态校验
`MatlabStaticValidator` 负责：
- 脚本是否包含可执行语句
- 基本结构、变量、保留字等问题检查

### 13.4 smoke 校验
`MatlabSyntaxSmokeTester` 支持：
- MATLAB runner
- Octave runner

相关行为：
- 通过 `MATLAB_SMOKE_REQUIRED` 控制是否“无 runner 也算失败”
- 默认找不到 runner 时返回 `skipped` 而不是强制失败
- 通过 `MATLAB_SMOKE_PREFERRED` / `MATLAB_SMOKE_RUNNER` 选择执行器

## 14. 本地动态库构建 MCP 设计

### 14.1 设计目标
本地动态库构建链的目标是：
- 不向上层暴露任意 shell
- 通过固定工具集合组织 MATLAB Coder + CMake + 产物检查
- 对作业目录和状态进行统一管理

### 14.2 调用入口
当前有两种使用方式：
- 进程内方式：`agents/tools.py` 中 `DynamicLibraryBuildTool` 直接实例化 `LocalBuildMCPServer`
- 标准 MCP 方式：直接运行 `tools/mcp_local_build/server.py`，通过 stdio JSON-RPC 交互

### 14.3 MCP 工具集合
`LocalBuildMCPServer` 当前注册的工具包括：
- `probe_toolchains`
- `create_build_job`
- `materialize_inputs`

- `cmake_configure`
- `cmake_build_dynamic`
- `cmake_build_static`
- `inspect_artifacts`

- `get_job_result`

其中：
- `cmake_build_static` 当前主要作为兼容别名存在
- `DynamicLibraryBuildTool` 实际主路径使用的是 `cmake_build_dynamic`

### 14.4 作业目录结构
每个作业创建在：`generated_builds/<job_id>/`

典型内容包括：
- `manifest.json`
- `result.json`
- `inputs/`
- `matlab/`

- `build/`

### 14.5 输入边界控制
`materialize_inputs()` 会调用 `_resolve_allowed_input()`，当前只允许：

这条边界是为了避免把任意绝对路径源文件拉入本地构建作业。

### 14.6 dry-run 与工具链探测
`MatlabCodegenTool` 和 `CMakeTool` 会基于环境变量判断是否执行真实外部工具。

相关环境变量包括：

## 18. 支持范围约束
- 当前支持模型族与宏观支持范围以 `项目架构说明.md` 为主，不在本文重复列出完整业务说明。
- 从实现角度看，可生成范围同时受以下三类资产共同约束：
  - `FAMILY_LIBRARY`
  - `schema_registry` 中的 family blueprint 与槽位定义
  - `FAMILY_RENDER_RULES` / `BLOCK_LIBRARY`
- 因此，“模型族已登记”不等于“一定具备完整原生渲染与运行验证能力”；仍需同时具备槽位 schema、fragment 规则和 block renderer 支撑。

## 19. 实现层设计边界
- 宏观边界以 `项目架构说明.md` 为主，本节仅补充实现层边界：
  - `api/query/routes.py` 已存在，但默认启动路径未挂载
  - `database/` 中的 SQLite CRM 模块未接入在线主生成链
  - draft fragment 目前主要以 `comment_only` 形式保留
  - 动态库构建主要通过 `CRMAgent` / `DynamicLibraryBuildTool` 触发，而非独立 HTTP 构建端点
  - `materialize_inputs()` 对输入路径做白名单约束，只接受仓库根目录及 `generated_models/` 内文件
  - 构建链能否真实执行仍取决于宿主机 MATLAB、CMake 和编译器工具链是否可用
  - 历史联网证据当前没有并入 `MatlabRAGRetriever` 主检索器，而是在 `CRMAgent` 层仅对 `request_web_research=True` 的请求做增量合并
  - `main.py build --with-qdrant` 只重建主知识库 collection，不负责重建独立联网证据 collection

## 20. 结论
- 当前项目已经从“RAG + 模板直出”的早期形态，演进为“检索门控 + 多阶段澄清 + 结构化 IR + 装配式代码生成 + 可选本地动态库构建”的多阶段系统。
- 其最重要的设计价值不在于“完全开放”，而在于：
  - 可检索
  - 可解释
  - 可澄清
  - 可回退
  - 可追踪
- 后续若继续演进，重点不应只是新增模板，而应继续提升：
  - family schema 覆盖度
  - OpenModelIR 的开放表达能力
  - fragment 到 native renderer 的落地率
  - 构建链与运行验证的闭环能力
