# 详细设计文档

## 1. 文档范围
- 本文档描述当前仓库已实现的运行链路、模块职责、接口形态、IR 设计、校验机制与部署方式。
- 本文档只描述**当前代码已落地**的能力，不描述尚未实现的理想终态。
- 当前系统的主目标不是通用开放域仿真，而是：在支持的模型族范围内，通过检索、澄清、结构化 IR 与 block 组装生成 MATLAB 脚本。

## 2. 当前技术栈

### 2.1 核心运行栈
- Python 3.10：来自 `environment.yml` 与 Docker `python:3.10-slim`
- FastAPI + Uvicorn：提供 HTTP API
- Pydantic：承担 API 模型、`OpenModelIR`、`AssemblyPlan` 的类型化定义
- Requests：用于访问 Ollama HTTP 接口

### 2.2 检索与模型能力
- Ollama：LLM 运行时
- 默认模型：`deepseek-r1:7b`
- Sentence Transformers：嵌入模型加载与向量编码
- 默认 embedding：`BAAI/bge-large-zh-v1.5`
- PyTorch：支撑 embedding 推理
- Qdrant：可选向量索引后端
- 自实现 BM25：词法检索基础层

### 2.3 校验与兼容能力
- `jsonschema`：`ModelSpec` JSON Schema 校验
- Pydantic typed schema：`OpenModelIR` 与 `AssemblyPlan` 校验
- `MatlabStaticValidator`：assembly / script 静态校验
- `MatlabSyntaxSmokeTester`：调用 MATLAB / Octave 进行语法级 smoke 校验

### 2.4 状态与部署能力
- Redis：可选会话状态存储
- 内存 SessionStore：Redis 不可用时自动回退
- Docker Compose：基础设施编排
- PowerShell 脚本：本地回归与环境启动辅助
- `tools/mcp_local_build/`：本地 stdio MCP Server，实现 MATLAB -> C/C++ -> CMake -> 动态库 的受限构建流水线

### 2.5 当前技术栈边界说明
- `LangChain` 虽存在于环境中，但当前主链路不依赖它。
- 配置层优先使用 `pydantic-settings`，缺失时回退到内置 `.env` 读取器。
- `database/` 中的 SQLite CRM 模块当前不参与在线生成主链路。

### 2.6 关键版本快照
- `requirements.txt` 当前显式声明的关键依赖为：
  - `fastapi==0.104.1`
  - `uvicorn[standard]==0.24.0`
  - `python-dotenv==1.0.0`
  - `qdrant-client==1.7.1`
  - `sentence-transformers==2.2.2`
  - `requests==2.31.0`
  - `torch==2.8.0`
  - `jsonschema==4.23.0`
  - `redis==5.0.1`
- `environment.yml` 当前声明的本地 Conda 运行时为 `Python 3.10.19`。
- `docker-compose.yml` 当前使用的关键基础设施镜像为：
  - `redis:5.0.14`
  - `qdrant/qdrant:v1.7.4`
  - `ollama/ollama:latest`

## 3. 系统总体结构

### 3.1 顶层结构
- 入口层：`main.py`、`api/server.py`
- 编排层：`agents/crm_agent.py`
- 检索层：`knowledge_base/rag_retriever.py`
- 结构化建模层：`agents/structured_generation_ir.py` + `agents/structured_generation/`
- 兼容规格层：`agents/model_spec_builder.py`、`agents/model_spec_validator.py`
- IR / 组装规划层：`agents/open_model_ir_*`、`knowledge_base/model_planner.py`、`knowledge_base/assembly_plan.py`
- 代码生成层：`agents/matlab_codegen.py`、`knowledge_base/model_family_codegen.py`、`knowledge_base/blocks/`
- 本地工具与构建层：`agents/tools.py`、`tools/mcp_local_build/`
- 校验层：`knowledge_base/matlab_static_validator.py`、`knowledge_base/matlab_smoke_tester.py`

### 3.2 当前架构主线
当前主线已经演进为：

`用户请求 -> 检索 -> 任务规划 -> 生成匹配门控 -> StructuredGenerationIR -> OpenModelIR -> ModelSpec兼容层 -> AssemblyPlan -> assembly -> block renderer -> 静态校验 -> 落盘 -> smoke校验`

其中：
- `ModelSpec` 仍然保留，但已不再是唯一中心结构。
- `OpenModelIR` 和 `AssemblyPlan` 已经进入主实现路径。
- 真正的代码渲染仍依赖 family / fragment / block 三套受限知识。
- 若显式调用动态库构建能力，则 `.m` 文件生成完成后还可进入 `DynamicLibraryBuildTool -> LocalBuildMCPServer -> generated_builds/<job_id>/` 的后置构建链路。

## 4. 入口与运行模式

### 4.1 CLI 入口
`main.py` 当前支持 4 种模式：
- `build`：构建知识索引
- `run`：CLI 交互模式
- `test`：快速测试入口
- `api`：启动 FastAPI 服务

### 4.2 API 入口
`api/server.py` 暴露以下接口：
- `GET /`
- `GET /ui`
- `GET /api/health`
- `GET /api/models`
- `POST /api/chat`
- `POST /api/query`

说明：
- `POST /api/chat` 使用字段 `message`
- `POST /api/query` 使用字段 `question`
- 两个接口最终都调用 `CRMAgent`
- `GET /api/health` 会返回 agent 初始化状态、session backend 和 retrieval health

### 4.3 可选路由模块
- `api/query/routes.py` 提供可拆分 router 版本
- 当前默认启动路径未显式挂载该 router，而是在 `api/server.py` 中直接定义了接口

## 5. 会话与状态管理设计

### 5.1 SessionStore 抽象
`agents/session_store.py` 定义了统一协议：
- `get_history()`
- `append()`
- `clear()`
- `count()`
- `get_state()`
- `set_state()`
- `clear_state()`

### 5.2 两种后端
- `InMemorySessionStore`
  - 基于 `defaultdict + deque`
  - 适合单机开发和调试
- `RedisSessionStore`
  - 基于 Redis list + string
  - 支持 TTL
  - 适合 API 实例多副本运行

### 5.3 自动回退策略
- `build_session_store()` 读取 `SESSION_STORE_BACKEND`
- 当值为 `redis` 时尝试初始化 Redis 后端
- 若 Redis 包缺失、连接失败或 ping 失败，则自动回退到 memory

### 5.4 会话中保存的状态
当前除普通聊天历史外，还保存两类关键挂起状态：
- `pending_generation_match`
  - 用于 object / family 级澄清
- `pending_generation_ir`
  - 用于 slot 级补槽续接

## 6. 知识库与检索系统设计

### 6.1 知识来源
知识库由两部分组成：
- `knowledge_base/matlab_model_data.py` 中维护的模型目录
- `knowledge_base/docs/` 下的 Markdown / Text 文档

### 6.2 文档加载与切分
`knowledge_base/document_loader.py` 的行为：
- 扫描 `.md` / `.txt`
- 自动识别标题
- 以字符窗口做 chunking
- 默认参数：
  - `chunk_size = 480`
  - `chunk_overlap = 80`

### 6.3 知识索引构建
`knowledge_base/builder.py` 负责：
- 整合模型目录与外部文档
- 生成本地索引 `knowledge_base/matlab_knowledge_index.json`
- 可选构建 Qdrant collection

其构建输出分为：
- 本地 JSON 索引：始终生成
- Qdrant 向量索引：显式启用时生成

### 6.4 检索文档类型
`MatlabRAGRetriever` 在线构建 / 合并以下文档类型：
- `model`
- `example`
- `document`
- `family_prototype`
- `family_example`

其中：
- `family_prototype` 与 `family_example` 由 retriever 基于 family schema 动态合成
- 这些文档用于增强 family 识别、对象澄清和槽位意图理解

### 6.5 检索策略
`knowledge_base/rag_retriever.py` 采用混合检索：
- BM25 词法检索
- 向量检索
  - 优先 Qdrant
  - Qdrant 不可用时回退到本地向量
- blend rerank

相关权重来自配置：
- `RETRIEVAL_BM25_WEIGHT`
- `RETRIEVAL_VECTOR_WEIGHT`
- `RETRIEVAL_RERANK_BLEND`
- `RETRIEVAL_CANDIDATE_MULTIPLIER`

### 6.6 向量后端策略
向量后端支持：
- `qdrant`
- `local`
- `auto`
- `none`

`auto` 模式下：
- 优先尝试 Qdrant
- 若失败则自动切到本地向量检索
- 若 embedding / vector backend 整体不可用，则退化为纯 BM25

### 6.7 检索健康检查
`get_retrieval_health()` 当前返回：
- `hybrid_effective`
- `vector_ready`
- `configured_vector_backend`
- `active_vector_backend`
- `vector_error`
- `effective_bm25_weight`
- `effective_vector_weight`

因此：
- API 层可以实时知道当前是否真正跑在 hybrid retrieval 上
- `CRMAgent` 启动时也会打印 startup check 日志

## 7. 任务规划与生成门控设计

### 7.1 `RAGTaskPlanner`
`agents/task_planner.py` 的基本策略是：
- 规则规划先给出基线判断
- LLM 规划在结果合法且可信时覆盖规则结果
- 否则回退到规则结果

这种设计保证：
- 有可解释性
- 有兜底
- 不完全依赖 LLM 稳定输出 JSON

### 7.2 生成匹配评估
`assess_generation_match()` 是当前防误生成功能的关键门控器，输出字段包括：
- `should_generate`
- `reason`
- `reject_reasons`
- `top_candidate` / `top_candidates`
- `top_family` / `family_candidates`
- `query_domains`
- `matched_domains`
- `conflicting_domains`
- `suggestions`
- `trace`

### 7.3 不直接生成的典型场景
- `no_candidate`
- `domain_conflict`
- `low_confidence`
- `ambiguous_family`
- `ambiguous_candidate`
- 上层领域守卫判定不匹配
- out-of-scope 守卫触发

### 7.4 澄清分级
当前澄清分为三层：
- `object`
  - 先确认对象 / 场景是否清晰
- `family`
  - family 冲突或家族不明确时确认家族
- `slot`
  - family 已明确，但关键参数缺失时继续补槽

## 8. `CRMAgent` 主编排设计

### 8.1 初始化依赖
`CRMAgent.__init__()` 当前会初始化：
- `MatlabRAGRetriever`
- `RAGTaskPlanner`
- `ModelSpecBuilder`
- `ModelSpecValidator`
- `StructuredGenerationIR`
- `MatlabCodeGenerator`
- `MatlabFileGeneratorTool`
- `DynamicLibraryBuildTool`
- `SessionStore`

### 8.2 `chat()` 主流程
核心顺序为：
1. 处理空输入与重置指令
2. 检查 `pending_generation_match`
3. 检查 `pending_generation_ir`
4. 进行检索
5. 调用 planner 决策
6. 进入聊天或生成链路

### 8.3 聊天链路
- 对话上下文由 SessionStore 提供
- 历史窗口由：
  - `CHAT_HISTORY_WINDOW`
  - `FALLBACK_HISTORY_WINDOW`
  控制
- 最终通过 Ollama 生成回复

### 8.4 生成链路
- 先做 `assess_generation_match()`
- 若需要 object / family 级澄清，则写入 `pending_generation_match`
- 若 family 已明确，进入 `StructuredGenerationIR.begin_collection()`
- 若仍缺槽，写入 `pending_generation_ir`
- 若收集完成，则继续生成 `ModelSpec`、校验并代码生成

### 8.5 兼容兜底
- 新链路失败时，仍可回退到旧 `MatlabFileGeneratorTool` / `matlab_generator.py` 模板路径
- 因此当前系统是“新链路优先、旧链路兜底”的兼容架构

### 8.6 动态库构建分支
- `CRMAgent` 额外暴露 `build_dynamic_library()`，内部直接调用 `DynamicLibraryBuildTool._run()`。
- 该分支不走 `chat()` 的常规对话编排，而是面向“已有 `.m` 文件 + 入口函数 + 入口参数 schema”的显式构建请求。
- 当前适配器位于 `agents/tools.py`，其职责是把多步 MCP 工具调用封装成一次端到端构建请求，并统一汇总 `probe`、`job_status`、`job_result` 与 `pipeline_steps`。

## 9. StructuredGenerationIR 与 OpenModelIR 设计

### 9.1 当前定位
`StructuredGenerationIR` 不是单纯的数据结构，而是一个 orchestrator：
- 协调 family schema registry
- 协调 slot extractor
- 协调 clarify policy
- 最终输出经过校验和标准化的 `OpenModelIR`

### 9.2 `generation_ir` / `OpenModelIR` 的主要区块
当前生成的结构包含：
- `task`
- `domain`
- `entities`
- `physics`
- `events`
- `constraints`
- `simulation`
- `outputs`
- `codegen`
- `slot_collection`
- `trace`
- `missing_info`

### 9.3 family schema 设计
当前 schema 以 family 为中心，而不是以单个 `model_id` 为中心。

每个 family schema 统一包含：
- `display_name`
- `scene`
- `required_slots`
- `recommended_slots`
- `slot_defs`
- `assumptions`
- `required_outputs`

### 9.4 槽位抽取与续接
- 首轮请求：`begin_collection()`
- 多轮补槽：`continue_collection()`
- 默认值策略：由 `clarify_policy` 决定何时自动填充
- 澄清消息：`build_clarify_message()` 生成

### 9.5 OpenModelIR 落地状态
当前代码中已落地：
- `OpenModelIR` Pydantic schema
- `OpenModelIRBuilder`
- `OpenModelIRValidator`
- `OpenModelIRCompatAdapter`

因此，“OpenModelIR 尚未落地”这一说法已经不再准确。更准确的表述应为：
- `OpenModelIR` 已经落地并进入当前实现链路
- 但 IR 的来源仍以 family 级 schema 驱动为主，不是任意开放域自动建模

## 10. `ModelSpec` 兼容层设计

### 10.1 当前角色
`ModelSpec` 目前是兼容规格层，而不是系统未来唯一中心。

它仍然承担以下职责：
- 兼容旧模板生成器
- 承接 LLM / heuristic 规格生成
- 为老链路保留稳定输入格式

### 10.2 `ModelSpecBuilder`
`agents/model_spec_builder.py` 的策略是：
- 先根据检索结果推断候选模型
- 尝试使用 LLM + RAG 证据构建 `ModelSpec`
- 若失败则回退到 heuristic 构建
- 最后补默认参数并写入 `_candidate_models`、`_build_source`

### 10.3 `ModelSpecValidator`
`agents/model_spec_validator.py` 会执行：
- JSON Schema 校验
- 模型相关语义校验
- 数值字段标准化
- 必要时 LLM repair
- repair 失败后 heuristic recover

### 10.4 当前关系
当前主链路中：
- `OpenModelIR` 更接近真实内部语义结构
- `ModelSpec` 主要承担兼容与修复职责

## 11. AssemblyPlan 与代码组装设计

### 11.1 `AssemblyPlan`
`knowledge_base/assembly_plan.py` 定义了以下核心对象：
- `AssemblyFragmentDef`
- `AssemblyOutputs`
- `BlockGroup`
- `AssemblyPlan`

`AssemblyPlan` 当前包含的关键字段包括：
- `template_family`
- `governing_form`
- `solver`
- `state_variables`
- `equation_fragments`
- `fragment_defs`
- `state_equations`
- `parameters`
- `outputs`
- `block_groups`
- `render_blocks`
- `assumptions`

### 11.2 `ModelPlanner`
`knowledge_base/model_planner.py` 负责：
- 验证 OpenModelIR
- 解析 family
- 解析 equation fragments
- 合并参数
- 生成 `render_blocks`
- 生成 `block_groups`
- 输出 `AssemblyPlan`

### 11.3 draft fragment 机制
当 IR 中包含以下情况时，planner 允许创建 draft fragment：
- 片段在 fragment library 中不存在
- 物理组件无法映射到已有原生片段
- 只有 state equation，没有可复用 fragment

此时：
- draft fragment 会进入 `fragment_defs`
- `source` 标记为 `draft`
- `render_mode` 标记为 `comment_only`

这意味着：
- 系统能保留语义痕迹
- 但不意味着系统已经拥有该片段的可执行 MATLAB 原生 renderer

## 12. Family Assembler 与 block renderer 设计

### 12.1 `MatlabFamilyAssembler`
`knowledge_base/model_family_codegen.py` 当前负责：
- 支持 family 判断
- `plan_from_ir()` / `plan_from_spec()`
- `render_from_ir()` / `render_from_spec()`
- assembly 到 MATLAB script 的 block 级渲染

### 12.2 领域知识库
当前核心静态知识包括：
- `FAMILY_LIBRARY`
- `FRAGMENT_LIBRARY`
- `FAMILY_PARAMETER_DEFAULTS`
- `FAMILY_RENDER_RULES`
- `FRAGMENT_RENDER_ORDER`

### 12.3 block 模块化结构
`knowledge_base/blocks/` 当前拆分为：
- `common.py`
- `aerospace.py`
- `underwater.py`
- `orbital.py`
- `tracking.py`
- `battlefield.py`

这种拆分的目的：
- 降低单文件复杂度
- 支持按领域扩展 block 库
- 让 render rule 和 block renderer 分层维护

### 12.4 渲染顺序
当前渲染过程本质上是：
1. 从 `AssemblyPlan` 取出 `render_blocks`
2. 根据 block id 的 stage 切为 block groups
3. 使用 `BLOCK_LIBRARY` 逐 block 渲染
4. 拼接成完整 MATLAB 脚本

## 13. 代码生成与校验设计

### 13.1 `MatlabCodeGenerator`
统一入口：
- `generate_from_open_model_ir()`
- `generate_from_spec()`

主要逻辑：
1. 判断输入是否已是 `OpenModelIR`
2. 若是，则先做兼容转换
3. 优先尝试 `MatlabFamilyAssembler.render_from_ir()`
4. 若 family 路径失败，回退旧模板生成器
5. 对最终脚本做静态校验
6. 保存文件到 `generated_models/`
7. 对落盘文件执行 smoke 校验

### 13.2 静态校验
`MatlabStaticValidator` 负责：
- assembly 级检查
- render blocks 合法性检查
- 参数与变量一致性检查
- 控制结构平衡检查
- 括号平衡检查
- 最终脚本标识符检查

### 13.3 smoke 校验
`MatlabSyntaxSmokeTester` 的策略：
- 优先检测 MATLAB runner
- 否则尝试 Octave runner
- 若都不存在：
  - 默认返回 `skipped`
  - 若 `MATLAB_SMOKE_REQUIRED=true`，则返回失败

相关环境变量包括：
- `MATLAB_SMOKE_TIMEOUT_SEC`
- `MATLAB_SMOKE_REQUIRED`
- `MATLAB_SMOKE_PREFERRED`
- `MATLAB_SMOKE_RUNNER`
- `MATLAB_EXE`

### 13.4 本地动态库构建 MCP 设计
当前仓库已落地一套最小可用的本地构建 MCP，实现位于 `tools/mcp_local_build/`，并由 `agents/tools.py` 中的 `DynamicLibraryBuildTool` 做上层适配。

核心组成如下：
- `server.py`
  - 提供 `LocalBuildMCPServer`
  - 支持 `initialize`、`ping`、`tools/list`、`tools/call`
  - 采用 JSON-RPC over stdio 读写 `Content-Length` 报文
  - 当前 `serverInfo.name` 为 `local-build-mcp`，版本为 `0.1.0`
- `schemas.py`
  - 为所有工具定义 `inputSchema`
  - 通过 `jsonschema` 做参数校验；缺失时回退到内置校验器
- `job_manager.py`
  - 负责生成 `generated_builds/<job_id>/` 作业空间
  - 固定目录包括：`inputs/`、`matlab/`、`src/`、`build/`、`artifacts/`、`logs/`
  - 持久化 `manifest.json` 与 `result.json`
  - 维护 step 级时间线、错误摘要和最终产物信息
- `toolchains.py`
  - 探测 `MATLAB`、`CMake`、`Visual Studio`、`gcc`、`g++`、`ar`
  - 支持的 profile 为：`windows_msvc_dll`、`windows_gcc_dll`、`linux_gcc_shared`
  - 自动给出 `recommended_profile`
- `matlab_codegen_tool.py`
  - `materialize_inputs()` 复制 `.m` 文件并写入 `build_request.json`
  - 输入路径被限制在仓库根目录与 `generated_models/` 下，避免任意文件访问
  - `matlab_generate_cpp()` 会生成 `matlab_codegen_runner.m` 与 `matlab_command.json`
  - 通过 `matlab -batch run(...)` 调 MATLAB Coder 产出 `matlab/codegen/` 下的 C/C++ 源文件
- `cmake_tool.py`
  - 基于模板渲染 `src/CMakeLists.txt`
  - 自动补 `wrapper.h` / `wrapper.cpp`
  - 依次执行 `cmake_configure()` 与 `cmake_build_dynamic()`
  - 构建目标固定为动态库，显式传入 `-DBUILD_SHARED_LIBS=ON`
- `artifact_tool.py`
  - 从 `build/` 中收集 `.dll` / `.so` / `.dylib` 以及可能生成的 `.lib` / `.a` 导入库
  - 复制最终库文件到 `artifacts/`
  - 汇总头文件并写回 `result.json`

当前工具集合共 10 个：
- `probe_toolchains`
- `create_build_job`
- `materialize_inputs`
- `matlab_generate_cpp`
- `cmake_configure`
- `cmake_build_dynamic`
- `cmake_build_static`（兼容别名）
- `inspect_artifacts`
- `get_job_status`
- `get_job_result`

标准执行顺序为：
1. `probe_toolchains`
2. `create_build_job`
3. `materialize_inputs`
4. `matlab_generate_cpp`
5. `cmake_configure`
6. `cmake_build_dynamic`
7. `inspect_artifacts`

其中：
- 若某一步返回 `planned`，上层适配器不会继续执行产物检查，而是返回 `next_action_hint`。
- 若某一步返回 `failed`，错误会同步写入 `manifest.json` 与 `result.json`。
- 当前实现中，`DynamicLibraryBuildTool` 默认直接实例化 `LocalBuildMCPServer` 并在进程内调用；但 `server.py` 仍保留了标准 stdio MCP 入口，便于后续被外部 MCP Client 拉起。

### 16.3 本地 MCP 运行模式
当前本地构建能力支持两种运行方式：
- 进程内调用：`agents/tools.py` 直接实例化 `LocalBuildMCPServer`，适合当前仓库内的 Python 侧编排。
- 标准 stdio MCP：直接运行 `tools/mcp_local_build/server.py`，由外部 MCP Client 通过 stdio JSON-RPC 连接。

无论采用哪种方式，构建产物与日志都统一写入 `generated_builds/<job_id>/`，因此运行方式不会改变作业目录结构。

## 18. 当前设计边界
- `OpenModelIR` 已经落地，但 IR 仍主要由 family schema 驱动，不是通用开放域自由建模器。
- `AssemblyPlan` 已经落地，但最终渲染仍依赖 `FAMILY_RENDER_RULES` 与 `BLOCK_LIBRARY`。
- draft fragment 已支持进入计划层，但尚不等价于“系统能自动生成该物理片段的可执行求解代码”。
- smoke 校验已接入，但能否真正执行依赖本机 MATLAB / Octave runner 是否存在。
- 本地动态库构建能力已经实现，但当前默认不通过 HTTP API 暴露独立端点，而是通过 `CRMAgent.build_dynamic_library()` / `DynamicLibraryBuildTool` 显式调用。
- 本地 MCP 构建默认开启 `LOCAL_BUILD_MCP_DRY_RUN`，因此在未关闭 dry-run 前，作业通常停留在 `planned` / `running` 观察态，而不会真正执行外部 MATLAB / CMake。
- `matlab_codegen_mode` 当前已进入接口 schema，但实际代码生成实现统一走 MATLAB Coder 的 `coder.config('dll') + codegen` 路径，尚未分化出独立的 Simulink Coder 分支。
- `materialize_inputs()` 当前只允许访问仓库根目录及 `generated_models/` 下的 `.m` 文件，这是有意保留的输入边界，不支持任意绝对路径源码。
- `database/` 中的 SQLite CRM 模块当前没有纳入在线请求主路径。

## 19. 结论
- 当前项目已经从“RAG + 模板”的单阶段系统，演进为“检索门控 + family 级槽位收集 + OpenModelIR + AssemblyPlan + block renderer + 双层校验”的多阶段架构。
- 当前项目的正确表述应当是：**已具备结构化 IR 和组装式代码生成能力的半开放域 MATLAB 建模生成系统**。
- 后续若要继续向开放域演进，重点不再是简单增加模板，而是：
  - 扩大 family schema 覆盖面
  - 提升 OpenModelIR 的开放表达能力
  - 将 draft fragment 从“注释保留”升级为“可执行渲染”
  - 建立更稳定的运行级验证闭环
