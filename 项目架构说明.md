# 项目架构说明

## 1. 项目定位
- 本项目是一个面向 MATLAB `.m` 脚本生成的对话式建模系统，核心目标是把自然语言建模需求转换为可执行脚本。
- 当前系统的准确定位是：**带检索约束、带澄清门控、带结构化 IR 的半开放域建模生成系统**。
- 相比早期“按 `model_id` 直接套模板”的实现，当前版本已经落地了：
  - 检索与生成匹配门控
  - family 级结构化槽位收集
  - `OpenModelIR` 类型化 schema / builder / validator
  - `AssemblyPlan -> assembly -> block renderer` 代码生成链路
  - 静态校验与可选 MATLAB / Octave smoke 校验
- 当前仓库还内置一条可选后置能力：把已生成或已有的 MATLAB 函数通过本地 MCP 构建链转换为动态库产物。
- 但系统仍未达到真正的开放域自动建模：当前 IR 生成、片段复用和代码渲染，仍然受 family schema、片段库与 block 库约束。

## 3. 架构原则
- **检索优先**：先做知识检索和候选 family / model 推断，再决定是否进入生成链路。
- **先门控、再生成**：在 `no_candidate`、`domain_conflict`、`ambiguous_family`、`ambiguous_candidate`、`low_confidence` 等场景下，不直接生成代码。
- **结构化优先**：当前生成主链已从“模板直出”升级为 `OpenModelIR / ModelSpec -> AssemblyPlan -> block renderer -> script`。
- **兼容优先**：`ModelSpec` 仍保留，作为旧链路和新链路之间的兼容层。
- **校验前置**：在真正落盘 `.m` 文件之前，先做 IR 校验、assembly 校验、脚本静态校验，再做可选 smoke 校验。
- **工具边界收敛**：动态库构建通过受限 MCP 工具集合暴露，不向上层开放任意 shell。
- **基础设施解耦**：应用层与 Redis / Qdrant / Ollama 解耦，可本机混合部署，也可全 Docker 部署。

## 4. 部署架构

### 4.1 推荐模式：混合部署
- Docker 负责基础设施：`redis`、`qdrant`、`ollama`、`ollama-init`。
- Python 应用层在本机 Conda 环境运行：
  - `python main.py build --with-qdrant`
  - `python main.py api`
- 本地动态库构建能力可由 Python 侧进程内直接实例化 `LocalBuildMCPServer`，也可按标准 stdio MCP 模式被外部客户端拉起。
- 这种模式更适合当前仓库，原因是：
  - 便于复用本地 Python 环境与模型缓存
  - 更便于调试知识库与生成链路
  - 更容易观察 Qdrant、Ollama 与应用日志

### 4.2 可选模式：全 Docker 部署
- `docker-compose.yml` 当前包含以下服务：
  - `redis`
  - `qdrant`
  - `ollama`
  - `ollama-init`
  - `kb-build`
  - `app1`
  - `app2`
- 其中：
  - `ollama-init` 用于自动拉取默认模型
  - `kb-build` 用于检测 / 重建 Qdrant 索引
  - `app1` / `app2` 用于多实例 API 部署

## 5. 分层架构

### 5.1 入口层
- `main.py`
  - `build`：构建本地知识索引，可选构建 Qdrant
  - `run`：启动 CLI 交互模式
  - `test`：运行快速测试入口
  - `api`：启动 FastAPI 服务
- `api/server.py`
  - 提供 `GET /`、`GET /ui`、`GET /api/health`、`GET /api/models`
  - 提供 `POST /api/chat`、`POST /api/query`
  - 当前尚未单独暴露动态库构建 HTTP 接口
- `api/query/routes.py`
  - 预留的可拆分 router 模块
  - 当前默认启动流程仍以 `api/server.py` 直接定义的路由为主

### 5.2 编排层
- `agents/crm_agent.py`
  - 全局编排入口
  - 负责对话、检索、意图分流、澄清续接、规格生成、代码生成以及显式动态库构建调用
- `agents/task_planner.py`
  - 用规则 + LLM 规划 `chat / matlab_generation / clarify`
- `agents/session_store.py`
  - 封装内存和 Redis 两类会话后端
  - 管理历史消息与挂起状态
- `agents/tools.py`
  - 提供 `MatlabFileGeneratorTool`、`DynamicLibraryBuildTool` 等工具适配器
  - 负责把多步 MCP 调用封装为一次动态库构建请求

### 5.3 结构化建模层
- `agents/structured_generation_ir.py`
  - family 级结构化收集编排器
  - 负责生成 / 更新 `generation_ir`
  - 负责把收集结果落入 `OpenModelIR`
- `agents/structured_generation/schema_registry.py`
  - 维护 family schema、槽位定义与展示信息
- `agents/structured_generation/slot_extractor.py`
  - 从自然语言抽取 family-aware 槽位
- `agents/structured_generation/clarify_policy.py`
  - 负责是否澄清、默认值填充、补槽消息生成与回复合并

### 5.4 IR、规划与兼容层
- `agents/open_model_ir_schema.py`
  - 定义 `OpenModelIR` typed schema
- `agents/open_model_ir_builder.py`
  - 负责 OpenModelIR 标准化构建
- `agents/open_model_ir_validator.py`
  - 对 IR 做结构与语义校验
- `agents/open_model_ir_compat.py`
  - 提供 `OpenModelIR -> ModelSpec` 与 `OpenModelIR -> AssemblyPlan` 兼容转换
- `knowledge_base/assembly_plan.py`
  - 定义 `AssemblyPlan` 与 `BlockGroup`
- `knowledge_base/model_planner.py`
  - 把 validated IR 转成 `AssemblyPlan`
  - 负责片段复用、draft fragment 降级和 block group 构建

### 5.5 规格与校验层
- `agents/model_spec_builder.py`
  - 基于 RAG 证据 + LLM / heuristic 构建 `ModelSpec`
- `agents/model_spec_validator.py`
  - 做 JSON Schema 校验、语义校验、LLM repair 与 heuristic recover
- `agents/model_spec_schema.py`
  - 定义 `ModelSpec` JSON Schema

### 5.6 知识与检索层
- `knowledge_base/builder.py`
  - 构建本地 JSON 索引
  - 可选构建 Qdrant collection
- `knowledge_base/document_loader.py`
  - 负责 `knowledge_base/docs/` 文档加载与 chunking
- `knowledge_base/rag_retriever.py`
  - 提供混合检索、候选模型推断、family prototype 文档构造、生成匹配评估与健康检查
- `knowledge_base/matlab_model_data.py`
  - 管理模型目录、默认参数、领域标签与示例语料

### 5.7 代码生成层
- `agents/matlab_codegen.py`
  - 统一生成入口
  - 优先走 IR 组装式渲染，失败时回退旧模板生成器
- `knowledge_base/model_family_codegen.py`
  - 负责 family assembler
  - 内含 `FAMILY_LIBRARY`、`FRAGMENT_LIBRARY`、`FAMILY_PARAMETER_DEFAULTS`
- `knowledge_base/blocks/`
  - 提供分领域 block renderer 与 `FAMILY_RENDER_RULES`
- `knowledge_base/matlab_generator.py`
  - 保留旧模板式生成器，作为兼容与兜底
- `knowledge_base/matlab_static_validator.py`
  - 校验 assembly 与最终脚本
- `knowledge_base/matlab_smoke_tester.py`
  - 可选使用 MATLAB 或 Octave 做语法 smoke 校验

### 5.8 资源与持久化层
- `knowledge_base/matlab_knowledge_index.json`
  - 本地知识索引产物
- `generated_models/`
  - 最终 `.m` 文件输出目录
- `generated_builds/`
  - 本地动态库构建作业目录
  - 保存 `manifest.json`、`result.json`、日志与产物副本
- Redis
  - 会话历史与挂起状态存储
- Qdrant
  - 向量索引存储
- `crm_database.db` / `database/`
  - 历史 CRM 数据模块
  - 当前未接入主对话生成链路

### 5.9 本地构建工具层
- `tools/mcp_local_build/server.py`
  - 提供 `LocalBuildMCPServer`
  - 实现 `initialize`、`tools/list`、`tools/call` 等 MCP 基础协议处理
- `tools/mcp_local_build/schemas.py`
  - 维护 9 个构建工具的 schema 与参数校验
- `tools/mcp_local_build/job_manager.py`
  - 负责作业目录、`manifest.json`、`result.json` 与 step 时间线管理
- `tools/mcp_local_build/toolchains.py`
  - 探测 MATLAB、CMake、Visual Studio、gcc、g++、ar 与推荐 profile
- `tools/mcp_local_build/matlab_codegen_tool.py`
  - 负责输入落盘、MATLAB runner 生成和 C/C++ 代码生成
- `tools/mcp_local_build/cmake_tool.py`
  - 负责渲染 `CMakeLists.txt`、自动生成 wrapper、执行 configure/build
- `tools/mcp_local_build/artifact_tool.py`
  - 负责动态库、导入库与头文件收集，并回写最终结果

## 6. 核心数据流

### 6.1 知识构建流
1. `main.py build` 进入 `KnowledgeBaseBuilder`
2. 读取 `matlab_model_data.py` 中的模型目录
3. 读取 `knowledge_base/docs/` 下的 Markdown / Text 文档
4. 对文档做 chunking 并写入 `knowledge_base/matlab_knowledge_index.json`
5. 若启用 `--with-qdrant`，则额外构建 Qdrant 向量索引

### 6.2 在线请求流
1. API / CLI 请求进入 `CRMAgent.chat()` / `process_query()`
2. 检查会话状态：
   - `pending_generation_match`
   - `pending_generation_ir`
3. 若存在挂起状态，优先走续接分支
4. 否则进入 `MatlabRAGRetriever.retrieve()`
5. `RAGTaskPlanner` 决定当前请求属于：
   - 聊天
   - 生成
   - 澄清
6. 若是聊天，走 Ollama 对话链路
7. 若是生成，进入生成匹配门控与结构化建模链路

### 6.3 生成链路
1. `assess_generation_match()` 判断是否可以生成
2. 若需要澄清，生成 object / family / slot 级澄清消息并写入挂起状态
3. 若可以继续，`StructuredGenerationIR` 产出 `OpenModelIR`
4. 通过兼容层转成 `ModelSpec`，或直接转入 `AssemblyPlan`
5. `ModelSpecBuilder` / `ModelSpecValidator` 完成规格补全、校验与修复
6. `MatlabCodeGenerator` 调用 `MatlabFamilyAssembler`
7. `ModelPlanner` 把 IR 转成 `AssemblyPlan`
8. `AssemblyPlan` 展开为 `assembly`
9. `BLOCK_LIBRARY` 按 `render_blocks` 组装 MATLAB 脚本
10. 执行静态校验、落盘、可选 smoke 校验
11. 返回文件路径、脚本文本、assembly、validation 信息

### 6.4 本地动态库构建链路
1. 调用 `CRMAgent.build_dynamic_library()` 或 `DynamicLibraryBuildTool._run()`
2. `probe_toolchains` 探测本机工具链，并确定可用 profile
3. `create_build_job` 创建 `generated_builds/<job_id>/` 作业空间
4. `materialize_inputs` 复制 `.m` 文件并写入 `build_request.json`
5. `matlab_generate_cpp` 生成 `matlab_codegen_runner.m`，调用 MATLAB Coder 输出 C/C++ 代码
6. `cmake_configure` 渲染 `CMakeLists.txt`、wrapper 文件，并执行 configure
7. `cmake_build_dynamic` 编译动态库目标
8. `inspect_artifacts` 收集 `.lib` / `.a` 与头文件，回写 `result.json`
9. 上层适配器汇总 `pipeline_steps`、`job_status`、`job_result` 与 `next_action_hint`

说明：
- 若任何一步返回 `planned`，流水线会停在计划态，等待人工关闭 dry-run 或补齐工具链。
- 若中间步骤失败，`inspect_artifacts` 不再继续执行，错误由 `manifest.json` 与 `result.json` 持久化。

## 8. 当前设计边界
- `OpenModelIR` 与 `AssemblyPlan` 已经在当前代码中落地，但**仍是受 family schema 与 block 库约束的半开放域实现**。
- `ModelPlanner` 支持 draft fragment 进入 `AssemblyPlan`，但没有原生 block 支持的片段目前只能降级为 `comment_only` 或兼容路径，不能自动变成通用可执行动力学求解器。
- 当前主要渲染能力仍依赖：
  - `FAMILY_RENDER_RULES`
  - `BLOCK_LIBRARY`
  - `FRAGMENT_LIBRARY`
- MATLAB / Octave smoke 校验已经接入，但是否真正执行取决于本机是否存在 runner；默认 runner 不存在时会跳过，而不是强制失败。
- 本地动态库构建能力已经落地，但当前默认不通过 FastAPI 暴露独立 HTTP 端点，而是通过 `CRMAgent.build_dynamic_library()` / `DynamicLibraryBuildTool` 显式调用。
- 本地 MCP 构建默认启用 `LOCAL_BUILD_MCP_DRY_RUN`，因此当前很多作业首先表现为 `planned`，而不是直接执行外部 MATLAB / CMake。
- `matlab_codegen_mode` 当前虽在接口上支持 `matlab_coder` / `simulink_coder` 两种枚举，但实际实现统一走 MATLAB Coder 的 `codegen` 分支。
- `materialize_inputs` 对输入路径做了白名单限制，目前只接受仓库根目录与 `generated_models/` 下的 MATLAB 文件。
- `database/` 目录中的 CRM 数据模块当前不是主生成链路的一部分，应视为历史遗留或预留扩展模块。
