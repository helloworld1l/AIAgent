# 项目架构说明

## 1. 文档目的与依据
- 本文用于说明当前 `rag_crm_agent` 仓库的真实架构形态，重点回答“系统现在是怎样组织的、主链路怎么走、哪些能力已经落地、哪些仍然受限”。
- 本文是三份核心说明文档中的**宏观架构文档**，职责边界如下：
  - `README.md`：项目简介、快速启动、常用命令与文档导航
  - `项目架构说明.md`：系统分层、核心数据流、模块职责、部署形态与架构边界
  - `详细设计文档.md`：面向实现的模块设计、状态管理、IR、校验、构建链与接口返回
- 因此，本文只保留理解系统整体所需的宏观信息，不展开逐函数实现细节，也不重复完整操作手册内容。

## 2. 项目定位
- 本项目是一个**面向 MATLAB `.m` 脚本生成的对话式建模系统**。
- 当前系统的准确定位不是“任意物理系统都能自动建模”的完全开放域生成器，而是一个：
  - 带检索约束的建模生成系统
  - 带澄清门控的生成式助手
  - 带结构化 IR 的半开放域 MATLAB 建模平台
- 当前主链路已经从“命中模板后直接套代码”演进为：

```text
用户请求
  -> 检索与候选识别
  -> 任务规划
  -> 生成匹配评估与澄清门控
  -> StructuredGenerationIR / OpenModelIR
  -> ModelSpec 兼容层
  -> AssemblyPlan
  -> family assembler + block renderer
  -> MATLAB 脚本落盘与校验
```

- 在此基础上，项目还提供一条**可选后置能力**：把生成好的 `.m` 文件通过本地 MCP 构建链转为 DLL / 动态库产物。
- 但系统目前仍明显受限于 `family schema`、`fragment library`、`block library`，因此它是**受限开放、可解释、可门控**的建模系统，而不是自由开放域建模器。

- 在同类后续请求中按会话检索历史网页证据，再与本地知识库结果合并

## 5. 核心运行链路

### 5.1 知识构建链路
`python main.py build [--with-qdrant]` 会执行：
- 读取 `knowledge_base/matlab_model_data.py` 中的模型目录
- 读取 `knowledge_base/docs/` 中的 `.md` / `.txt` 文档
- 生成本地索引 `knowledge_base/matlab_knowledge_index.json`
- 在显式启用时构建 Qdrant 向量集合

这里的 `--with-qdrant` 仅构建**主知识库 collection**，不负责重建 `WEB_RESEARCH_QDRANT_COLLECTION` 对应的联网证据库。

### 5.2 在线请求链路
在线请求统一走 `CRMAgent.chat()`：
- 处理会话指令，如 `/new`、`/reset`、`/models`
- 恢复挂起澄清状态：
  - `pending_generation_match`
  - `pending_generation_ir`
  - `last_generation_result`
- 调用 `MatlabRAGRetriever.retrieve()` 获取证据文档
- 调用 `RAGTaskPlanner.plan()` 判定当前请求属于：
  - `chat`
  - `matlab_generation`
  - `matlab_generation_dll`
  - `clarify`
- 若 `request_web_research=True`：
  - 调用 `WebResearchTool` 执行搜索、抓取、结果包落盘
  - 可选将成功网页证据分块写入独立 Qdrant collection
  - 调用 `WebEvidenceRetriever` 读取历史联网证据
  - 按“本次联网证据 -> 历史联网证据 -> 本地知识库证据”的顺序合并运行时 `docs`

### 5.3 生成链路
当规划器认为用户意图是建模生成时：
- `MatlabRAGRetriever.assess_generation_match()` 对 family / model 候选做门控评估
- 若存在 `no_candidate`、`domain_conflict`、`ambiguous_family`、`out_of_scope` 等情况，则优先澄清
- 若门控通过，则进入 `StructuredGenerationIR.begin_collection()`
- 若缺关键槽位，则进入补槽续接；否则直接转 `OpenModelIR`
- `OpenModelIR` 经兼容适配后转换成 `ModelSpec`
- `ModelPlanner` 基于 IR 生成 `AssemblyPlan`
- `MatlabFamilyAssembler` 按 block 规则渲染脚本
- `MatlabCodeGenerator` 执行静态校验和可选 smoke 校验，并把结果写入 `generated_models/`

### 5.4 动态库构建链路
当用户明确要求 DLL / 动态库时：
- `CRMAgent` 在脚本生成成功后触发 `DynamicLibraryBuildTool`
- `DynamicLibraryBuildTool` 在进程内实例化 `LocalBuildMCPServer`
- 按顺序调用 MCP 工具：
  - `probe_toolchains`
  - `create_build_job`
  - `materialize_inputs`
  - `matlab_generate_cpp`
  - `cmake_configure`
  - `cmake_build_dynamic`
  - `inspect_artifacts`
- 作业结果统一落到 `generated_builds/<job_id>/`

## 6. 关键模块职责

### 6.1 入口层
- `main.py`：统一 CLI 入口。
- `api/server.py`：默认 FastAPI 实现，直接定义 `/api/chat`、`/api/query`、`/api/models`、`/api/health`。
- `api/query/routes.py`：可选拆分式 router，目前默认启动路径未显式挂载。

### 6.2 编排层
- `agents/crm_agent.py`：系统核心编排器，负责聊天、生成、澄清、续接、联网研究触发、历史联网证据合并、构建触发与返回结果整合。
- `agents/task_planner.py`：结合规则与 LLM 规划当前请求类型。
- `agents/session_store.py`：抽象出 memory / redis 两种会话后端。

### 6.3 结构化建模层
- `agents/structured_generation_ir.py`：组织 family schema、槽位抽取、澄清策略、IR 构造。
- `agents/structured_generation/schema_registry.py`：维护 family blueprint 与槽位元数据。
- `agents/structured_generation/slot_extractor.py`：抽取数值、单位、成对参数等槽位。
- `agents/structured_generation/clarify_policy.py`：决定是否继续澄清以及如何追问。

### 6.4 检索与知识层
- `knowledge_base/builder.py`：构建本地索引与可选 Qdrant 索引。
- `knowledge_base/document_loader.py`：按字符窗口切分 Markdown / Text 文档。
- `knowledge_base/rag_retriever.py`：混合检索、family / model 候选识别、生成门控、健康检查。
- `knowledge_base/web_evidence_retriever.py`：从独立联网证据 collection 中检索历史网页证据，并转换为 agent 兼容 `docs`。
- `tools/mcp_web_research/qdrant_indexer.py`：把成功抓取网页的正文分块增量写入独立 Qdrant collection。

### 6.5 规划与渲染层
- `agents/open_model_ir_*`：定义、构造、校验 `OpenModelIR`。
- `agents/model_spec_*`：保留 `ModelSpec` 兼容链和修复链。
- `knowledge_base/assembly_plan.py`：定义 `AssemblyPlan` typed schema。
- `knowledge_base/model_planner.py`：把 `OpenModelIR` 转成 `AssemblyPlan`。
- `knowledge_base/model_family_codegen.py`：family assembler，负责从 plan 生成具体脚本。
- `knowledge_base/blocks/`：block 粒度渲染规则与顺序。

### 6.6 校验与构建层
- `agents/matlab_codegen.py`：统一调度模板渲染、IR 渲染、静态校验、smoke 校验。
- `knowledge_base/matlab_static_validator.py`：静态语法与结构检查。
- `knowledge_base/matlab_smoke_tester.py`：MATLAB / Octave 运行级冒烟检查。
- `agents/tools.py`：把文件生成和动态库构建能力封装为工具对象。
- `tools/mcp_local_build/`：本地构建 MCP 服务与作业管理。

## 7. 部署架构

### 7.1 推荐模式：混合部署
- Docker 启动基础设施：`redis`、`qdrant`、`ollama`、`ollama-init`
- 本机 Conda 环境运行：
  - `python main.py build --with-qdrant`
  - `python main.py api`
- 该模式最贴合当前开发、调试和本地 MATLAB 工具链调用方式。

### 7.2 可选模式：全 Docker
`docker-compose.yml` 当前可拉起：
- `redis`
- `qdrant`
- `ollama`
- `ollama-init`
- `kb-build`
- `app1`
- `app2`

该模式适合演示和统一交付，但若涉及 MATLAB 本地编译链，仍通常依赖宿主机环境或额外集成。

## 8. 可观测性与质量保障
- `GET /api/health` 会返回 `retrieval.hybrid_effective`、当前会话后端等运行状态。
- `CRMAgent` 和 `MatlabRAGRetriever` 都会记录 `generation_trace`，便于回放“为什么生成 / 为什么澄清”。
- `generated_builds/<job_id>/manifest.json` 和 `result.json` 记录本地构建作业状态。
- `generated_research/` 结果包以及 `result.json` / `modeling_brief.json` 中的 `qdrant_index` 字段，可用于追踪联网抓取与证据入库状态。
- `tools/golden_match_regression.py` 提供匹配回归验证。
- `tests/` 目录中覆盖了 API、补槽续接、构建环境、动态库构建等核心场景。

## 9. 当前支持范围
- 航空航天：`launch_dynamics`、`trajectory_ode`、`powered_ascent`、`reentry_dynamics`、`aircraft_point_mass`、`interceptor_guidance`
- 水下：`underwater_launch`、`underwater_cruise`、`submarine_depth_control`
- 轨道：`orbital_dynamics`、`relative_orbit`、`orbit_transfer`
- 跟踪融合：`tracking_estimation`、`sensor_fusion_tracking`、`bearing_only_tracking`
- 战场态势：`combat_attrition`、`battlefield_awareness`、`threat_assessment`、`salvo_engagement`

## 10. 当前架构边界
- `OpenModelIR` 已进入主链路，但 IR 的真实表达能力仍受 family schema 和 fragment library 约束。
- `AssemblyPlan` 已落地，但最终渲染仍依赖 `FAMILY_RENDER_RULES` 与 `BLOCK_LIBRARY`。
- draft fragment 已可进入计划层，但当前主要以 `comment_only` 形式保留，不等于系统能自动把所有新物理片段转成可执行求解代码。
- `api/query/routes.py` 是可选 router，不是默认启动路径的一部分。
- `database/` 中的 SQLite CRM 模块当前未接入在线请求主路径。
- 动态库构建当前主要通过 `CRMAgent.build_dynamic_library()` 或 `DynamicLibraryBuildTool` 触发，没有独立 HTTP 构建端点。
- `materialize_inputs()` 对输入路径做了收敛控制，只允许仓库根目录及 `generated_models/` 下的 `.m` 文件进入构建作业。
- 历史联网证据当前没有并入 `MatlabRAGRetriever` 主检索器，而是在 `CRMAgent` 层仅对 `request_web_research=True` 的请求做增量合并。
- `python main.py build --with-qdrant` 不会重建独立联网证据 collection；联网证据的消失主要受单独清理策略与保留期配置影响。
