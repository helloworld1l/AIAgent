# Open Model IR 草案

本文档定义当前项目从“闭集模板建模”演进到“开放域建模生成”的中间表示（IR, Intermediate Representation）草案。

目标：

- 在自然语言需求与 MATLAB 代码之间增加一层结构化表示
- 降低当前系统对固定 `model_id` 模板库的强依赖
- 为未来的开放域建模、自动澄清、方程校验、代码组合生成打基础
- 保持与当前 `ModelSpec` 和现有代码生成链路兼容

适用项目目录：`D:\python\rag_crm_agent`

---

## 1. 为什么需要 Open Model IR

当前项目的建模主链路本质上是：

1. 检索知识库
2. 推断候选 `model_id`
3. 由 LLM 或启发式填充 `ModelSpec`
4. 根据 `model_id` 走固定 MATLAB 模板生成代码

这种方式适合“已知模板集合”下的快速生成，但存在明显限制：

- 当用户需求不在知识库模板内时，系统会倾向于匹配最相近的现有模板
- LLM 不是在自由构建物理模型，而是在候选模板中选择并补参数
- 校验层会拒绝未知 `model_id`
- 生成器主要是模板渲染器，而不是通用建模代码生成器

因此，需要一层比 `ModelSpec` 更强的中间表示，用来表达：

- 用户到底想建什么系统
- 系统有哪些对象、状态、输入、输出
- 用什么方程和物理机制描述它
- 哪些参数缺失、哪些假设被采用
- 最终该如何映射到 MATLAB 代码

---

## 2. 设计目标

Open Model IR 需要满足以下设计目标：

1. **开放域表达能力**：不强依赖预定义 `model_id`
2. **可组合**：支持对象、力项、事件、约束的组合表示
3. **可降级兼容**：能映射到现有 `ModelSpec`
4. **适合 MATLAB 生成**：便于渲染为脚本、函数或 Simulink 构造逻辑
5. **便于校验**：支持缺参识别、假设记录、仿真参数检查
6. **可扩展**：未来可支持更高保真度、更复杂多体系统和多阶段事件

---

## 3. 分层结构

建议把开放域 IR 分为三层：

### 3.1 `TaskIR`

描述用户意图和任务上下文：

- 用户目标是什么
- 任务类型是聊天、建模、澄清还是修复
- 使用什么语言
- 当前理解置信度是多少

### 3.2 `ModelIR`

描述模型本体：

- 领域和场景
- 对象、状态、输入、输出
- 方程、力项、事件、约束、参数、假设

这是开放域建模的核心层。

### 3.3 `CodegenIR`

描述代码生成偏好：

- MATLAB 脚本还是函数
- 使用 `ode45` 还是离散步进
- 是否生成图像、日志、注释

---

## 4. 顶层结构草案

建议的 Open Model IR 顶层结构如下：

```json
{
  "ir_version": "0.1",
  "task": {},
  "domain": {},
  "entities": [],
  "physics": {},
  "events": [],
  "constraints": [],
  "simulation": {},
  "outputs": {},
  "codegen": {},
  "trace": {}
}
```

其中各字段职责如下：

- `ir_version`：IR 版本号
- `task`：任务目标、类型、语言、置信度
- `domain`：领域、场景、保真度、坐标系、模型家族
- `entities`：建模对象列表
- `physics`：状态方程、力项、参数、初始条件、假设、缺失信息
- `events`：阶段切换、点火、分离、接触等事件
- `constraints`：边界、范围、守恒、逻辑约束
- `simulation`：求解器、时长、步长、采样信息
- `outputs`：输出信号、绘图、产物要求
- `codegen`：代码生成风格和目标
- `trace`：检索证据、推断来源、内部说明

## 5. 字段定义建议

### 5.1 `task`

建议结构：

```json
{
  "goal": "构建一个潜艇水下发射鱼雷模型，考虑水下阻力等因素",
  "request_type": "model_generation",
  "language": "zh-CN",
  "confidence": 0.86
}
```

字段说明：

- `goal`：用户原始目标
- `request_type`：`model_generation | chat | clarify | repair`
- `language`：输出语言
- `confidence`：当前理解置信度

### 5.2 `domain`

建议结构：

```json
{
  "primary": "marine_dynamics",
  "secondary": ["underwater_vehicle", "launch_dynamics"],
  "scene": "submarine_torpedo_underwater_launch",
  "model_family": "1d_ode_motion",
  "fidelity": "lumped_1d",
  "coordinate_system": "body_axis_1d"
}
```

字段说明：

- `primary`：一级领域
- `secondary`：二级领域标签
- `scene`：具体应用场景
- `model_family`：模型家族，用于替代或弱化固定 `model_id`
- `fidelity`：保真度级别
- `coordinate_system`：坐标系定义

### 5.3 `entities`

建议结构：

```json
[
  {
    "id": "torpedo",
    "type": "rigid_body",
    "role": "main_vehicle",
    "states": ["x", "v"],
    "inputs": ["thrust"],
    "outputs": ["x", "v", "a"]
  },
  {
    "id": "water",
    "type": "environment",
    "role": "fluid_medium"
  }
]
```

字段说明：

- `id`：对象唯一标识
- `type`：对象类型，如 `rigid_body`、`environment`、`sensor`
- `role`：对象在系统中的角色
- `states`：状态变量
- `inputs`：控制输入或外部驱动
- `outputs`：对外输出变量

### 5.4 `physics`

建议结构：

```json
{
  "governing_form": "ode",
  "state_equations": [
    "dx/dt = v",
    "m * dv/dt = thrust - drag - buoyancy_effect - gravity_effect"
  ],
  "forces": [
    {
      "name": "thrust",
      "type": "prescribed_or_parametric",
      "expression": "T(t)"
    },
    {
      "name": "drag",
      "type": "quadratic_drag",
      "expression": "0.5 * rho * Cd * A * v * abs(v)"
    }
  ],
  "parameters": [
    {"name": "m", "value": null, "unit": "kg", "required": true},
    {"name": "rho", "value": 1000, "unit": "kg/m^3", "required": false},
    {"name": "Cd", "value": null, "unit": "-", "required": true},
    {"name": "A", "value": null, "unit": "m^2", "required": true}
  ],
  "initial_conditions": {
    "x": 0.0,
    "v": 0.0
  },
  "assumptions": [
    "一维纵向运动",
    "忽略姿态变化",
    "阻力采用二次阻力模型"
  ],
  "missing_info": ["m", "Cd", "A"]
}
```

字段说明：

- `governing_form`：模型形式，如 `ode`、`state_space`、`difference_equation`
- `state_equations`：核心状态方程
- `forces`：力项/作用项/物理子模块
- `parameters`：参数列表，包含单位和缺省情况
- `initial_conditions`：初值条件
- `assumptions`：建模假设
- `missing_info`：当前仍缺失的关键信息

### 5.5 `events`

建议结构：

```json
[
  {
    "name": "launch_start",
    "trigger": "t == 0",
    "effect": "activate thrust"
  }
]
```

适用于：

- 点火
- 发射开始
- 分离
- 碰撞
- 模式切换

### 5.6 `constraints`

建议结构：

```json
[
  {
    "type": "range",
    "target": "Cd",
    "rule": "Cd > 0"
  }
]
```

用于表达：

- 参数范围
- 单位一致性约束
- 边界条件
- 逻辑约束

### 5.7 `simulation`

建议结构：

```json
{
  "solver": "ode45",
  "stop_time": 20.0,
  "time_step_hint": 0.01,
  "sample_count": 2000
}
```

字段说明：

- `solver`：求解器名称
- `stop_time`：仿真结束时间
- `time_step_hint`：建议步长
- `sample_count`：建议采样点数

### 5.8 `outputs`

建议结构：

```json
{
  "signals": ["x", "v", "a", "drag", "thrust"],
  "plots": [
    {"x": "t", "y": "x", "title": "Position vs Time"},
    {"x": "t", "y": "v", "title": "Velocity vs Time"}
  ],
  "artifacts": ["matlab_script"]
}
```

### 5.9 `codegen`

建议结构：

```json
{
  "target": "matlab_script",
  "style": "procedural",
  "include_comments": true,
  "function_mode": false
}
```

字段说明：

- `target`：目标产物类型
- `style`：代码风格
- `include_comments`：是否添加注释
- `function_mode`：是否生成函数式 MATLAB 文件

### 5.10 `trace`

建议结构：

```json
{
  "evidence_domains": ["underwater", "launch", "drag"],
  "selected_reason": "open_domain_physics_assembly"
}
```

用于记录：

- 检索证据摘要
- 路由原因
- 推断来源
- 内部调试线索

## 7. 最小可行版本（MVP）

为了降低改造成本，建议先落一个“工程可用”的精简版 IR：

```json
{
  "task_goal": "...",
  "model_family": "1d_ode_motion",
  "domain": "marine_dynamics",
  "scenario": "underwater_launch",
  "states": ["x", "v"],
  "parameters": {
    "m": null,
    "rho": 1000,
    "Cd": null,
    "A": null,
    "g": 9.81
  },
  "forces": [
    "thrust",
    "quadratic_drag",
    "buoyancy",
    "gravity"
  ],
  "equations": [
    "dx/dt = v",
    "m*dv/dt = thrust - 0.5*rho*Cd*A*v*abs(v) + buoyancy - m*g"
  ],
  "initial_conditions": {
    "x": 0,
    "v": 0
  },
  "simulation_plan": {
    "solver": "ode45",
    "stop_time": 20
  },
  "required_outputs": ["x", "v"],
  "missing_info": ["m", "Cd", "A"],
  "assumptions": [
    "1D motion",
    "constant water density"
  ]
}
```

这个版本的优势：

- 与现有 `ModelSpec` 字段风格接近
- 便于先实现 `IR -> MATLAB ODE 脚本` 生成器
- 不必一开始就重构整套系统

## 8. 与现有 `ModelSpec` 的字段映射表

下面给出 Open Model IR 与当前 `ModelSpec` 的建议映射关系。

### 8.1 现有 `ModelSpec` 主要字段

当前 `ModelSpec` 包含：

- `task_goal`
- `model_id`
- `assumptions`
- `parameters`
- `simulation_plan`
- `required_outputs`
- `missing_info`

### 8.2 映射表

| Open Model IR 字段 | 现有 `ModelSpec` 字段 | 映射方式 | 说明 |
|---|---|---|---|
| `task.goal` | `task_goal` | 直接映射 | 保留用户原始目标 |
| `domain.model_family` | 无直接字段 | 新增适配层 | 用于开放域生成，不建议直接塞进 `model_id` |
| `domain.scene` | 无直接字段 | 可写入内部元数据 | 建议放入 `_scene` |
| `domain.primary` | 无直接字段 | 可写入内部元数据 | 建议放入 `_domain` |
| `physics.assumptions` | `assumptions` | 直接映射 | 与当前字段天然兼容 |
| `physics.parameters` | `parameters` | 需扁平化 | 列表形式参数需转换为 `key -> value` |
| `simulation.stop_time` | `simulation_plan.stop_time` | 直接映射 | 当前系统最关键的仿真字段 |
| `simulation.solver` | `simulation_plan.solver` | 扩展映射 | 当前可先作为附加字段保留 |
| `outputs.signals` | `required_outputs` | 直接/裁剪映射 | 只保留主要输出变量名 |
| `physics.missing_info` | `missing_info` | 直接映射 | 当前字段天然兼容 |
| `trace.selected_reason` | `_build_source` 或内部元数据 | 部分映射 | 不建议覆盖现有内部字段语义 |
| `trace.evidence_domains` | 无直接字段 | 内部元数据 | 可写入 `_evidence_domains` |
| `entities` | 无直接字段 | 暂不直接映射 | 作为开放域生成器专用信息 |
| `physics.state_equations` | 无直接字段 | 暂不直接映射 | 由通用 ODE 生成器消费 |
| `physics.forces` | 无直接字段 | 暂不直接映射 | 由 IR 渲染器消费 |
| `events` | 无直接字段 | 暂不直接映射 | 由多阶段/事件驱动生成器消费 |
| `constraints` | 无直接字段 | 暂不直接映射 | 供校验器使用 |
| `codegen.*` | 无直接字段 | 暂不直接映射 | 供 MATLAB 生成器使用 |

## 9. `IR -> ModelSpec` 适配建议

当 Open Model IR 可以落回现有模板时，建议用一个适配器进行转换。

### 9.1 推荐适配流程

1. 从 Open Model IR 中识别是否存在可复用模板族
2. 如果存在，映射出 `model_id`
3. 把参数、假设、缺失信息、仿真计划映射到 `ModelSpec`
4. 继续走现有校验和代码生成链路

### 9.2 伪代码示例

```python
def adapt_ir_to_model_spec(ir: dict) -> dict:
    return {
        "task_goal": ir["task"]["goal"],
        "model_id": infer_legacy_model_id(ir),
        "assumptions": ir.get("physics", {}).get("assumptions", []),
        "parameters": flatten_parameter_list(ir.get("physics", {}).get("parameters", [])),
        "simulation_plan": {
            "stop_time": ir.get("simulation", {}).get("stop_time", 10)
        },
        "required_outputs": ir.get("outputs", {}).get("signals", ["plot"]),
        "missing_info": ir.get("physics", {}).get("missing_info", []),
    }
```

### 9.3 什么时候不该强行适配

以下情况不建议强行映射成当前 `ModelSpec`：

- 系统没有合适的现有 `model_id`
- 方程结构已经明显超出模板能力
- 需要事件驱动、多阶段状态切换
- 需要组合多个对象或多体系统

这时应直接走“开放域 IR 专用生成器”，而不是硬套现有模板。

## 10. 对当前项目的推荐落地步骤

### 10.1 第一阶段：半开放域

建议先做：

1. 新增 `OpenModelIR` Schema
2. 在现有 `ModelSpecBuilder` 前插入 `IRBuilder`
3. 增加 `IR -> ModelSpec` 适配器
4. 保留现有模板生成器不动

这样能快速获得：

- 更好的需求结构化
- 更好的澄清能力
- 更强的模板适配前置判断

### 10.2 第二阶段：开放域 ODE 生成

增加一个通用生成器：

- 输入：`OpenModelIR`
- 输出：基于 `ode45` 的 MATLAB 脚本

优先支持：

- `1d_ode_motion`
- `2nd_order_mechanical`
- `state_space_system`

### 10.3 第三阶段：校验与修复闭环

增加：

- 参数缺失追问
- 方程完整性检查
- 单位一致性检查
- MATLAB 语法与仿真可运行性检查

---

## 11. 一句话总结

当前项目的 `ModelSpec` 更像“模板参数单”，而 Open Model IR 应该成为“建模说明书”。

建议演进方向是：

- 从 `固定 model_id 选模板`
- 逐步升级为 `先生成结构化 Open Model IR，再决定走模板适配还是通用物理建模生成`

这样才能真正从闭集模板系统，逐步走向开放域建模生成器。
