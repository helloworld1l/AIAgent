# 开放域升级实施路线图

## 1. 当前定位
- 当前系统是“半开放域 MATLAB 建模生成系统”，还不是“真正可用的开放域系统”。
- 它已具备检索门控、结构化澄清、IR 驱动生成、block 组装这几个关键能力。
- 但当前能力仍然主要受“固定 slot schema + 固定模型族 + 预定义方程片段”限制。

## 2. 现状优势与核心短板
### 2.1 已有优势
- 能做“无匹配先澄清 / 拒绝误配”，避免直接套错模板。
- 能做结构化槽位收集，而不是只靠自然语言硬推参数。
- 已从 `model_id` 渲染升级到 `generation_ir` + `render_blocks` 组装。
- `knowledge_base/blocks/` 已具备按领域扩展的结构基础。

### 2.2 核心短板
- 入口仍然依赖候选模型命中，还不能对任意新问题直接构建新模型。
- IR 还不是真正意义上的 `OpenModelIR`，对实体、状态、约束、事件、观测的表达还不够完整。
- 方程片段仍以预定义库为主，缺少“新片段推导”能力。
- 缺少“生成 -> 验证 -> 回修 -> 重生成”的工程化闭环。

## 3. 总体策略
- 不建议直接追求“一步到位的开放域生成器”。
- 建议按“强半开放域 -> 开放域 IR + Planner -> 工程化闭环”三阶段推进。
- 这样做可以先把航空航天、军工装备、战场态势三大重点域做稳，再向开放域过渡。

## 4. 三阶段路线图

### 阶段一：强半开放域化
**目标**
- 把当前系统做成一个“稳定、可扩、可拒绝误配”的半开放域基座。

**主要任务**
- 扩展 `knowledge_base/docs/`，补齐航空航天、军工装备、战场态势三大域的文档、标签、术语、参数词典。
- 扩展 `knowledge_base/blocks/`，把模型族从 6 个主干族扩展到更完整的领域族谱。
- 把 `SLOT_SCHEMAS` 从“单模型配置”升级为“按 family 组织的 schema”。
- 强化 `assess_generation_match` 的域冲突、低置信度、候选模型模糊等拒绝策略。
- 增加最小静态校验：参数齐全性、变量定义、block 依赖闭合、基础 MATLAB 脚本健全性。

**交付物**
- 更强的知识库和 block 库
- 按 family 组织的槽位 schema
- 更稳的澄清 / 拒绝策略
- 基础校验脚本

**验收标准**
- 三大重点域的主干问题能稳定命中正确 family
- 明显超范围请求不会被错误生成
- 对关键参数缺失的问题会优先澄清

### 阶段二：开放域 IR + Planner
**目标**
- 让系统从“选模板”进化为“先构模，再生成”。

**主要任务**
- 把 `开放模型中间表示草案.md` 落成为真实 schema，定义 `OpenModelIR`。
- 新增 `OpenModelIRBuilder`、`OpenModelIRValidator`、`ModelPlanner`。
- 新增 `AssemblyPlan` 中间层，负责从 IR 生成 block 组装计划。
- 支持“优先复用既有 fragment，不足时受控生成新 fragment 草案”的机制。
- 打通 `OpenModelIR -> ModelSpec` 兼容链路，同时预留 `OpenModelIR -> AssemblyPlan -> MATLAB` 直通路径。

**交付物**
- `OpenModelIR` schema 与 builder
- 通用建模 Planner
- `AssemblyPlan` 中间层
- fragment 推理与受控生成机制

**验收标准**
- 对未命中既有 `model_id` 的问题，系统仍可生成结构化 IR
- 对中等复杂度问题，可输出合理的实体 / 状态 / 方程骨架
- 可复用 family 的问题继续走现有稳定链路

### 阶段三：工程化验证闭环
**目标**
- 把开放域能力从“能生成脚本”升级到“能验证、能执行、能回修、能评测”。

**主要任务**
- 新建 4 层 validator：IR 结构校验、物理一致性校验、MATLAB 代码校验、运行校验。
- 建立“错误归因 -> 回到 IR / AssemblyPlan 层修正 -> 重生成”的自动修复回路。
- 建设 benchmark，按半开放域、开放域迁移、工程可靠性三类组织。
- 记录 generation trace，支持追踪检索命中、澄清轮次、IR 完整度、代码首次通过率等指标。

**交付物**
- `validation/` 下的验证组件
- `evaluation/` 下的 benchmark 题集与评测脚本
- 自动修复回路
- generation trace 和能力边界说明

**验收标准**
- 新问题生成后能自动做结构、物理、代码、运行四层检查
- 主要 benchmark 上的 MATLAB / Octave 可运行率达到可接受水平
- 对不确定问题不再“看起来生成成功，实则逻辑错误”

## 5. 推荐执行顺序
1. 先完成阶段一，把三大重点域做深做稳。
2. 再落地 `OpenModelIR` schema、builder、planner。
3. 最后完成验证闭环、自动修复和 benchmark 体系。

## 6. 近期最值得立即开始的工作
- 把 `agents/structured_generation_ir.py` 重构成“family-level schema + slot extractor + clarify policy”三层结构。
- 继续扩展 `knowledge_base/blocks/`，优先补齐姿态、控制、观测、多目标跟踪、对抗交战等 block 族。
- 新建 `schemas/open_model_ir.schema.json` 和 `agents/open_model_ir_builder.py`，正式把开放域 IR 落为代码。
- 新建 `validation/` 目录，先做静态 validator，再接 runtime validator。

## 7. 一句话总结
**当前项目已经进入“可用的半开放域阶段”，但要成为“真正可用的开放域系统”，必须依次完成“强半开放域化 -> OpenModelIR + Planner -> 工程化验证闭环”这三步。**
