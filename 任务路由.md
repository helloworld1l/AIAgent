- 任务路由器在这个项目里就是内部的“任务判定器”，类名是 RAGTaskPlanner，不是 FastAPI 路由，而是智能体内部用来判断“这句
    话该走哪条处理链路”的模块，定义在 agents/task_planner.py:16。
  - 它在 CRMAgent 初始化时创建，见 agents/crm_agent.py:36。

  作用

  - 它的核心作用是把用户输入分成 3 类：matlab_generation、chat、clarify，见 agents/task_planner.py:30。
  - 这样项目就不会把所有问题都当成“普通聊天”，也不会把所有提到 MATLAB 的话都误判成“直接生成代码”。
  - 简单说，它负责“先判断意图，再决定走聊天链路还是建模生成链路”。

  在项目里怎么工作

  - 用户请求进入 CRMAgent.chat() 后，系统先做知识检索，拿到前 10 条相关证据，见 agents/crm_agent.py:89。
  - 然后把 query + retrieved_docs + 当前会话历史 交给任务路由器，见 agents/crm_agent.py:90。
  - 路由器输出一个结构化结果，典型字段包括：task_type、confidence、reason、source、llm_error，见 agents/
    task_planner.py:31 和 agents/task_planner.py:40。
  - CRMAgent 再根据这个结果决定是否进入生成链路：只有当 task_type == "matlab_generation"，并且置信度和动作词条件满足时，
    才真正调用建模生成，见 agents/crm_agent.py:99。
  - 如果不满足，就走普通聊天回复链路，见 agents/crm_agent.py:107。

  它是怎么判定的

  - 这个路由器是“规则 + LLM”双层判定。
  - 第一层是规则判定 _rule_based_plan()：看用户句子里有没有“生成/构建/build”等动作词、有没有“matlab/simulink/pid/卡尔曼/
    火箭”等模型标记，还会参考检索到的最高分证据，见 agents/task_planner.py:45。
  - 比如用户问“PID 是什么”，没有明显生成动作词，就更偏向 chat，见 agents/task_planner.py:67。
  - 如果用户说“生成一个 PID 闭环 Simulink 模型”，动作词和模型词都很强，就更偏向 matlab_generation，见 agents/
    task_planner.py:82。
  - 第二层是 LLM 判定 _llm_plan()：把用户输入、最近历史、前 5 条检索证据拼成提示词，让 Ollama 只输出 JSON 结果，见
    agents/task_planner.py:96。
  - LLM 路由时目前只看最近 4 条历史，见 agents/task_planner.py:109；这点和主聊天窗口最近 12 条不是一回事。
  - 如果 LLM 返回合法 JSON，就优先采用 LLM 的判断；如果失败或格式不对，就回退到规则判定，见 agents/task_planner.py:26 和
    agents/task_planner.py:40。

  它对后续还有什么影响

  - 路由结果不只是“选链路”，还会作为“路由建议”写进后续聊天的 RAG 上下文里，帮助 LLM 更稳地回答，见 agents/
    crm_agent.py:391 和 agents/crm_agent.py:400。
  - 同时，接口返回里也会把 planner 结果带回去，便于你调试和观察系统为什么这么判，见 agents/crm_agent.py:124 和 agents/
    crm_agent.py:261。

  当前特点和限制

  - clarify 这个类型已经在路由器里定义了，见 agents/task_planner.py:30；但目前 CRMAgent 没有给它单独分支，所以它实际上会
    落回普通聊天链路。
  - 也就是说，当前真正落地的核心分流还是两条：matlab_generation 和 chat。
  - 另外，虽然你前面已经把聊天记忆调到 50/12/6，但任务路由器自己仍只看最近 4 条历史，位置在 agents/task_planner.py:109。
