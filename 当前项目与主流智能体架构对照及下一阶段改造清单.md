# 当前项目与主流智能体架构对照及下一阶段改造清单

## 1. 文档目的

本文用于回答两个问题：

1. 当前项目的演进方向是否正确。
2. 如果要从当前“半开放域、可组合生成”的系统继续演进，下一阶段最应该优先做什么。

本文基于当前仓库实现现状整理，结合截至 **2026-03-13** 可公开查阅的主流智能体工程实践，包括：

- OpenAI 关于 agent 的工程化建议
- OpenAI Agents SDK 对 tracing / guardrails / workflow 的抽象
- Anthropic 关于 tool schema 与 eval-driven development 的建议
- MCP 关于 structured elicitation 的标准化方向
- LangGraph 关于 durable state / persistence / human-in-the-loop / graph execution 的工程方向

本文不是泛泛而谈的“趋势报告”，而是尽量映射到本仓库现有实现，形成可执行的下一阶段改造清单。

---

## 2. 当前项目应该被定义成什么

从当前代码和使用方式来看，本项目更准确的定位是：

**一个面向 MATLAB 建模生成的、垂直领域工作流型智能体系统。**

它不是单纯的聊天机器人，也不是完全开放域的自治型智能体平台，而是：

- 以用户自然语言需求为输入
- 通过 RAG、结构化澄清、IR、规则与 LLM 结合完成任务规划
- 最终输出 MATLAB `.m` 文件

从主流分类上看，它更接近：

- **单主代理 + 工具链 / 工作流编排**
- **强结构化输出驱动**
- **高约束、低幻觉容忍度的生产型 agent**

这个定位是合理的，而且对于当前项目目标而言，**比“做一个完全自治的大而全 agent”更正确**。

## 3. 当前项目的主链路现状

当前仓库已经具备较完整的主链路，核心模块大致如下：

- 会话与状态：`agents/session_store.py`、`agents/crm_agent.py`
- 结构化 IR 与槽位收集：`agents/structured_generation_ir.py`
- 检索与误配判断：`knowledge_base/rag_retriever.py`
- 知识库构建：`knowledge_base/builder.py`
- MATLAB 代码生成入口：`agents/matlab_codegen.py`
- IR 组合式渲染：`knowledge_base/model_family_codegen.py`
- Block 库：`knowledge_base/blocks/`
- API 暴露：`api/server.py`

当前实际运行链路，已经不是“用户输入 → 直接问大模型 → 输出代码”，而更接近：

```text
用户请求
  → 会话状态读取
  → 检索召回
  → 匹配评估
  → 无匹配先澄清 / 信息不足先补槽位
  → IR / ModelSpec 生成
  → 校验 / 修复
  → 按 IR / block 组合渲染 MATLAB
  → 输出 .m 文件
```

这条主链路本身，已经非常接近当前主流的“工作流型 agent”实现方式。

## 4. 当前项目与主流智能体架构的对照表

### 4.1 总体判断

结论先给出：

- **方向正确**
- **与当前主流生产实践总体一致**
- **已经超过很多只做“提示词 + 几个工具”的 demo 型智能体**
- 但距离“可长期演进、可稳定上线、可持续评估”的生产级平台，还缺少若干基础设施能力

### 4.2 对照表

| 架构维度 | 当前主流实践 | 当前项目现状 | 判断 | 下一步建议 |
| --- | --- | --- | --- | --- |
| Agent 基本形态 | 先从单智能体 / 工作流做起，必要时再多智能体 | 当前本质上是单主代理编排，不盲目拆多 agent | 正确 | 继续保持，暂不优先多智能体 |
| Model + Tools + Instructions | 用模型推理，用工具取数/执行，用指令与 guardrails 约束 | 当前已有 Ollama、RAG、生成器、澄清与规则判断 | 对齐 | 继续把内部能力抽象成正式工具层 |

| Tracing | 每次 run 可回放：检索、工具、修复、失败原因 | 当前缺统一 trace 体系 | 明显短板 | P0 优先补齐 |
| Evals | 有回归集、任务集、误配集、上线前对比 | 当前缺一等 eval 体系 | 明显短板 | P0 优先补齐 |
| Human-in-the-loop | 高风险步骤支持人工确认/审批/修改 | 当前有澄清，但还不是正式审批节点 | 半对齐 | 增加人工确认节点 |
| Tool schema 标准化 | 工具定义清晰、参数 schema 明确、便于替换 | 当前多为内部 Python 调用与隐式契约 | 可用但不标准 | 后续抽象统一 tool contract |
| 安全与 guardrails | 输入、检索、工具、输出多层保护 | 当前已有“无匹配先澄清/拒绝误配” | 正确但不够完整 | 增加输入风险、输出约束、执行前校验 |
| 多模型路由 | 小模型做轻任务，大模型做复杂规划/修复 | 当前主要依赖统一 LLM 后端 | 可接受 | 未来可做模型分层路由 |
| 多智能体 | 只有确有必要时才拆成多个 agent | 当前未复杂拆分 | 正确 | 暂不优先 |

## 5. 为什么说当前方向是正确的

### 5.1 你没有停留在“纯聊天套壳”阶段

很多所谓智能体系统，本质上仍然是：

```text
用户输入 → LLM 直接回答 → 偶尔调用工具
```

这种模式做演示很快，但在“代码生成、专业建模、低容错输出”任务上非常脆弱。

当前项目已经走到了更合理的阶段：

- 先检索知识
- 再评估是否匹配
- 无法稳定生成就先澄清
- 先形成 IR / 规格
- 再做组合式代码生成

这比“直接让模型写 MATLAB”更像生产系统。

### 5.2 你已经在向“结构化 agent”演进

主流生产 agent 越来越强调：

- Structured outputs
- Tool contracts
- Durable state
- Human-in-the-loop
- Guardrails
- Evals
- Tracing

当前项目已经明显覆盖了其中一部分：

- 有 durable session state
- 有 structured IR
- 有澄清机制
- 有误配保护
- 有组合式执行

这说明方向并没有跑偏，反而是在走一条更稳的路。

### 5.3 你优先做“半开放域”，而不是直接追求全开放域

这是非常正确的工程策略。

在专业建模场景中，真正困难的不是“让模型说点像样的话”，而是：

- 不误配
- 不胡编方程
- 在信息不足时知道停下来问
- 在可生成时稳定输出结构正确、可用的代码

所以从“受限模板系统”逐步演进到“半开放域、可组合生成”，比直接做“万能开放域建模智能体”更符合主流工程路径。

## 6. 当前系统距离主流生产级智能体，还缺什么

下面这些能力，是当前系统最值得补的短板。

### 6.1 缺少一等的 tracing 能力

当前系统已经具备复杂链路：

- 检索
- 误配判断
- 澄清
- IR 续接
- 规格修复
- 代码生成

但如果没有统一 trace，很难回答这些关键问题：

- 这次为什么误配？
- 是检索候选错了，还是规划错了？
- 是 IR 不完整，还是修复轮次不够？
- 哪一步耗时最多？
- 哪一类请求最容易失败？

如果不能系统回答这些问题，后续演进会越来越靠经验和人工猜测。

### 6.2 缺少一等的 eval 体系

当前最危险的状态不是“效果不够好”，而是：

- 改完一版 prompt，不知道是不是把别的场景搞坏了
- 扩一批知识库，不知道误配率是否上升
- 加一个 block，不知道组合生成是否更稳定
- 换一个模型，不知道收益还是退化

没有 eval，系统无法稳定迭代。

### 6.3 工作流还是隐式的，不够显式

现在主链路已经存在，但很多编排逻辑还埋在代码分支里。

这会带来几个问题：

- 不容易看清楚状态转移
- 不容易做暂停 / 恢复 / 审批
- 不容易做可视化 trace
- 不容易引入后台任务和重试

主流工程系统通常会把这类链路抽成明确状态机或 graph。

### 6.4 澄清已经有了，但还没“协议化”

当前澄清能力已经不错，但还可以进一步升级：

- 让 API 返回正式结构化澄清对象
- 明确每个槽位的 schema、默认值、是否必填、取值范围
- 让前端、日志、评测、回放都能理解这一轮到底在问什么

这一步做完后，系统会更接近 MCP 风格的 elicitation。

### 6.5 human-in-the-loop 还不是正式一等能力

你现在有“用户补参数”，但还没有“系统把高风险点显式交给人确认”的正式节点。

例如这些场景，未来都应该触发人工确认：

- 跨域候选分值接近
- 同时匹配多个模型族
- 修复轮次过多
- 需要覆盖关键物理假设
- 涉及高风险输出（例如军工装备敏感建模）

## 7. 下一阶段改造优先级清单

下面给出建议的优先级排序。

### 7.1 P0：必须先补的底座能力

这部分不一定最“炫”，但最关键。

#### P0-1 统一 trace / run 体系

目标：让每次请求都有完整可回放的执行记录。

建议落地点：

- 定义 `run_id` / `trace_id` / `session_id`
- 记录关键阶段事件：
  - 检索候选
  - 匹配评估
  - 澄清触发
  - IR 变化
  - 修复轮次
  - codegen 结果
- 记录关键指标：
  - 耗时
  - token 使用量
  - 候选分数
  - 是否进入降级路径
  - 最终成功/失败原因

目标不是先做漂亮面板，而是先能把每次运行完整记录下来。

#### P0-2 建立最小 eval 套件

目标：让系统具备“改完还能知道变好还是变坏”的能力。

第一版 eval 建议至少覆盖：

- 正常命中并生成成功
- 应当澄清而不是直接生成
- 应当拒绝误配
- 应当从 pending IR 正确续接
- 应当进入修复并成功
- 应当在未知域保持保守

建议把失败样本来源于：

- 你已经遇到的误配 case
- 典型跨域 case
- 多轮补参 case
- 长文本需求 case

#### P0-3 把主链路显式状态机化

建议把当前链路抽成类似下面的状态：

```text
RETRIEVE
→ ASSESS_MATCH
→ CLARIFY
→ BUILD_IR
→ BUILD_SPEC
→ VALIDATE
→ REPAIR
→ CODEGEN
→ FINALIZE
```

这样做的价值：

- 更容易 trace
- 更容易评测
- 更容易暂停 / 恢复
- 更容易插入人工审批
- 更容易做异步任务

#### P0-4 给 IR / 澄清对象做 schema 版本化

建议增加：

- `ir_version`
- `clarification_schema_version`
- `block_contract_version`

否则后续随着字段扩展，很容易出现兼容问题。

---

### 7.2 P1：从强工程原型升级到可靠 workflow agent

#### P1-1 让澄清返回正式结构化对象

不要只返回自然语言提示，而是返回类似：

- 缺失槽位列表
- 每个槽位的说明
- 类型 / 范围 / 默认值
- 当前已收集值
- 是否允许继续生成

这样前端、API、日志和评测都会更稳定。

#### P1-2 引入人工确认节点

建议在以下条件触发人工确认：

- 候选分数过于接近
- 跨域冲突明显
- 缺失关键物理参数
- 生成前假设较多
- 修复次数超过阈值

人工确认不是系统失败，而是高可靠系统的重要组成部分。

#### P1-3 支持异步任务和恢复

因为未来可能会出现：

- 长耗时建模
- 多轮补参中断
- 需要后台批量生成
- 生成后还要跑验证

建议提前把任务抽象为：

- `queued`
- `running`
- `waiting_user`
- `waiting_review`
- `succeeded`
- `failed`

#### P1-4 做模型路由

建议逐步尝试：

- 小模型负责意图判断、分类、轻量守门
- 大模型负责规格生成、修复、复杂解释

这样更贴近主流生产实践，也更利于控制成本和延迟。

#### P1-5 强化知识治理

重点包括：

- 文档来源可信度标签
- 域标签规范化
- 文档冲突检测
- 模型族覆盖率清单
- 失败样本回灌知识库

---

### 7.3 P2：从半开放域演进到平台化 agent 系统

#### P2-1 抽象统一工具层

建议把内部能力显式抽象为工具，例如：

- `retrieve_knowledge`
- `assess_match`
- `collect_slots`
- `build_ir`
- `validate_spec`
- `repair_spec`
- `render_matlab`
- `verify_generated_model`

这会让系统更容易替换实现，也更容易对接标准 agent framework。

#### P2-2 视需要兼容 MCP

如果未来希望系统更容易接入别的 agent host 或 IDE / 平台，建议考虑：

- 用 MCP 暴露工具
- 用 MCP 风格暴露 elicitation
- 用资源接口暴露知识或运行状态

这一步不是现在必须做，但很适合作为平台化方向的中期目标。

#### P2-3 加入生成后自动验证闭环

当前系统已经能生成代码，下一阶段可考虑加入：

- MATLAB / Octave 语法检查
- 简单数值稳定性检查
- 输出变量存在性检查
- 与 IR / Spec 的一致性检查

这样能进一步降低“生成了但不可用”的问题。

#### P2-4 再考虑多智能体

只有在以下情况明显成立时，才建议拆成多 agent：

- 单主代理 prompt / 工具上下文已经失控
- 检索、规划、验证、执行之间职责界限非常清晰
- 单 agent 的 eval 和 trace 已经足够成熟

也就是说，**多智能体是后期优化，不是当前第一优先级。**

## 8. 建议的实际落地顺序

如果按 3 个迭代阶段推进，我建议如下：

### 阶段一：先把系统“看清楚”

优先做：

1. trace
2. eval
3. 显式状态机
4. schema 版本化

目标：知道系统每次为什么成功、为什么失败。

### 阶段二：再把系统“管起来”

优先做：

1. 结构化澄清 API
2. 人工确认节点
3. 异步任务 / 可恢复运行
4. 模型路由

目标：让系统更稳定、更可控、更容易在线运行。

### 阶段三：最后把系统“平台化”

优先做：

1. 工具层抽象
2. MCP / 标准化接口
3. 自动验证闭环
4. 有必要时再拆多智能体

目标：从单系统演进为可复用平台。

---

## 9. 当前阶段不建议优先做的事情

为了避免投入分散，下面这些事情不建议现在抢先做：

- 不建议优先做“多智能体炫技式拆分”
- 不建议优先追求“完全开放域”
- 不建议优先接很多新模型、很多新工具
- 不建议只扩知识库而不做 eval / tracing
- 不建议把大量时间花在前端样式，而忽略后端链路可观测性

当前系统最关键的问题不是“功能还不够多”，而是：

**是否能稳定迭代、稳定评估、稳定解释失败原因。**

## 10. 一句话结论

如果只用一句话评价当前项目的演进方向：

**当前项目的技术方向是正确的，而且整体上符合当前主流的生产型智能体建设路线；下一阶段最重要的不是“更像一个花哨的 agent”，而是“更可观测、更可评测、更可恢复、更可控”。**

## 11. 对当前项目的建议性结论

给出一个简化判断：

- **战略方向：正确**
- **架构方向：正确**
- **工程形态：接近主流生产实践**
- **成熟度：还处于从强原型走向生产级平台的中间阶段**

如果继续沿着下面这条路线推进：

```text
半开放域知识增强
→ 结构化澄清
→ IR 与组合式生成
→ tracing / eval / state machine
→ human-in-the-loop
→ 标准化工具与平台化
```

那么这条演进路线是健康的，也比较符合主流智能体系统的长期建设方式。
