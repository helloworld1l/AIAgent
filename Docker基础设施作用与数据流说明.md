# Docker基础设施作用与数据流说明

## 1. 文档目的

本文说明当前项目里通过 Docker 启动的 `redis`、`qdrant`、`ollama` 分别承担什么职责，同时补充 `.env` / 运行环境中配置的嵌入模型 `BAAI/bge-large-zh-v1.5` 在系统中的作用、缓存位置和数据流向。

重点回答四个问题：

- `redis` 在项目里为什么存在。
- `qdrant` 在项目里为什么存在。
- `ollama` 在项目里为什么存在。
- `EMBEDDING_MODEL=BAAI/bge-large-zh-v1.5` 为什么虽然不是独立容器，却仍然是检索链路里的关键基础组件。

---

## 2. 当前部署方式里的定位

当前项目推荐的运行方式，不是把全部 Python 逻辑都塞进 Docker，而是采用“**Docker 跑基础设施，Conda 跑应用**”的混合模式：

```powershell
docker compose up -d redis qdrant ollama ollama-init
python main.py build --with-qdrant
python main.py api
```

这意味着：

- Docker 里的 `redis`、`qdrant`、`ollama` 是项目的基础设施层。
- 本地 `rag_crm` Conda 环境中的 Python 应用，是项目的业务编排层。
- 嵌入模型 `BAAI/bge-large-zh-v1.5` 不是独立容器，但会在 `kb-build` 容器、应用容器或本地 Python 进程中被加载，用于向量化文档和查询。
- 即使后续切到完整 Docker 模式（`app1`、`app2`、`kb-build` 一起启动），这些组件的职责也不变。

可以把它们理解为四个不同层次：

| 组件 | 所属层次 | 在项目中的核心作用 |
| --- | --- | --- |
| `redis` | 状态层 | 保存会话历史、澄清中断状态、跨实例共享上下文 |
| `BAAI/bge-large-zh-v1.5` | 向量化层 | 把文档和用户查询编码成向量，供 Qdrant 入库和语义检索使用 |
| `qdrant` | 检索层 | 保存知识库向量索引，支持运行期向量召回 |
| `ollama` | 推理层 | 承载本地 LLM，负责规划、澄清、规格推导、回复生成 |

其中需要特别区分：

- `BAAI/bge-large-zh-v1.5` 负责“**把文本变成向量**”。
- `qdrant` 负责“**把向量存起来并按相似度检索**”。

前者是向量化模型，后者是向量数据库，二者不是一回事。

---

## 3. 为什么这些基础设施和模型缓存要这样组织

把 `redis`、`qdrant`、`ollama` 放进 Docker，而把嵌入模型通过环境变量和缓存目录复用起来，主要有几个好处：

- 三个基础服务本质上都是长驻进程，适合容器化运行。
- 生命周期独立于 Python 代码，应用重启不会影响底层服务进程。
- 可以通过数据卷保留 Redis / Qdrant / Ollama 的数据和模型缓存。
- 嵌入模型可以通过宿主机缓存目录复用，避免每次构建和运行都重新下载。
- 本地 Python 应用和容器内应用都能通过统一环境变量连接同一套基础设施。
- 后续如果启用 `app1` / `app2` 双实例，也能天然共享同一套状态层、检索层和推理层能力。

与嵌入模型直接相关的环境项通常包括：

- `EMBEDDING_MODEL`：默认是 `BAAI/bge-large-zh-v1.5`
- `SENTENCE_TRANSFORMERS_HOME`：Sentence Transformers 模型缓存目录
- `DOCKER_ST_CACHE_DIR`：Docker 场景下将宿主机模型缓存挂载到容器内所使用的目录

在当前 `docker-compose.yml` 中，宿主机缓存会被挂载到容器内的：

- `/root/.cache/torch/sentence_transformers`

这意味着：

- 嵌入模型权重文件主要落在缓存目录里。
- 文档和查询经过模型编码后得到的向量，不会单独以一个业务文件长期保存。
- 构建阶段生成的向量主要写入 `Qdrant`。
- 运行时如果走本地向量降级链路，向量可能只存在于当前 Python 进程内存中。

其中还有一个辅助容器：

- `ollama-init`：不是业务服务，而是初始化助手，用来确保 `OLLAMA_MODEL` 指定的模型已经在 Ollama 中可用。

## 4. Redis 在项目中的作用

### 4.1 Redis 的角色

`redis` 在本项目中不是知识库，也不是模型服务，而是**会话状态中心**。

它主要保存两类数据：

1. **对话历史**
   - 用户消息
   - 助手消息
   - 保留最近若干轮上下文

2. **会话状态**
   - 当前是否处于“结构化澄清 / 槽位收集”中
   - 尚未补全的 IR / 规格信息
   - 典型状态键：`pending_generation_ir`

### 4.2 Redis 解决了什么问题

如果没有 Redis，系统只能把上下文放在当前 Python 进程内存里。这样会有几个问题：

- 应用一重启，会话历史就丢失。
- 如果将来启用 `app1`、`app2` 双实例，请求打到不同实例时，上下文无法连续。
- 结构化澄清进行到一半时，无法稳定续接上一轮收集到的槽位。

Redis 解决的是“**会话能否连续**”的问题，而不是“知识能否检索”或“模型能否推理”的问题。

### 4.3 Redis 在当前链路中的典型用途

在用户提出建模请求后，应用会：

- 按 `session_id` 从 Redis 读取历史对话。
- 判断之前是否已经存在待补全的 `pending_generation_ir`。
- 如果系统判断当前信息不足，需要先澄清参数，会把半成品 IR 再写回 Redis。
- 用户下一轮补充信息后，系统再从 Redis 取出这份 IR 继续收集，而不是从头开始。

### 4.4 Redis 异常时会怎样

当前代码里，如果 Redis 不可用，系统会降级到内存会话存储：

- 单进程下通常仍可临时运行。
- 但会丢失跨实例共享能力。
- 进程一重启，上下文就没了。
- “澄清到一半换实例 / 重启后续问”的稳定性会明显变差。

所以 Redis 的关键词是：**状态共享、会话连续、多轮澄清续接**。

## 5. 嵌入模型 `BAAI/bge-large-zh-v1.5` 在项目中的作用

### 5.1 它是什么

`BAAI/bge-large-zh-v1.5` 是当前项目默认使用的中文嵌入模型，对应配置项：

- `EMBEDDING_MODEL=BAAI/bge-large-zh-v1.5`

它通过 `sentence-transformers` 被加载，用来把自然语言文本编码成稠密向量。这个模型不是一个单独的 HTTP 服务，而是由 Python 进程在本地直接调用。

### 5.2 它负责什么

它的核心职责有三类：

1. **知识库构建时的文档向量化**
   - 把模型目录条目、示例语句、`knowledge_base/docs/` 里的文档分块编码成向量。
   - 然后把这些向量写入 Qdrant collection。

2. **运行时的查询向量化**
   - 把用户输入的问题编码成查询向量。
   - 用这个查询向量去 Qdrant 中做相似度搜索。

3. **Qdrant 不可用时的本地向量降级**
   - 如果 Qdrant 不可用，但嵌入模型还能加载，系统可以在本地准备文档向量并进行本地向量检索。
   - 也就是说，嵌入模型不只在“建库”时用到，在“运行期检索降级”时也会用到。

### 5.3 它不负责什么

嵌入模型虽然很重要，但它不负责：

- 保存向量索引
- 保存会话历史
- 进行 LLM 对话推理
- 直接生成最终 `.m` 文件

因此它的边界非常清晰：**只负责向量化，不负责存储、推理和状态管理**。

### 5.4 它的数据落点在哪里

需要把“模型文件”和“模型输出的向量”区分开看：

1. **模型文件 / 权重缓存落点**
   - 宿主机缓存目录，例如你当前复用的 Sentence Transformers 缓存路径
   - Docker 场景下通过 `DOCKER_ST_CACHE_DIR` 挂载到容器内
   - 容器内默认落点：`/root/.cache/torch/sentence_transformers`

2. **模型输出向量的落点**
   - 构建阶段：主要写入 `Qdrant`
   - 本地索引文件：`knowledge_base/matlab_knowledge_index.json` 主要保存文档文本和元数据，不是完整持久化向量库
   - 运行期本地降级检索：可能只保存在当前 Python 进程内存中

所以更准确地说：

- `BAAI/bge-large-zh-v1.5` 的**模型本体**落在缓存目录。
- 它生成的**业务向量结果**主要落在 `Qdrant` 或运行时内存里。

### 5.5 它在系统中的意义

嵌入模型的价值不在于“理解任务意图并做复杂推理”，而在于：

- 把文本映射到可比较的语义空间。
- 让“潜艇发射”“水下阻力”“鱼雷动力学”这类语义接近的描述，在检索层更容易彼此接近。
- 为 Qdrant 的语义召回提供输入基础。
- 为“无匹配先澄清 / 拒绝误配”提供更可靠的候选证据基础。

一句话说，它解决的是“**如何把文本变成可检索的向量证据**”。

### 5.6 嵌入模型异常时会怎样

如果 `BAAI/bge-large-zh-v1.5` 无法加载，通常会出现以下影响：

- 无法正常构建 Qdrant 向量索引。
- 运行时无法做高质量向量召回。
- Qdrant 即使在线，也缺少有效的查询向量输入。
- 系统可能退化到 BM25 词法检索或更弱的 fallback。

因此，嵌入模型异常通常不会像 Redis 故障那样影响会话连续性，也不会像 Ollama 故障那样直接让推理链路失效，但会明显削弱整个系统的检索质量。

它的关键词是：**文档编码、查询编码、语义向量化、检索前置条件**。

## 6. Qdrant 在项目中的作用

### 6.1 Qdrant 的角色

`qdrant` 是项目的**向量检索库**，负责保存知识库文档和模型候选的向量索引。

它保存的不是聊天记录，而是经过向量化后的知识条目，例如：

- 模型目录中的模型条目
- 模型示例语句
- `knowledge_base/docs/` 下文档切分后的 chunk

需要特别强调：

- 向量是由 `BAAI/bge-large-zh-v1.5` 生成的。
- Qdrant 负责保存这些向量并执行相似度检索。

### 6.2 Qdrant 参与的是哪一段流程

Qdrant 主要参与两件事：

1. **构建阶段**
   - 执行 `python main.py build --with-qdrant`
   - 将模型目录和知识文档切块
   - 通过嵌入模型向量化
   - 写入 Qdrant collection

2. **运行阶段**
   - 用户发起请求后
   - 检索器先准备查询向量
   - 从 Qdrant 找到语义上最接近的知识和候选模型
   - 再与 BM25 词法检索结果融合、重排

### 6.3 Qdrant 存的是什么，不存什么

Qdrant 中主要存的是“**检索用数据**”：

- 向量
- 文本
- 元数据 payload
  - `model_id`
  - `template_family`
  - `domain_tags`
  - `source_file`
  - `chunk_index`
  - 其他检索标签

Qdrant **不负责**：

- 生成向量
- 保存会话历史
- 保存用户澄清状态
- 执行 LLM 推理
- 生成最终 `.m` 文件

### 6.4 Qdrant 对生成质量的意义

Qdrant 的价值不在于“直接生成 MATLAB”，而在于：

- 让系统先找到更接近用户需求的知识证据。
- 降低错误模板被误选中的概率。
- 给后续的任务规划、IR 生成、规格修复提供更可靠的上下文。

换句话说，Qdrant 决定的是“**先找到什么向量近邻证据**”，而不是“**最后说什么**”。

### 6.5 Qdrant 异常时会怎样

当前系统有降级策略：

- 优先使用 Qdrant 做向量检索。
- 如果 Qdrant 不可用，可以降级到本地向量或 BM25。
- 如果向量链路整体不可用，还可能退到本地 fallback 文档。

因此 Qdrant 挂掉以后：

- 系统不一定立刻完全不可用。
- 但检索质量、候选稳定性、跨域误配控制能力会下降。
- “无匹配先澄清 / 拒绝误配”的判断也可能变得更保守或更不稳定。

所以 Qdrant 的关键词是：**向量存储、语义召回、候选收敛、误配抑制**。

## 7. Ollama 在项目中的作用

### 7.1 Ollama 的角色

`ollama` 是项目的**本地大模型推理服务**。

它通过 HTTP 接口对外提供模型能力，当前项目通过 `OLLAMA_BASE_URL` 和 `OLLAMA_MODEL` 连接它。默认场景下，承载的是本地 DeepSeek 模型（如 `deepseek-r1:7b`）。

### 7.2 Ollama 具体负责哪些能力

在本项目里，Ollama 不只是普通聊天接口，它被用于多个关键环节：

- 任务规划
  - 判断当前输入更像 `matlab_generation`、`chat` 还是 `clarify`
- 对话回复
  - 给出常规问答或解释性回复
- 规格推导
  - 根据检索结果和用户需求，生成 `ModelSpec` / IR
- 规格修复
  - 当结构不合法或语义校验失败时，尝试自动修复
- 生成阶段辅助推理
  - 在槽位不全时组织澄清问题
  - 在信息足够时推动进入代码生成链路

### 7.3 为什么 Ollama 不是知识库

Ollama 会“推理”，但它本身不等于项目知识库。

- 它不存储检索索引。
- 它不生成文档向量。
- 它不保证回答与本项目知识域一致。
- 它更像一个通用推理引擎。
- 真正给它“喂上下文”的，是检索层拿回来的候选知识和证据。

因此本项目不是“直接问大模型生成 MATLAB”，而是“**检索增强后再让大模型推理**”。

### 7.4 `ollama-init` 的作用

`ollama-init` 容器是 Ollama 的配套初始化组件：

- 先等待 `ollama` 服务起来
- 检查 `OLLAMA_MODEL` 是否已经存在
- 如果不存在，则执行 `ollama pull`

它的作用是让 Docker 启动后，模型处于可调用状态，避免第一次请求时才发现模型没准备好。

### 7.5 Ollama 异常时会怎样

如果 Ollama 不可用，项目影响最大，因为：

- 任务规划会受影响。
- 对话生成会受影响。
- `ModelSpec` / IR 推导与修复会受影响。
- 最终能否稳定输出 `.m` 文件会明显受影响。

当前代码里有一定的 fallback 提示机制，但总体上只能算“有限降级”，不能替代真正的 LLM 推理能力。

所以 Ollama 的关键词是：**任务理解、规格推导、修复归一、回复生成**。

## 8. 结合数据流看它们如何配合

## 8.1 构建阶段数据流：知识库入库

当执行下面命令时：

```powershell
python main.py build --with-qdrant
```

典型数据流如下：

```text
模型目录 + knowledge_base/docs 文档
        ↓
文档读取 / 分块 / 标题提取
        ↓
BAAI/bge-large-zh-v1.5 对文档块做向量化
        ↓
一条链路写入本地知识索引 matlab_knowledge_index.json（文本+元数据）
        ↓
另一条链路把向量写入 Qdrant collection
```

这个阶段里各组件的参与度分别是：

- `BAAI/bge-large-zh-v1.5`：负责把文档变成向量。
- `qdrant`：负责承接和保存向量索引。
- `redis`：通常不参与核心构建流程。
- `ollama`：通常不是构建索引的核心参与者。

也就是说，构建阶段的重点是“**把知识整理好、编码好，并放进检索层**”。

---

## 8.2 运行阶段数据流：对话与 MATLAB 生成

当执行下面命令并开始使用接口时：

```powershell
python main.py api
```

用户的一次请求，大致会经历下面的链路：

```text
用户请求
   ↓
API / Agent 接收输入
   ↓
Redis 读取 session 历史与 pending_generation_ir
   ↓
BM25 词法召回 + BAAI/bge-large-zh-v1.5 查询向量化
   ↓
Qdrant 语义检索（或本地向量降级检索）
   ↓
返回相近知识 / 候选模型
   ↓
系统评估：可直接生成？还是应先澄清 / 拒绝误配？
   ↓
Ollama 执行任务规划、IR/ModelSpec 推导、修复或对话生成
   ↓
代码生成器输出 MATLAB `.m` 文件
   ↓
结果写入 `generated_models/`
   ↓
响应返回给用户，并把新历史写回 Redis
```

### 8.2.1 如果信息不足

如果系统判断当前槽位不完整，流程会变成：

```text
用户请求
   ↓
检索 + 规划
   ↓
发现当前无法稳定生成
   ↓
系统构造结构化澄清问题
   ↓
pending_generation_ir 写入 Redis
   ↓
等待用户下一轮补充参数
   ↓
再次从 Redis 取出 IR 继续生成
```

这里最关键的是 Redis，因为它让“半成品规格”能够跨轮对话保存下来。

### 8.2.2 如果检索命中良好且信息充分

当知识命中较好、槽位也足够时，流程会更接近：

```text
用户需求
   ↓
BAAI/bge-large-zh-v1.5 把查询编码成向量
   ↓
Qdrant / BM25 找到最相关知识证据
   ↓
Ollama 基于证据生成较稳定的规格
   ↓
校验 / 修复
   ↓
按 IR 组合渲染 MATLAB 代码
   ↓
输出 `.m` 文件
```

这里：

- 嵌入模型决定“查询如何进入语义空间”。
- Qdrant 决定“能召回哪些相似证据”。
- Ollama 决定“如何基于证据完成推理和生成”。

三者共同影响最终结果。

## 8.3 多实例数据流：为什么双实例时更需要它们

如果采用完整 Docker 方案，项目中会出现 `app1` 和 `app2` 两个应用实例。此时这些基础设施的重要性会更明显：

```text
          ┌───────── app1 ─────────┐
用户请求 ─┤                        ├─→ 共享 Redis
          └───────── app2 ─────────┘
                     │
                     ├─→ 共享 BGE 文档/查询向量化能力
                     │
                     ├─→ 共享 Qdrant
                     │
                     └─→ 共享 Ollama
```

这带来的意义分别是：

- 共享 `redis`
  - 无论请求落到 `app1` 还是 `app2`，只要 `session_id` 一样，就能续上同一段会话与同一个待补全 IR。
- 共享嵌入模型缓存
  - 两个实例可复用同一份 Sentence Transformers 模型缓存，避免重复下载和不一致加载。
- 共享 `qdrant`
  - 两个实例看到的是同一份知识索引和同一套召回结果，不会因为实例不同而知识视图不同。
- 共享 `ollama`
  - 两个实例使用同一个模型服务，推理能力和模型版本保持一致。

所以“双实例”的真正前提不是多开两个 Python，而是：**它们必须共享同一套状态层、向量化基础、检索层、推理层基础设施**。

## 9. 这些组件在项目中的边界总结

为了避免概念混淆，可以这样记：

- `redis` 管“**这次对话进行到哪一步了**”。
- `BAAI/bge-large-zh-v1.5` 管“**如何把文档和查询变成可比较的语义向量**”。
- `qdrant` 管“**这些向量该如何存储与检索**”。
- `ollama` 管“**基于这些证据如何理解、规划、补全并生成结果**”。

它们缺一不可，但解决的是不同问题：

| 组件 | 主要回答的问题 | 如果缺失会怎样 |
| --- | --- | --- |
| `redis` | 当前会话状态是否可连续？ | 可临时降级到内存，但跨实例和续接能力变差 |
| `BAAI/bge-large-zh-v1.5` | 文本能否被稳定映射成语义向量？ | 向量检索和建库质量下降，系统更依赖 BM25 或 fallback |
| `qdrant` | 能否快速找到更相关的向量证据？ | 可退到本地检索，但召回质量和稳定性下降 |
| `ollama` | 拿到证据后，能否完成规划、澄清、推导和生成？ | 生成链路受明显影响，只能有限 fallback |

## 10. 一句话理解当前架构

如果只用一句话概括当前项目的运行数据流，可以写成：

```text
用户请求 → Redis 提供上下文状态 → BAAI/bge-large-zh-v1.5 把查询映射为向量 → Qdrant 提供知识召回 → Ollama 完成推理与规格构建 → 代码生成器输出 MATLAB 文件
```

如果概括构建数据流，则可以写成：

```text
知识文档 / 模型目录 → BAAI/bge-large-zh-v1.5 向量化 → Qdrant 建立语义索引 → 运行期供检索器调用
```

其中：

- `Redis` 是状态层。
- `BAAI/bge-large-zh-v1.5` 是向量化层。
- `Qdrant` 是检索层。
- `Ollama` 是推理层。

这也是为什么当前项目即使推荐“Docker 跑基础设施 + Conda 跑应用”，仍然强烈建议把这几类组件稳定地组织好。它们共同构成了整个 MATLAB 建模生成系统的底座。
