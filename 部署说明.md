# 部署说明

本文档说明当前仓库的最新部署方式、推荐运行模式、环境变量和健康检查方式。

---

## 1. 当前推荐模式

当前最推荐的部署方式是：

- Docker 负责基础设施：`redis`、`qdrant`、`ollama`、`ollama-init`
- Python 应用层在本机 `rag_crm` 环境运行

也就是“混合部署”。

这样做的原因是：

- 与当前仓库的开发调试方式最一致
- 能直接复用宿主机的 Python 依赖环境
- 能直接复用宿主机已缓存的 Ollama 和 Sentence Transformers 模型
- 遇到 Docker Python 基础镜像拉取受限时更稳妥

仓库中的 `docker-compose.yml` 仍然保留了完整的全 Docker 编排，因此全 Docker 模式仍可用，但不作为当前默认推荐。

---

## 2. 部署前准备

### 2.1 基础依赖

- 已安装 Docker Desktop / Docker Engine
- 本机可启动 Docker Compose
- 本机存在 `rag_crm` Python 环境，或已通过 `RAG_CRM_PYTHON` 指向对应 `python.exe`

如果你不想手动找 `rag_crm` 环境解释器，直接使用仓库自带包装脚本即可：

```powershell
powershell -ExecutionPolicy Bypass -File tools\run_in_rag_crm.ps1 main.py api
```

### 2.2 建议检查 `.env`

本机混合部署通常至少需要确认这些变量：

```powershell
QDRANT_HOST=localhost
QDRANT_PORT=6333
QDRANT_COLLECTION=crm_filters
OLLAMA_BASE_URL=http://localhost:11434
OLLAMA_MODEL=deepseek-r1:7b
SESSION_STORE_BACKEND=memory
REDIS_HOST=localhost
REDIS_PORT=6379
API_PORT=8000
```

说明：

- 本机 API 默认是 `SESSION_STORE_BACKEND=memory`
- 如果你希望本机 API 也共享 Redis 会话，把 `.env` 中 `SESSION_STORE_BACKEND` 改成 `redis`
- 当使用 Compose 内部应用容器时，`docker-compose.yml` 会自动把 `SESSION_STORE_BACKEND` 覆盖为 `redis`

### 2.3 可选缓存目录

如果你希望 Docker 复用宿主机已下载缓存，请确认：

```powershell
DOCKER_OLLAMA_HOME=C:/Users/Lenovo/.ollama
DOCKER_ST_CACHE_DIR=C:/Users/Lenovo/.cache/torch/sentence_transformers
```

注意：

- `DOCKER_OLLAMA_HOME` 必须指向 `.ollama` 根目录，而不是只挂某个 `manifests/...` 子目录
- `DOCKER_ST_CACHE_DIR` 建议指向 `sentence_transformers` 根目录

## 3. 混合部署步骤

### 3.1 启动基础设施

```powershell
docker compose up -d redis qdrant ollama ollama-init
```

这会启动：

- `redis`
- `qdrant`
- `ollama`
- `ollama-init`

其中 `ollama-init` 会在模型不存在时自动拉取 `OLLAMA_MODEL`，默认是 `deepseek-r1:7b`。

### 3.2 构建知识索引

推荐使用包装脚本，避免误跑到系统 Python：

```powershell
powershell -ExecutionPolicy Bypass -File tools\run_in_rag_crm.ps1 main.py build --with-qdrant
```

如果你已经激活 `rag_crm` 环境，也可以直接执行：

```powershell
python main.py build --with-qdrant
```

说明：

- `python main.py build`：只构建本地知识索引
- `python main.py build --with-qdrant`：同时构建本地索引并写入 Qdrant

### 3.3 启动本机 API

```powershell
powershell -ExecutionPolicy Bypass -File tools\run_in_rag_crm.ps1 main.py api
```

启动后访问：

- `http://127.0.0.1:8000/ui`
- `http://127.0.0.1:8000/docs`

### 3.4 健康检查

```powershell
curl http://127.0.0.1:8000/api/health
curl http://127.0.0.1:6333/collections
```

建议重点检查：

- `status`
- `session_store_backend`
- `retrieval.hybrid_effective`

对于本机 API：

- 如果 `.env` 还是默认值，`session_store_backend` 常见为 `memory`
- 如果你把 `SESSION_STORE_BACKEND=redis`，则应显示为 `redis`

如果 `retrieval.hybrid_effective=false`，说明当前并没有真正跑在完整混合检索状态，应优先检查 Qdrant、向量模型加载和索引构建是否成功。

### 3.5 Golden 回归验证

```powershell
powershell -ExecutionPolicy Bypass -File tools\verify_golden.ps1
```

该脚本会固定使用 `rag_crm` 环境运行 `tools/golden_match_regression.py`，适合在知识库或检索逻辑改动后做回归。

## 4. 全 Docker 模式

如果你希望把应用层也放入容器，可以使用完整编排：

```powershell
docker compose up -d --build
```

当前完整编排包含以下服务：

- `redis`：共享会话存储
- `qdrant`：向量检索服务
- `ollama`：模型推理服务
- `ollama-init`：模型初始化拉取
- `kb-build`：等待 Qdrant 就绪后按需构建 collection
- `app1`：对外暴露 `8000`
- `app2`：对外暴露 `8001`

访问地址：

- `http://127.0.0.1:8000/ui`
- `http://127.0.0.1:8001/ui`
- `http://127.0.0.1:11434`
- `http://127.0.0.1:6333/collections`

健康检查：

```powershell
curl http://127.0.0.1:8000/api/health
curl http://127.0.0.1:8001/api/health
```

在全 Docker 模式下，`app1` / `app2` 的 `session_store_backend` 应为 `redis`。

### 4.1 `kb-build` 的当前行为

`kb-build` 不会无条件重建 Qdrant。

它的当前行为是：

- 先等待 Qdrant 就绪
- 检查目标 collection 是否已经存在
- 如果 collection 已存在，则跳过重建
- 如果 collection 不存在，执行 `python main.py build --with-qdrant`

如果你希望强制重建，请在启动前设置：

```powershell
$env:FORCE_REBUILD_QDRANT = '1'
docker compose up -d --build
```

### 4.2 查看初始化日志

```powershell
docker compose logs -f ollama-init kb-build app1 app2
```

## 5. 环境变量说明

### 5.1 模型与检索相关

- `OLLAMA_BASE_URL`：本机 API 访问 Ollama 的地址，默认 `http://localhost:11434`
- `OLLAMA_MODEL`：默认 `deepseek-r1:7b`
- `OLLAMA_TIMEOUT_SEC`：Ollama 请求超时
- `OLLAMA_NUM_PREDICT`：生成长度上限控制
- `QDRANT_HOST` / `QDRANT_PORT` / `QDRANT_COLLECTION`：Qdrant 连接配置
- `EMBEDDING_MODEL`：默认 `BAAI/bge-large-zh-v1.5`
- `EMBEDDING_DEVICE`：默认 `cpu`
- `RETRIEVAL_VECTOR_BACKEND`：向量后端策略，默认 `auto`
- `RETRIEVAL_BM25_WEIGHT`：BM25 融合权重
- `RETRIEVAL_VECTOR_WEIGHT`：向量融合权重
- `RETRIEVAL_RERANK_BLEND`：重排融合权重
- `RETRIEVAL_CANDIDATE_MULTIPLIER`：候选扩展倍率
- `MODEL_SPEC_REPAIR_MAX_ROUNDS`：规格修复最大轮数

### 5.2 会话与状态相关

- `SESSION_STORE_BACKEND`：`memory` 或 `redis`
- `SESSION_HISTORY_SIZE`：会话缓存上限
- `CHAT_HISTORY_WINDOW`：聊天历史窗口
- `FALLBACK_HISTORY_WINDOW`：兜底历史窗口
- `PLANNER_HISTORY_WINDOW`：规划器使用的历史窗口
- `SESSION_TTL_SEC`：Redis 会话 TTL
- `REDIS_HOST` / `REDIS_PORT` / `REDIS_DB` / `REDIS_PASSWORD` / `REDIS_KEY_PREFIX`：Redis 配置

### 5.3 Docker 编排相关

- `PYTHON_BASE_IMAGE`：应用镜像基础镜像，默认 `python:3.10-slim`
- `APP_IMAGE_NAME`：应用镜像名称
- `DOCKER_OLLAMA_HOME`：Docker 绑定的宿主机 Ollama 缓存目录
- `DOCKER_ST_CACHE_DIR`：Docker 绑定的宿主机 Sentence Transformers 缓存目录
- `DOCKER_OLLAMA_BASE_URL`：容器内访问 Ollama 的地址，默认 `http://ollama:11434`
- `DOCKER_QDRANT_HOST`：容器内访问 Qdrant 的地址，默认 `qdrant`
- `DOCKER_REDIS_HOST`：容器内访问 Redis 的地址，默认 `redis`
- `FORCE_REBUILD_QDRANT`：设为 `1` 时强制重建 collection

如果当前网络无法访问 Docker Hub，可将 `PYTHON_BASE_IMAGE` 替换为公司私有仓库地址或可访问镜像代理地址。

## 6. 常用运维命令

### 6.1 查看基础设施状态

```powershell
docker compose ps
```

### 6.2 查看基础设施日志

```powershell
docker compose logs -f redis qdrant ollama ollama-init
```

### 6.3 停止基础设施

```powershell
docker compose down
```

### 6.4 停止并清理卷数据

```powershell
docker compose down -v
```

注意：`down -v` 会清理 Redis、Qdrant 和生成文件卷，适合彻底重置环境。

## 7. 常见问题

### 7.1 `ollama-init` 一直重复拉模型

优先检查：

- `DOCKER_OLLAMA_HOME` 是否指向 `.ollama` 根目录
- 是否只挂载了 `manifests` 而没有挂载 `blobs`

### 7.2 构建索引时报 Qdrant 连接失败

优先检查：

- `qdrant` 是否已经启动
- `QDRANT_HOST` / `QDRANT_PORT` 是否正确
- 是否执行过 `python main.py build --with-qdrant`

### 7.3 本机 API 能启动，但 `/api/health` 显示 `degraded`

优先检查：

- Qdrant 是否可访问
- 嵌入模型是否成功加载
- 当前 collection 是否存在
- `retrieval.hybrid_effective` 是否为 `true`

### 7.4 全 Docker 模式下应用容器没起来

优先检查：

- `PYTHON_BASE_IMAGE` 是否可拉取
- `kb-build` 是否执行失败
- `ollama-init` 是否还在初始化模型

---

## 8. 建议结论

对于当前仓库，建议优先使用“Docker 起基础设施 + 本机 `rag_crm` 环境运行应用”的混合部署方式；
当网络、镜像源和容器构建链路都稳定后，再切换到全 Docker 模式。
