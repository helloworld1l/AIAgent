# 提示词优化清单与多模型拆分改造方案

## 1. 目标与结论

本文面向当前项目的继续升级，回答两个问题：

1. 除了 `详细设计文档.md` 第 19 节给出的主线方向，是否还能通过提示词工程和更强模型获得增益。
2. 如果要引入 Qwen 系列模型，应该如何分阶段落地，并尽量减少对现有代码的扰动。

先给结论：

- 可以做，而且值得做。
- 但提示词优化和更换大模型，更多是“提升现有链路表现”的手段，不能替代开放域演进的主线建设。
- 当前系统的核心边界仍然在于：`OpenModelIR` 主要由 family schema 驱动，最终渲染仍依赖 `FAMILY_RENDER_RULES` / `BLOCK_LIBRARY`，draft fragment 还未稳定转化为可执行片段，运行级验证闭环也还不够强。
- 因此，建议采用“双轨制”：
  - 短期：做提示词工程 + 多模型拆分，提高当前系统的路由、澄清、槽位抽取、Spec 修复和普通对话质量。
  - 中期：仍按 `详细设计文档.md` 第 19 节推进 family schema、OpenModelIR、draft fragment 和运行级验证闭环。

## 2. 当前系统里 LLM 的真实作用点

从当前代码看，LLM 不是只用于普通聊天，而是贯穿了多条链路：

- `agents/crm_agent.py`
  - 普通对话与回退对话共用 `settings.OLLAMA_MODEL`
  - `/api/chat` 失败时还会回退到 `/api/generate`
- `agents/task_planner.py`
  - 负责 `matlab_generation / chat / clarify` 的任务路由判断
- `agents/model_spec_builder.py`
  - 负责基于检索证据构造 `ModelSpec`
  - 负责对不合法 `ModelSpec` 进行 LLM 修复
- `config/settings.py`
  - 当前只有一个统一的 `OLLAMA_MODEL`
- `.env` / `docker-compose.yml`
  - 当前也是围绕单一 `OLLAMA_MODEL` 在拉取和注入模型

这意味着：

- 提示词工程一定会见效，因为当前多个关键环节都在“吃”提示词质量。
- 简单把模型从 `deepseek-r1:7b` 换成更强模型，也会有直接收益。
- 但继续全链路共用一个模型，会把“路由任务的低成本诉求”和“Spec/Repair 的高质量诉求”绑定在一起，形成延迟与成本问题。

## 3. 提示词优化清单

下面的优化按“优先级从高到低”排列，建议先做前 5 项。

### 3.1 共用提示词骨架统一化

当前问题不是提示词完全没有约束，而是不同阶段的提示词各写一份，结构不完全一致，容易出现以下问题：

- 角色定义力度不一致
- 输出契约不一致
- 不确定信息的处理策略不一致
- 候选模型优先级不一致

建议统一成下面的骨架：

1. **角色定义**
   - 明确说明当前模型是“任务路由器 / ModelSpec 规划器 / ModelSpec 修复器 / 通用中文助手”。
2. **输入边界**
   - 告知模型只能使用“用户输入 + 最近对话 + 检索证据 + 候选模型”。
3. **决策规则**
   - 先判断是否证据充足；不足则澄清；充足再输出结果。
4. **不确定性策略**
   - 不能确定的参数一律进入 `missing_info`，禁止编造。
5. **输出契约**
   - 严格 JSON；不得输出 Markdown、解释、思维过程、前后缀客套语。
6. **失败降级**
   - 无法确定 family 或候选模型冲突时，优先输出 `clarify`，而不是硬猜。

### 3.2 任务路由提示词优化

当前 `agents/task_planner.py` 的提示词已经比较简洁，但还可以强化为“证据驱动”的路由器，而不是“泛化聊天模型顺手做分类”。

建议补充以下规则：

- 明确 `clarify` 触发条件：
  - 用户目标不明确
  - 候选 family 冲突明显
  - 关键参数缺失导致不能直接生成
- 明确 `matlab_generation` 触发条件：
  - 用户意图是“生成模型 / 写 MATLAB / 建模仿真 / 导出代码”
  - 检索证据里出现与某个 family 高一致的模型或术语
- 明确 `chat` 触发条件：
  - 用户是在问概念、方案、比较、说明，而不是直接要生成代码
- 明确“不要被关键词误导”：
  - 单独出现“模型”“轨迹”“发射”等词，不必然进入生成链

建议输出结构从当前的 3 个字段，扩为：

```json
{
  "task_type": "matlab_generation|chat|clarify",
  "confidence": 0.0,
  "reason": "...",
  "evidence_family": "...",
  "missing_slots": ["..."],
  "should_ask": true
}
```

这样后续链路可以更稳地决定是直接生成、先澄清还是回到普通聊天。

### 3.3 ModelSpec 构建提示词优化

这是最值得投入的一段，因为它直接决定生成链路的“可执行性”。

建议增加以下约束：

- 明确优先级：
  - 候选模型优先于自由发挥
  - 检索证据优先于常识补全
  - 缺失信息优先写入 `missing_info`，而不是猜测
- 明确参数归一化：
  - 参数名尽量映射到 family schema 中的标准字段
  - 单位不确定时，不要自造单位
  - `stop_time`、初始状态、边界条件等强约束字段必须显式处理
- 明确 family 冲突策略：
  - 如果“发射模型”同时可能落到火箭 / 水下 / 导弹，必须回退澄清
- 明确输出风格：
  - 只保留必要假设
  - 不输出与代码生成无关的科普内容

建议在提示词里显式加入“错误倾向约束”：

- 不要因为检索证据中出现高频词就忽略用户主语
- 不要把平台层动力学误判成战场态势模型
- 不要把轨道动力学误判成大气层内发射段

### 3.4 ModelSpec 修复提示词优化

`repair_spec_with_llm()` 当前已经有“错误信息 + 原始 Spec + Schema”的输入，这很好，但还可以再加强两个点：

1. **最小修复原则**
   - 明确要求“优先保留合法字段，只改出错字段”。
2. **错误对齐原则**
   - 把 Schema 错误与语义错误区分开，要求模型逐类修。

建议增加以下指令：

- 不得删除与错误无关的合法字段
- 不得把 `model_id` 改成候选集之外的值，除非当前值无效且证据强烈支持其他候选
- 如果某个值不确定，优先恢复为默认值或放入 `missing_info`

### 3.5 普通对话提示词优化

`agents/crm_agent.py` 里的 `self.system_prompt` 当前适合通用问答，但对“系统升级建议类问题”还可以再加强：

- 回答先给结论，再给理由，再给建议动作
- 若问题本质是架构评估，不要误导用户进入代码生成链
- 若问题涉及项目改造，优先引用当前系统已有能力与边界作答
- 保持简洁，但允许输出结构化要点

### 3.6 检索证据呈现格式优化

当前证据是纯文本拼接，建议改造成更有层级的“半结构化证据块”：

```text
[evidence_1]
family=launch_dynamics
model_id=rocket_launch_1d
score=0.83
source=knowledge_doc
text=...
```

这样做的价值是：

- 对路由模型更友好
- 对 Spec 构建更友好
- 更容易在后续 trace 中定位“是证据问题还是模型问题”

### 3.7 加入少量 Few-shot，但只加高价值样本

建议不要大规模堆示例，而是只放“最容易误配”的少量样本，例如：

- “发射模型”在火箭 / 水下 / 导弹间的歧义
- “轨迹模型”在轨道传播 / 弹道飞行间的歧义
- “态势模型”在战场感知 / 跟踪估计 / 兵力消耗间的歧义

Few-shot 只需 3~5 组，放在路由器和 Spec 规划器里即可。

### 3.8 输出稳健性增强

建议在所有 JSON 输出型调用里统一加入以下约束：

- 输出只能是一个 JSON 对象
- 不允许 ```json 代码块
- 不允许在 JSON 前后补解释
- JSON 中禁止出现注释

同时保留现有的 `_extract_json_obj()` 兜底逻辑，但把它视为兜底，而不是常态。

### 3.9 增加提示词版本化

建议为不同链路增加显式版本号，例如：

- `PROMPT_VERSION_CHAT=v1`
- `PROMPT_VERSION_ROUTER=v2`
- `PROMPT_VERSION_SPEC=v3`
- `PROMPT_VERSION_REPAIR=v2`

这样做便于：

- A/B 对比
- 回滚
- 观察线上退化点

## 4. 多模型拆分改造方案

## 4.1 目标

把“一个模型做所有事情”的模式，改为“按任务阶段选模型”的模式。

目标不是堆复杂度，而是让：

- 低成本模型处理高频、轻任务
- 强模型处理低频、重任务
- 提高 JSON 输出稳定性
- 降低整条链路的平均时延

## 4.2 建议的角色拆分

建议至少拆成 4 个角色：

1. `CHAT_MODEL`
   - 用于普通对话、方案说明、升级建议类问题
2. `ROUTER_MODEL`
   - 用于 `matlab_generation / chat / clarify` 判定
3. `SPEC_MODEL`
   - 用于 `ModelSpec` 构建
4. `REPAIR_MODEL`
   - 用于 `ModelSpec` 修复

如果想再细一点，可以把 `ROUTER_MODEL` 再拆成：

- `INTENT_MODEL`
- `CLARIFY_MODEL`

但第一阶段没有必要。

## 4.3 推荐的配置项

建议在 `config/settings.py` 中保留兼容默认值，同时新增以下环境变量：

```env
OLLAMA_MODEL=deepseek-r1:7b

OLLAMA_CHAT_MODEL=
OLLAMA_ROUTER_MODEL=
OLLAMA_SPEC_MODEL=
OLLAMA_REPAIR_MODEL=

OLLAMA_CHAT_NUM_PREDICT=
OLLAMA_ROUTER_NUM_PREDICT=
OLLAMA_SPEC_NUM_PREDICT=
OLLAMA_REPAIR_NUM_PREDICT=
```

解释：

- 若某个角色专用模型为空，则自动回退到 `OLLAMA_MODEL`
- 这样能保持向后兼容，避免一次性改太猛

推荐的解析策略：

- `OLLAMA_CHAT_MODEL or OLLAMA_MODEL`
- `OLLAMA_ROUTER_MODEL or OLLAMA_MODEL`
- `OLLAMA_SPEC_MODEL or OLLAMA_MODEL`
- `OLLAMA_REPAIR_MODEL or OLLAMA_MODEL`

## 4.4 代码改造点

### A. `config/settings.py`

新增：

- `OLLAMA_CHAT_MODEL`
- `OLLAMA_ROUTER_MODEL`
- `OLLAMA_SPEC_MODEL`
- `OLLAMA_REPAIR_MODEL`
- 对应的 `*_NUM_PREDICT`

同时建议补一个统一访问函数，例如：

```python
def llm_model_for(role: str) -> str:
    mapping = {
        "chat": settings.OLLAMA_CHAT_MODEL or settings.OLLAMA_MODEL,
        "router": settings.OLLAMA_ROUTER_MODEL or settings.OLLAMA_MODEL,
        "spec": settings.OLLAMA_SPEC_MODEL or settings.OLLAMA_MODEL,
        "repair": settings.OLLAMA_REPAIR_MODEL or settings.OLLAMA_MODEL,
    }
    return mapping[role]
```

### B. `agents/task_planner.py`

把当前：

- `payload["model"] = settings.OLLAMA_MODEL`

改为：

- `payload["model"] = llm_model_for("router")`

同时把 `num_predict` 从统一配置改成 `router` 专属配置。

### C. `agents/model_spec_builder.py`

拆成两类：

- `_build_with_llm()` 使用 `llm_model_for("spec")`
- `repair_spec_with_llm()` 使用 `llm_model_for("repair")`

这样 Spec 生成和 Spec 修复就可以独立评估。

### D. `agents/crm_agent.py`

普通聊天链路改用：

- `llm_model_for("chat")`

如果仍然保留 `/api/generate` 回退，建议也走同一个 `chat` 角色，避免回退时模型意外切换。

### E. 新增一个轻量的 LLM 调用封装

建议新增一个轻量模块，例如：

- `utils/ollama_client.py`

至少统一以下逻辑：

- 根据角色选择模型
- 根据角色选择 `num_predict`
- 构造 `/api/chat` payload
- 统一错误处理
- 统一 trace 记录

这样可以避免现在 `crm_agent.py`、`task_planner.py`、`model_spec_builder.py` 各自拼一套 payload。

## 4.5 Docker 改造点

当前 `docker-compose.yml` 的 `ollama-init` 只围绕单个 `OLLAMA_MODEL` 拉取模型。

建议改成两级策略：

### 方案 1：最小改动

仍保留 `OLLAMA_MODEL`，但允许额外传一个要预拉取的列表：

```env
OLLAMA_MODEL=qwen3.5:27b
OLLAMA_PULL_MODELS=qwen3:8b,qwen2.5:14b,qwen3.5:27b
```

然后在 `ollama-init` 中循环 `OLLAMA_PULL_MODELS`。

### 方案 2：角色感知拉取

直接读取：

- `OLLAMA_CHAT_MODEL`
- `OLLAMA_ROUTER_MODEL`
- `OLLAMA_SPEC_MODEL`
- `OLLAMA_REPAIR_MODEL`

把非空模型去重后统一拉取。

第一阶段建议先做方案 1，脚本简单，部署风险更小。

## 5. 我建议优先尝试的 Qwen 方案

## 5.1 先说标签名

如果要用你前面提到的那一路模型，按官方 Ollama 库的标签格式，建议使用：

- `qwen3.5:27b`

而不是：

- `qwen3.5-27b`

也就是说，Ollama 里通常是“模型名 + 冒号 + 规格”的写法。

## 5.2 我的首选试法

我最想试的组合不是“全链路都换成一个大模型”，而是下面这个分工：

```env
OLLAMA_CHAT_MODEL=qwen3:8b
OLLAMA_ROUTER_MODEL=qwen2.5:14b
OLLAMA_SPEC_MODEL=qwen3.5:27b
OLLAMA_REPAIR_MODEL=qwen3.5:27b
```

原因如下：

### `qwen3:8b` 用于普通对话

- 目标是低延迟、足够自然、中文表现稳定
- 适合承担解释、比较、方案类回答
- 不建议把最重的 Spec/Repair 压到它身上

### `qwen2.5:14b` 用于路由

- 路由任务本质上是“短输入 + 强约束分类 + 结构化 JSON 输出”
- 这一环更看重稳定服从规则，而不是长篇推理
- 用 14B 档位做 router，通常性价比更高

### `qwen3.5:27b` 用于 Spec 构建和修复

- 这是最接近“高价值、低频、强约束”的环节
- 它需要综合用户目标、检索证据、候选模型、Schema 约束和缺失槽位
- 用更强模型处理这一步，收益通常最大

## 5.3 为什么不是“一把梭全用 qwen3.5:27b”

因为这样会有三个问题：

1. 普通聊天也会被大模型拖慢
2. 路由这种轻任务没有必要占用最重模型
3. 整体资源成本和响应时延都会被最慢环节主导

因此，更推荐“强模型只放在 Spec 和 Repair 上”。

## 5.4 如果你想先做最小实验

先做单模型替换对比也完全可以：

```env
OLLAMA_MODEL=qwen3.5:27b
```

先观察这 4 项：

- 任务路由误判率是否下降
- `ModelSpec` 的 `missing_info` 是否更合理
- 修复回合数是否下降
- 平均响应时延是否明显上升

如果单模型替换已经显著改善，再进入多模型拆分；这样更容易看清收益来自“模型本身”还是“拆分架构”。

## 6. 落地改法

## 6.1 第一阶段：只改提示词，不改模型

目标：低风险验证提示词工程收益。

动作：

1. 重写 `task_planner` 的 system/user prompt
2. 重写 `ModelSpecBuilder` 的 build/repair prompt
3. 重写 `crm_agent` 的 chat prompt
4. 给每条链路增加 prompt version
5. 记录路由结果、Spec 合法率、repair 次数、用户追问率

验收指标：

- `clarify` 是否更早出现，而不是错误直生
- `llm_json_parse_failed` 是否下降
- 自动修复成功率是否上升

## 6.2 第二阶段：单模型替换实验

目标：验证更强模型是否值得引入。

建议实验组：

```env
OLLAMA_MODEL=qwen3.5:27b
```

与当前基线对比：

- 当前基线：`deepseek-r1:7b`
- 实验组：`qwen3.5:27b`

至少抽 30~50 条样本，覆盖：

- 普通对话
- 明确生成请求
- 需要澄清的高歧义请求
- 容易修复失败的 Spec 样本

## 6.3 第三阶段：多模型拆分

目标：在质量提升基础上，把平均时延和资源占用压下来。

推荐实施顺序：

1. 先拆 `router`
2. 再拆 `spec`
3. 最后拆 `repair`

原因：

- `router` 改动小、收益容易观察
- `spec` 是质量收益最大的环节
- `repair` 在前两者稳定后再拆，更容易做归因

## 6.4 第四阶段：把观测补齐

建议把以下字段补进 trace：

- `llm_role`
- `llm_model`
- `prompt_version`
- `num_predict`
- `temperature`
- `json_parse_ok`
- `repair_round`

这样一旦效果退化，可以快速定位：

- 是模型换了导致退化
- 还是 prompt 改了导致退化
- 还是检索证据质量本身有问题

## 7. 建议的提示词样式

以下不是最终定稿，而是建议的收敛方向。

### 7.1 Router System Prompt 建议

```text
你是任务路由器。
你只能基于用户输入、最近对话和检索证据做判断。
你的任务是输出一个 JSON 对象，判断当前请求属于 matlab_generation、chat 或 clarify。
若 family 存在冲突、关键槽位缺失或目标不明确，优先输出 clarify。
不要输出 markdown，不要输出解释，不要输出思维过程。
```

### 7.2 Spec Builder System Prompt 建议

```text
你是 MATLAB 建模规格规划器。
你必须根据用户需求、候选模型和检索证据输出合法的 ModelSpec JSON。
候选模型优先于自由发挥；证据优先于常识补全。
不能确定的内容写入 missing_info，禁止编造。
如果 family 冲突明显，应输出更保守的参数，并保留缺失项。
只输出 JSON 对象。
```

### 7.3 Spec Repair System Prompt 建议

```text
你是 MATLAB 建模规格修复器。
你的任务是对一个不合法的 ModelSpec 做最小修复。
优先保留合法字段，只修改错误字段。
不要删除与错误无关的内容。
若某字段无法确定，恢复默认值或写入 missing_info。
只输出修复后的完整 JSON 对象。
```

### 7.4 Chat System Prompt 建议

## 8. 风险与回滚

### 8.1 风险

- 多模型后，部署和初始化会比现在更复杂
- trace 不补齐时，容易看不清是哪个角色模型出了问题
- 如果所有角色都上大模型，平均响应时延可能明显上升

### 8.2 回滚策略

保留统一总开关：

```env
OLLAMA_MODEL=deepseek-r1:7b
```

并允许所有角色专用模型为空。

这样一旦效果不好，只需清空：

- `OLLAMA_CHAT_MODEL`
- `OLLAMA_ROUTER_MODEL`
- `OLLAMA_SPEC_MODEL`
- `OLLAMA_REPAIR_MODEL`

即可回退到单模型模式。

## 9. 最终建议

如果只给一条最实际的路线，我建议按下面顺序推进：

1. 先做提示词优化
2. 再做 `qwen3.5:27b` 的单模型 A/B 实验
3. 确认确有收益后，再拆成：
   - `chat = qwen3:8b`
   - `router = qwen2.5:14b`
   - `spec = qwen3.5:27b`
   - `repair = qwen3.5:27b`
4. 最后补 trace 和回归样本集

这样做的好处是：

- 每一步都能单独验证收益
- 不会把“提示词收益”“模型收益”“架构收益”混在一起
- 对当前代码侵入相对较小，风险可控
