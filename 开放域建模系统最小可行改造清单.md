# 开放域建模系统最小可行改造清单

## 1. 文档目的

本文把 `开放域建模系统路线图.md` 进一步收敛成“最小可行改造清单”，目标不是一步到位做成全开放域万能生成器，而是在**尽量少改当前主干架构**的前提下，让系统具备下面 4 个最小能力：

1. 能识别开放域建模请求，而不是只能在固定模板里硬套。
2. 能先产出 `OpenModelIR` 草案，而不是遇到未知 family 就直接失败。
3. 能把“可落现有 family 的请求”和“暂时只能半结构化表达的请求”区分开。
4. 能保留现有代码组装和校验链路，避免因为追求开放域而失去可执行性。

一句话定义本次 MVP（最小可行产品）：

> 把当前系统从“半开放域 MATLAB 建模生成系统”，升级成“具备开放域理解、IR 草拟、有限执行落地和验证回修能力的最小可行开放域建模系统”。

## 3. 本次 MVP 的设计原则

### 3.1 保留现有主链路

以下主链路不推翻：

- 检索
- 路由
- 澄清
- `ModelSpec` / `OpenModelIR`
- `AssemblyPlan`
- renderer
- 静态校验 / smoke 校验

### 3.2 开放域只先加在“前端决策层”

开放域先体现在：

- 更好的开放域理解
- 更强的 `OpenModelIR` 草拟
- 更稳的结构化澄清
- 更清晰的“可执行 / 不可执行”分流

不在 MVP 阶段做：

- 任意物理系统从零自动求解器生成
- 彻底去掉 family schema
- 彻底去掉 block renderer
- 彻底去掉 `ModelSpec`

### 3.3 把 Qwen3.5 放在最值钱的位置

MVP 中建议：

- `Qwen3.5` 负责开放域理解、`OpenModelIR` 草拟、Spec/IR 修复
- 现有 renderer 和 validator 继续承担可执行性与稳定性职责

---

## 4. MVP 完成后的目标链路

目标链路如下：

1. 用户输入进入 `CRMAgent`
2. 路由器判断：聊天 / 澄清 / 开放域建模 / 已知 family 建模
3. 若是开放域建模：
   - 先生成 `OpenModelIR` 草案
   - 再判断 renderability（是否可落现有 family / block）
4. 若可落地：
   - 映射到 `ModelSpec` / `AssemblyPlan`
   - 进入现有代码生成与校验链
5. 若暂时不可落地：
   - 返回结构化澄清
   - 或返回“已理解但当前只能生成 IR 草案”的结果
6. 失败时：
   - 进入 IR / Spec 修复环

MVP 的关键不是“所有请求都能生成代码”，而是：

- **所有请求都能被正确分流**
- **更多请求能被结构化理解**
- **能生成代码的请求仍然稳定生成**

## 5. 最小可行改造清单

下面按优先级列出建议动作，优先级分为：

- `P0`：必须做，否则不构成 MVP
- `P1`：强烈建议做，否则效果不稳
- `P2`：可延后

## 6. P0 清单：必须完成的改造

### P0-1 统一 LLM 调用与多角色模型路由

**目标**

把当前“单一 `OLLAMA_MODEL` 贯穿所有链路”的模式，改成按任务角色选模型，为开放域前端升级做准备。

**涉及模块**

- `config/settings.py`
- `.env`
- `docker-compose.yml`
- `agents/crm_agent.py`
- `agents/task_planner.py`
- `agents/model_spec_builder.py`
- 建议新增：`utils/ollama_client.py`

**最小改法**

1. 在 `config/settings.py` 增加：
   - `OLLAMA_CHAT_MODEL`
   - `OLLAMA_ROUTER_MODEL`
   - `OLLAMA_IR_MODEL`
   - `OLLAMA_SPEC_MODEL`
   - `OLLAMA_REPAIR_MODEL`
2. 增加统一的 LLM 调用封装：
   - `chat(role="chat")`
   - `chat(role="router")`
   - `chat(role="ir")`
   - `chat(role="spec")`
   - `chat(role="repair")`
3. 现有模块不再直接拼接模型名，而是调用统一封装。

**推荐模型分工**

- `chat`：轻模型
- `router`：中等模型
- `ir/spec/repair`：`qwen3.5:27b`

**验收标准**

- `crm_agent`、`task_planner`、`model_spec_builder` 不再各自维护一套 Ollama payload。
- trace 中可看到 `llm_role` 和 `llm_model`。

---

### P0-2 把开放域建模从“隐含状态”升级为显式任务类型

**目标**

当前路由器主要区分 `matlab_generation / chat / clarify`，MVP 需要把“开放域建模”显式表达出来。

**涉及模块**

- `agents/task_planner.py`
- `agents/crm_agent.py`

**最小改法**

1. 将 `task_planner` 输出扩展为：
   - `task_type`
   - `confidence`
   - `reason`
   - `domain_candidates`
   - `family_candidates`
   - `needs_open_model_ir`
   - `renderability_hint`
2. 新增 `task_type` 值：
   - `open_model_generation`
3. 在 `crm_agent` 中增加显式分支：
   - `chat`
   - `clarify`
   - `matlab_generation`
   - `open_model_generation`

**验收标准**

- 高歧义开放域请求不会直接硬落某个 `model_id`。
- 对未知 family 请求，系统能走 `open_model_generation` 分支。

---

### P0-3 让 `OpenModelIR` 真正进入主链路

**目标**

当前 `OpenModelIR` 已存在，但还不是所有开放域请求的第一落点。MVP 需要把它变成开放域链路的主中间表示。

**涉及模块**

- `agents/crm_agent.py`
- `agents/open_model_ir_schema.py`
- `agents/open_model_ir_builder.py`
- `agents/open_model_ir_validator.py`
- `agents/open_model_ir_compat.py`
- 建议新增：`agents/open_domain_planner.py`

**最小改法**

1. 新增一个轻量 planner：`agents/open_domain_planner.py`
   - 输入：用户请求、最近对话、检索证据
   - 输出：`OpenModelIR` 草案
2. `crm_agent` 在 `open_model_generation` 分支里：
   - 调用 `open_domain_planner`
   - 存储 `pending_generation_ir`
   - 调用 `OpenModelIRValidator`
3. 若 IR 合法：
   - 进入 renderability 判断
4. 若 IR 不合法：
   - 进入 IR 修复或澄清

**最小 schema 增补建议**

建议在 `open_model_ir_schema.py` 里补 3 类字段：

- `evidence_refs`
- `renderability`
- `draft_fragments`

这里不要求 schema 大改，只要能支持“开放域草拟 -> renderability 分流”即可。

**验收标准**

- 开放域请求先产出 `OpenModelIR`，而不是先猜 `model_id`。
- session 中能保留 `pending_generation_ir` 和状态续接。

---

### P0-4 增加 renderability 判断层

**目标**

开放域系统的关键不是“懂不懂”，而是“当前能不能落到现有执行栈”。因此必须新增一个 renderability 分类节点。

**涉及模块**

- `agents/crm_agent.py`
- `agents/open_model_ir_compat.py`
- `agents/model_planner.py`
- `knowledge_base/assembly_plan.py`
- `knowledge_base/model_family_codegen.py`
- 建议新增：`agents/renderability_router.py`

**最小改法**

新增 3 类 renderability：

1. `family_renderable`
   - 可直接映射到已有 family / `ModelSpec`
2. `block_renderable`
   - 不能完全映射 family，但可落到有限 block 组合
3. `ir_only`
   - 只能保留为结构化草案，暂不生成代码

**推荐流程**

- `OpenModelIR` -> `renderability_router`
- 若 `family_renderable`：走现有 `ModelSpec` / `AssemblyPlan`
- 若 `block_renderable`：走有限片段组装
- 若 `ir_only`：返回澄清 / 结构化结果

**验收标准**

- 系统不再把“暂时无法执行”误判为“完全无法理解”。
- 对无法渲染的请求，能返回 `ir_only` 而不是乱套模板。

---

### P0-5 保留并强化结构化澄清

**目标**

开放域系统不是让模型盲猜，而是让系统知道什么时候该追问。

**涉及模块**

- `agents/structured_generation/schema_registry.py`
- `agents/structured_generation/slot_extractor.py`
- `agents/structured_generation/clarify_policy.py`
- `agents/crm_agent.py`
- `agents/open_model_ir_validator.py`

**最小改法**

1. 让 `clarify_policy` 同时支持：
   - family 冲突澄清
   - 对象层澄清
   - 关键参数澄清
   - 机制缺失澄清
2. 让 `slot_extractor` 既能抽 `ModelSpec` 参数，也能抽 `OpenModelIR.ParameterIR`
3. 把澄清结果回填到：
   - `pending_generation_ir`
   - `pending_generation_match`

**验收标准**

- 歧义请求优先进入澄清，而不是高置信误配。
- 澄清后可以续接 IR，而不是重新开始。

---

### P0-6 为开放域返回增加“理解成功但暂不可执行”的结果类型

**目标**

开放域 MVP 必须允许这样的结果：系统理解了问题，但当前执行栈还不能自动落代码。

**涉及模块**

- `agents/crm_agent.py`
- `api/server.py`
- `api/query/routes.py`

**最小改法**

返回结构里补充：

- `open_model_ir`
- `renderability`
- `next_action`
- `missing_info`
- `can_generate_code`

**验收标准**

- API 能清晰区分：
  - 已生成代码
  - 需要澄清
  - 已理解但暂不可执行

## 7. P1 清单：强烈建议同步完成

### P1-1 重构知识层：从“大而全知识簇”改成“硬约束 + 软背景”

**目标**

把知识库从“无边界累加文档”转成分层结构。

**涉及模块**

- `knowledge_base/docs/*`
- `knowledge_base/rag_retriever.py`
- 建议新增：`knowledge_base/open_domain_constraints.py`

**最小改法**

把知识分成两层：

1. **硬约束层**
   - family 边界
   - 反误配规则
   - 关键参数定义
   - 单位/范围规则
2. **软背景层**
   - 背景知识
   - 机制候选
   - 长尾术语
   - 示例表达

在 `rag_retriever.py` 中给检索结果打标签，例如：

- `doc_type=constraint`
- `doc_type=prototype`
- `doc_type=background`

**验收标准**

- 检索结果可区分“硬规则证据”和“背景证据”。
- 不再只靠追加文档去兜所有问题。

---

### P1-2 让 draft fragment 从注释升级为有限可执行片段

**目标**

当前最缺的是：系统知道一些机制，但还不能把它们变成有限可执行片段。MVP 不求全，只先做少量高价值 fragment。

**涉及模块**

- `agents/open_model_ir_schema.py`
- `agents/model_planner.py`
- `knowledge_base/blocks/*.py`
- `knowledge_base/model_family_codegen.py`

**最小改法**

先只支持 3~5 个 fragment：

- 阻力项
- 推力项
- 重力项
- 一阶损耗项
- 简单双边消耗项 / 一阶跟踪更新项

新增一个 fragment 到 block 的有限映射表。

**验收标准**

- 至少一小部分“非标准 family 请求”能通过 fragment 组合产出可执行代码。

---

### P1-3 打通 IR -> ModelSpec 的兼容降级

**目标**

**涉及模块**

- `agents/open_model_ir_compat.py`

- `agents/model_spec_validator.py`

**最小改法**

- 明确 `OpenModelIR -> ModelSpec` 的降级规则

**验收标准**

- 已知 family 请求不会因为引入开放域而变差。

### P1-4 补 trace 和最小评测集

**目标**

## 11. 不建议在 MVP 阶段做的事

以下动作很诱人，但不适合当前最小可行改造阶段：

1. 把现有 `ModelSpec` 全部删除
2. 一次性把所有 family 都改造成纯开放域 IR 驱动
3. 让大模型直接替代 renderer 和 validator
4. 一次性把所有知识库重写
5. 引入过重的多 agent 自治架构

原因很简单：

- 当前系统已经有可执行主链路
- MVP 的目标是“扩边界而不丢稳定性”
- 不是“为了开放域而拆毁现有可用能力”

---

## 12. 建议的最小完成定义

如果要判断这轮改造是否完成，我建议采用下面 6 条作为最小完成标准：

1. 未知 family 请求能被识别为 `open_model_generation`
2. 系统能产出合法的 `OpenModelIR` 草案
3. 系统能区分 `family_renderable / block_renderable / ir_only`
4. 澄清后能续接到 `pending_generation_ir`
5. 已知 family 生成质量不下降
6. 至少有一组开放域样本可回归评测

只要这 6 条达成，就说明系统已经从“半开放域模板生成器”迈入了“最小可行开放域建模系统”的阶段。
