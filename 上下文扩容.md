  - 当前“上下文缓存”是应用层会话缓存，不是独立的缓存服务；API 启动后只创建一个进程内 CRMAgent 单例，见 api/server.py:53
    和 api/server.py:65。
  - 会话缓存按 session_id 分桶，数据结构是 defaultdict(deque)；每个会话一个 deque(maxlen=history_size)，见 agents/
    crm_agent.py:34 和 agents/crm_agent.py:42。
  - 每次请求先做检索，再把当前会话历史交给任务路由器；这里传入的是该会话当前全部已保留历史，见 agents/crm_agent.py:89 和
    agents/crm_agent.py:93。
  - 真正发给聊天 LLM 的上下文是：系统提示词 + RAG证据摘要 + 最近若干条历史 + 当前用户消息，见 agents/crm_agent.py:272、
    agents/crm_agent.py:273、agents/crm_agent.py:277、agents/crm_agent.py:278。
  - 对话完成后，会把本轮 user 和 assistant 追加回缓存；普通聊天和生成任务两条链路都会写回，见 agents/crm_agent.py:113、
    agents/crm_agent.py:114、agents/crm_agent.py:236、agents/crm_agent.py:237。
  - /new 会清空当前 session_id 的历史，见 agents/crm_agent.py:74；架构文档也明确写了“缓存不落盘，重启服务后清空”，见
    项目架构说明.md:130 和 项目架构说明.md:135。

  缓存多大

  - 内存缓存大小：默认每个会话最多保留 20 条消息，不是 20 轮，而是 20 条 message，约等于 10 轮问答，见 agents/
    crm_agent.py:34、agents/crm_agent.py:41、agents/crm_agent.py:43，文档也一致，见 项目架构说明.md:132。
  - LLM 上下文缓存大小：主聊天链路实际只喂给 LLM 最近 8 条历史消息，约 4 轮，见 agents/crm_agent.py:277；文档也写了最近
    8 条，见 项目架构说明.md:133。
  - 降级/重试链路：若第一次聊天超时，重试只带最近 4 条历史，见 agents/crm_agent.py:299 和 agents/crm_agent.py:304。
  - /api/generate 兜底链路：也只拼最近 4 条历史，见 agents/crm_agent.py:359 和 agents/crm_agent.py:360。
  - 任务路由器 LLM：当前只参考最近 4 条历史，见 agents/task_planner.py:109。
  - 不是缓存大小的参数：.env 里的 OLLAMA_NUM_PREDICT=96 只是最大生成长度，不是上下文缓存大小，见 .env:10 和 config/
    settings.py:19。

  一个重要限制

  - 当前代码只限制了“每个会话最多 20 条消息”，没有限制会话总数；也没有 TTL、LRU、全局淘汰策略。
  - 这意味着单个 session_id 不会无限长，但如果系统不断出现新的 session_id，进程总内存还是会继续增长；这一点可以从
    defaultdict 常驻在单例 CRMAgent 里推断出来，见 api/server.py:65 和 agents/crm_agent.py:42。
  - 另外，仓库里没有看到显式的 Ollama 模型级上下文参数（比如模型真实上下文窗口配置）；所以项目当前控制的是“发给 LLM 的历
    史条数”，不是模型内部 KV cache 的硬上限。这是基于代码搜索得到的推断。

  能不能扩容

  - 可以，而且比较容易扩。
  - 但现在不是纯配置化的，很多值写死在代码里；所以要扩容，最好改成配置项，而不是只改一个地方。

  如何扩容

  - 最小改法：直接改 agents/crm_agent.py:34 的 history_size: int = 20，例如改成 50 或 100。
  - 一起放大 LLM 可见窗口：把 agents/crm_agent.py:277 的 [-8:] 改成 [-12:] 或 [-16:]。
  - 同步放大降级链路：把 agents/crm_agent.py:304 和 agents/crm_agent.py:360 的 [-4:] 一并调大，否则主链路和兜底链路行为
    会不一致。
  - 同步放大路由器上下文：把 agents/task_planner.py:109 的 recent_history[-4:] 一并调大，否则任务判断仍只看很短历史。
  - 更合理的做法：把这些硬编码提到 config/settings.py 和 .env，例如新增 SESSION_HISTORY_SIZE、LLM_HISTORY_SIZE、
    LLM_RETRY_HISTORY_SIZE、PLANNER_HISTORY_SIZE。
  - 真正支持大规模扩容：把 session_histories 从进程内 deque 换成 Redis 或 SQLite；文档里也已经把“会话持久化（Redis/
    SQLite）替代纯内存缓存”列为扩展方向，见 项目架构说明.md:220。

  扩容建议

  - 如果你只是想让对话“记得更多”：先把 20/8/4 调到 50/12/6，通常是比较稳的起点。
  - 如果你要支持多用户并发或多实例部署：不要继续只用内存缓存，直接上 Redis。
  - 如果你要让模型真正吃下更长上下文：除了增大这里的历史条数，还要同步检查 Ollama 侧的上下文窗口，否则只会更慢，不一定真
    有效。
