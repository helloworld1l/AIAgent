# 项目操作手册

本文档用于说明当前项目在 Docker Compose 一键部署模式下的使用方法，覆盖：

- 首次启动
- 日常使用
- 重建索引
- 排错
- 常用命令速查

适用项目目录：`D:\python\rag_crm_agent`

---

## 1. 当前运行架构

当前项目**推荐使用混合模式**：

- Docker 只启动基础设施：`redis`、`qdrant`、`ollama`、`ollama-init`
- Python 应用层使用你本机的 `rag_crm` Conda 环境运行

推荐这样做的原因：

1. 可以直接复用你已经配置好的 `rag_crm` Python 依赖环境
2. 避开 `python:3.10-slim` 基础镜像拉取失败问题
3. 继续复用 Docker 管理的 Qdrant / Ollama / Redis 服务
4. 更适合当前这台机器的实际使用方式

推荐启动链路如下：

1. 在 Docker 中启动 `redis`
2. 在 Docker 中启动 `qdrant`
3. 在 Docker 中启动 `ollama`
4. 在 Docker 中运行 `ollama-init`，检查或拉取 `deepseek-r1:7b`
5. 在本机 `rag_crm` 环境执行 `python main.py build --with-qdrant`
6. 在本机 `rag_crm` 环境执行 `python main.py api`

默认访问地址：

- `http://127.0.0.1:8000/ui`：本机 API 页面
- `http://127.0.0.1:11434`：Ollama API
- `http://127.0.0.1:6333/collections`：Qdrant API

说明：

- 当前 `app1` / `app2` / `kb-build` 仍保留在 `docker-compose.yml` 中，属于“全 Docker 模式”的可选方案
- 如果后续解决了 Python 基础镜像拉取问题，仍可切回全 Docker 模式

## 2. 首次启动

### 2.1 启动前检查

请先确认：

1. 已安装并启动 `Docker Desktop`
2. 项目根目录下 `.env` 中的缓存路径正确
3. 如果本机网络无法访问 Docker Hub，需要提前准备可访问的 Python 基础镜像

当前默认缓存路径为：

- `DOCKER_OLLAMA_HOME=C:/Users/Lenovo/.ollama`
- `DOCKER_ST_CACHE_DIR=C:/Users/Lenovo/.cache/torch/sentence_transformers`
- `PYTHON_BASE_IMAGE=python:3.10-slim`

说明：

- `DOCKER_OLLAMA_HOME` 必须指向 `.ollama` 根目录，而不是只指向 `manifests/.../deepseek-r1/7b`
- `DOCKER_ST_CACHE_DIR` 建议指向 `sentence_transformers` 根目录，这样容器能直接复用已下载的模型缓存
- 如果 Docker Hub 网络不可达，可以把 `PYTHON_BASE_IMAGE` 改成公司私有仓库、镜像代理地址，或先离线导入后再使用

### 2.2 首次启动命令

先在 Docker 中只启动基础设施：

```powershell
docker compose up -d redis qdrant ollama ollama-init
```

然后在你已经激活的 `rag_crm` 终端中执行：

```powershell
python main.py build --with-qdrant
python main.py api
```

### 2.3 查看初始化过程

首次启动建议打开基础设施日志观察：

```powershell
docker compose logs -f redis qdrant ollama ollama-init
```

首次启动时可能耗时较长，原因通常是：

- Ollama 首次拉取模型
- Sentence Transformers 首次加载缓存
- Qdrant 首次就绪

之后知识库索引由本机 `python main.py build --with-qdrant` 写入 Qdrant。

如果启动时在 `FROM python:3.10-slim` 阶段报错，通常说明 Docker 无法访问 Docker Hub，而不是项目代码本身有问题。

如果本机缓存目录中已经有模型，一般会明显更快。

### 2.4 启动成功后的检查

执行：

```powershell
curl http://127.0.0.1:6333/collections
curl http://127.0.0.1:8000/api/health
```

成功时通常表现为：

- 本机 API 的返回中 `session_store_backend=memory` 或 `redis`
- Qdrant 能返回 collection 列表

说明：

- 如果你现在是 `python main.py api` 本机运行，默认会读 `.env`，其中 `SESSION_STORE_BACKEND` 当前是 `memory`
- 如果你后续想让本机 API 也共享 Redis 会话，可以把 `.env` 中 `SESSION_STORE_BACKEND` 改成 `redis`

## 3. 日常使用

### 3.1 启动服务

日常启动推荐分两步：

先启动基础设施：

```powershell
docker compose up -d redis qdrant ollama ollama-init
```

再在 `rag_crm` 环境中启动应用：

```powershell
python main.py api
```

如果你改了知识库文档、检索配置或模型数据，先执行：

```powershell
python main.py build --with-qdrant
```

### 3.2 查看服务状态

```powershell
docker compose ps
```

### 3.3 查看运行日志

查看基础设施日志：

```powershell
docker compose logs -f qdrant ollama ollama-init
```

本机 API 日志直接看当前终端输出即可。

查看全部关键基础服务日志：

```powershell
docker compose logs -f redis qdrant ollama ollama-init
```

### 3.4 访问系统

浏览器访问：

- `http://127.0.0.1:8000/ui`

你可以直接输入：

- 普通问答，例如：`什么是卡尔曼滤波？`
- 建模请求，例如：`生成一个PID闭环模型，kp=1.5, ki=0.8, kd=0.02`

系统内部会自动执行：

1. 检索知识库
2. 调用 Ollama
3. 如果识别为建模任务，则执行规格构建、校验、代码生成
4. 返回结果给前端

### 3.5 常规停止

停止本机 API：

- 在运行 `python main.py api` 的终端按 `Ctrl + C`

停止 Docker 基础设施但保留数据：

```powershell
docker compose down
```

停止并清理卷数据：

```powershell
docker compose down -v
```

说明：

- `down` 会停止容器，但保留 Redis/Qdrant 数据卷
- `down -v` 会删除 Docker 卷，适合彻底重置基础服务环境

## 4. 重建索引

### 4.1 什么时候需要重建

以下情况建议重建知识库或 Qdrant 索引：

1. 你修改了 `knowledge_base/docs`
2. 你修改了知识数据或检索逻辑
3. 你更换了嵌入模型
4. 你希望刷新 Qdrant collection

### 4.2 当前默认行为

当前混合模式下，索引构建由本机命令控制：

- `python main.py build`：只重建本地知识索引
- `python main.py build --with-qdrant`：重建本地索引并写入 Qdrant

推荐把 `build --with-qdrant` 作为知识库变更后的标准操作。

### 4.3 强制重建索引

如果你需要重建 Qdrant 索引，直接执行：

```powershell
python main.py build --with-qdrant
```

重建完成后建议重启本机 API：

```powershell
python main.py api
```

### 4.4 完全重置后重建

如果你想彻底清空 Docker 基础设施环境再重建：

```powershell
docker compose down -v
docker compose up -d redis qdrant ollama ollama-init
python main.py build --with-qdrant
```

这会删除并重建：

- Redis 数据
- Qdrant 数据
- 生成文件卷

## 5. 排错手册

### 5.1 `ollama-init` 一直拉模型

常见原因：

- `DOCKER_OLLAMA_HOME` 没指向 `.ollama` 根目录
- 只挂载了 `manifests/.../deepseek-r1/7b`，但没有 `blobs`

排查建议：

```powershell
docker compose logs -f ollama-init
```

确认 `.env` 中：

```powershell
DOCKER_OLLAMA_HOME=C:/Users/Lenovo/.ollama
```

### 5.2 向量模型重复下载

常见原因：

- `DOCKER_ST_CACHE_DIR` 没有指向 `sentence_transformers` 根目录
- 挂载路径错误，容器内没有读到已存在缓存

建议检查 `.env`：

```powershell
DOCKER_ST_CACHE_DIR=C:/Users/Lenovo/.cache/torch/sentence_transformers
```

### 5.3 `python main.py build --with-qdrant` 建库失败

排查步骤：

```powershell
curl http://127.0.0.1:6333/collections
```

重点检查：

- Qdrant 是否正常启动并监听 `6333`
- `QDRANT_COLLECTION` 是否配置正确
- 嵌入模型是否能正常加载

如果报 `ConnectError` / `WinError 10061`，通常表示 Qdrant 没有启动。

### 5.4 `python main.py api` 无法启动

排查步骤：

```powershell
docker compose ps
python main.py api
```

重点检查：

- `ollama` 是否正常启动
- `OLLAMA_BASE_URL` 是否指向 `http://localhost:11434`
- API 端口 `8000` 是否被占用

### 5.5 端口冲突

当前默认会使用这些端口：

- `8000`
- `8001`
- `11434`
- `6333`

如果有冲突，请先关闭占用进程，或修改 `docker-compose.yml` 端口映射。

### 5.6 验证 Compose 配置是否正确

在修改 `.env` 或 `docker-compose.yml` 后，可以先执行：

```powershell
docker compose config
```

这个命令可以帮助你确认：

- 变量是否正确展开
- 挂载路径是否正确生效
- 服务依赖关系是否正确

### 5.7 `python:3.10-slim` 拉取失败

你看到类似下面的错误时：

```text
failed to fetch anonymous token
auth.docker.io/token
connectex: A connection attempt failed
```

说明是 Docker 访问 Docker Hub 失败。

当前项目里这一步发生在“全 Docker 模式”的应用镜像构建阶段，因为 `Dockerfile` 需要基于 Python 基础镜像构建 `app1/app2/kb-build`。

如果你采用本手册推荐的“Conda 跑应用 + Docker 跑基础设施”模式，这个问题不会影响 `python main.py build --with-qdrant` 和 `python main.py api`。

处理方式有三种：

1. 修改 `.env` 中的 `PYTHON_BASE_IMAGE` 为你能访问的镜像源地址
2. 在其他能联网的机器上先拉取并导出 `python:3.10-slim`，再拷回本机 `docker load`
3. 使用公司内部 Docker Registry，并把 `PYTHON_BASE_IMAGE` 指向内部地址

如果你已经把本机导入的镜像重新打了标签，也可以直接把 `.env` 中的 `PYTHON_BASE_IMAGE` 改成那个本地标签。

## 7. 其他运行模式（可选）

除了本手册推荐的混合模式外，项目还保留其他运行方式。

### 7.1 全本地模式

如果你完全不使用 Docker，也可以手动确保本机已有 Qdrant / Ollama 服务后执行：

```powershell
python main.py build --with-qdrant
python main.py api
```

### 7.2 本地交互模式

```powershell
python main.py run
```

### 7.3 全 Docker 模式

如果后续解决了 Python 基础镜像拉取问题，也可以重新使用：

```powershell
docker compose up -d --build
```

说明：

- 当前最推荐的是：`Docker 起基础设施 + rag_crm 环境运行 Python 应用`
- 全本地模式适合手工调试
- 全 Docker 模式适合网络、镜像源和构建环境都稳定后再使用
